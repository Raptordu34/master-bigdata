Exercice 1 — Index de jointure binaire

Intermédiaire ◎ 12 pts ⏱ 30–40 min
Objectifs
  • Construire un index de jointure binaire (Binary Join Index) à partir de tables relationnelles
  • Comprendre comment l'optimiseur exploite les RIDs pour éviter les scans inutiles
  • Calculer manuellement le résultat d'une requête multi-tables via intersection de RIDs

Prérequis : Cours — Index et accès aux données, notion de RID

Principe : un index de jointure binaire (BJI) entre deux tables R et S sur un prédicat de jointure mémorise, pour chaque valeur d'un attribut de dimension, la liste des RIDs de la table de faits correspondants.

Il évite de lire les tuples des tables de dimension à l'exécution : l'intersection de plusieurs listes de RIDs identifie directement les tuples qualifiants dans la table de faits.

  • JI(R, S) sur A : pour chaque valeur de A dans R, stocke les RIDs des tuples de S qui se jointuraient avec R sur cet attribut.
  • Utilisation : les conditions sur les dimensions (ex: Level='HCSC') sélectionnent une liste de RIDs dans Grades ; l'intersection de plusieurs listes donne les tuples qualifiants.

Repère

Si on a k conditions sur k dimensions, on intersecte k listes de RIDs de la table de faits — sans jamais lire les tables de dimension au moment de la requête.

Q1 4 pts Fondamental

Construire un index de jointure binaire JI1 entre Student et Grades en utilisant l'attribut Level, et un index JI2 entre Course et Grades en utilisant l'attribut Dept.

Écrire les requêtes SQL de création de ces index.

Piège courant : Stocker les paires (Student.RID, Grades.RID) sans les indexer par valeur d'attribut rend l'index inutilisable. L'index doit être organisé par valeur de dimension (ex: HCSC → liste de RIDs Grades).
Pour JI1 : parcourir la table Grades, et pour chaque tuple, retrouver le Level de l'étudiant correspondant dans Student. Regrouper les RIDs de Grades par valeur de Level.
Exemple pour JI2, valeur Stat : S402 (Stat) est dans Grades aux RIDs {9,16,17,27} et S466 (Stat) aux RIDs {18,23,24}. Donc Dept=Stat → RIDs Grades = {9,16,17,18,23,24,27}.

Solution — Q1 : Construction des index de jointure binaire

Requêtes SQL de création :

-- JI1 : Index de jointure binaire Student ⋈ Grades sur Level
CREATE BITMAP INDEX JI1_Level
ON Grades (Student.Level)
FROM Student, Grades
WHERE Student.StudId = Grades.StudId;
-- JI2 : Index de jointure binaire Course ⋈ Grades sur Dept
CREATE BITMAP INDEX JI2_Dept
ON Grades (Course.Dept)
FROM Course, Grades
WHERE Course.CourId = Grades.CourId;

Q2 3 pts Intermédiaire

Montrer comment l'optimiseur de requêtes utilise les index JI1 et JI2 pour éviter les accès non nécessaires aux tables lors de l'exécution de la requête suivante : 

SELECT Count(*)
FROM Student, Grades, Course
WHERE Level = 'HCSC'
  AND Dept  = 'Stat'
  AND Student.StudId = Grades.StudId
  AND Course.CourId  = Grades.CourId;
Piège courant : Penser que l'optimiseur doit quand même lire Student et Course pour vérifier les conditions. Avec les BJI, ces vérifications ont déjà été pré-calculées — les RIDs dans l'index sont directement les tuples qualifiants.
Cherchez dans JI1 la liste des RIDs de Grades correspondant à Level='HCSC', puis dans JI2 la liste correspondant à Dept='Stat'. Que fait-on ensuite avec ces deux listes ?
L'intersection des deux listes de RIDs donne exactement les tuples de Grades qui satisfont à la fois les deux conditions de jointure. Aucun accès à Student ou Course n'est nécessaire à ce stade.

Solution — Q2 : Utilisation des BJI par l'optimiseur

Plan d'exécution avec les index de jointure :

  • Chercher dans JI1 l'entrée Level = 'HCSC' → obtenir la liste L1 = {1, 3, 6, 7, 8, 9, 14, 15, 16, 24, 25, 26, 27} (RIDs dans Grades)
  • Chercher dans JI2 l'entrée Dept = 'Stat' → obtenir la liste L2 = {9, 16, 17, 18, 23, 24, 27} (RIDs dans Grades)
  • Calculer l'intersection L1 ∩ L2 = {9, 16, 24, 27}
  • Accéder directement aux 4 tuples de Grades ayant ces RIDs

Avantage clé

L'optimiseur n'accède jamais aux tables Student et Course durant l'exécution : les conditions sur Level et Dept ont déjà été matérialisées dans les index. Seuls les tuples utiles de Grades sont lus.

ÉtapeTable lueTuples accédés
Sans BJIStudent + Course + Grades6 + 7 + 27 = 40 tuples
Avec BJIGrades uniquement4 tuples (intersection des RIDs)
Q3 2 pts Fondamental

Quel est le résultat retourné par cette requête ?

Piège courant : Compter le nombre d'étudiants HCSC (3) ou le nombre de cours Stat (2) au lieu de compter les inscriptions qui satisfont les deux conditions simultanément.
L'intersection calculée en Q2 donne directement les tuples de Grades qualifiants. Il suffit de les compter.

Q4 3 pts Intermédiaire

Quel est le nombre de tuples de la table Grades que l'optimiseur doit scanner lors de l'exécution avec les index de jointure binaire ?

Piège courant : Confondre "lire tous les Grades pour vérifier les conditions" (scan complet = 27 tuples) avec "accéder uniquement aux Grades qualifiants identifiés par les BJI" (accès ciblé).
Après l'intersection des RIDs, l'optimiseur connaît exactement quels tuples de Grades accéder. Combien y en a-t-il dans le résultat de l'intersection ?

Solution — Q4 : Tuples Grades accédés

Grâce aux index de jointure binaire, l'optimiseur calcule l'intersection des RIDs sans lire Grades. Il accède ensuite uniquement aux 4 tuples identifiés (RIDs 9, 16, 24, 27), au lieu de scanner les 27 tuples de la table.

ApprocheTuples Grades lusRatio
Sans index (scan séquentiel)27100 %
Avec BJI (intersection RIDs)4≈ 15 %

En pratique, les BJI opèrent sur des bitmaps : l'intersection est une opération AND bit-à-bit, très rapide en mémoire. Les tables de dimension (Student, Course) ne sont pas du tout accédées.
