Exercice 1 — Index de jointure binaire

Intermédiaire ◎ 12 pts ⏱ 30–40 min
Objectifs
  • Construire un index de jointure binaire (Binary Join Index) à partir de tables relationnelles
  • Comprendre comment l'optimiseur exploite les RIDs pour éviter les scans inutiles
  • Calculer manuellement le résultat d'une requête multi-tables via intersection de RIDs

Prérequis : Cours — Index et accès aux données, notion de RID

Définition : Un index de jointure binaire (BJI — Binary Join Index) entre deux tables R et S prématé­rialise le résultat de la jointure sous forme d'une liste de RIDs. Pour chaque valeur d'un attribut de R, l'index stocke les RIDs des tuples de S qui lui correspondraient dans une jointure.

Analogie

Image mentale : pensez à un répertoire téléphonique organisé par ville. Plutôt que de parcourir toutes les lignes pour trouver les abonnés de Lyon, vous allez directement à la lettre L, qui pointe une liste d'entrées.

De même, JI₁[HCSC] pointe directement les RIDs des inscriptions (Grades) d'étudiants HCSC — sans relire la table Student au moment de la requête.

Architecture logique d'un BJI :

Le schéma ci-dessous montre les deux phases : la construction (lecture de Student + Grades → index) et l'utilisation (l'index remplace la lecture de Student).

  • Phase 1 — Construction : on parcourt Grades + Student une fois pour construire l'index. Résultat : pour chaque valeur de Level, une liste de Grades.RIDs.
  • Phase 2 — Requête : au moment de la requête, on lit directement JI₁[Level='HCSC'] → on obtient les RIDs sans toucher à Student.
  • Intersection : si k conditions de dimension, on intersecte k listes de RIDs → seuls les tuples qualifiants de Grades sont lus.

Repère

Un BJI ne stocke pas les valeurs des attributs — uniquement des RIDs. Il élimine les accès aux tables de dimension pendant l'exécution. C'est sa différence fondamentale avec un index B-tree classique.

1. Construction du BJI : ce que la base fait tuple par tuple

En pratique, un moteur de type Oracle construit un bitmap join index sur la table de faits en utilisant les valeurs de dimension comme clés, puis en stockant les rowids / bitmaps de la table de faits correspondants. L'idée pédagogique ici reste la même que dans l'exercice : Level ou Dept → tuples de Grades visés.

1
Scanner la table de faits

La base prend un tuple de Grades. Exemple : RID 9, avec StudId = 101 et CourId = S402.

Grades RID 9 → StudId = 101 CourId = S402
2
Suivre la jointure vers la dimension

Pour construire JI₁, la base suit la clé de jointure Grades.StudId = Student.StudId, retrouve l'étudiant, puis lit Student.Level.

StudId = 101 → Student Ali → Level = HCSC
3
Utiliser la valeur de dimension comme clé d'index

La clé lue dans la dimension devient l'entrée de l'index. Ici, la base n'ajoute pas “Ali” dans l'index : elle ajoute le RID du tuple de faits dans l'entrée JI₁[HCSC].

clé = HCSC → JI₁[HCSC] ← ajouter RID 9
4
Répéter jusqu'à couvrir tous les tuples de Grades

La base recommence pour RID 1, 2, 3, ..., 27. A la fin, chaque valeur de dimension pointe vers l'ensemble complet des tuples de Grades qui la satisfont.

RID 1 → HCSC RID 2 → MCSC RID 3 → HCSC ... JI₁ terminé
Version “moteur réel” : dans un SGBD orienté data warehouse, ces ensembles sont souvent stockés comme des bitmaps compressés plutôt qu'en simples listes affichées. Le principe logique reste identique : une valeur de dimension sélectionne directement les lignes de la table de faits.

2. Exécution de la requête : ce que fait la base avant de lire Grades

La doc Oracle décrit ce traitement en deux phases : d'abord identifier exactement les lignes utiles de la table de faits via les bitmaps / rowids, puis seulement ensuite accéder à la table de faits. C'est précisément ce que ton exercice illustre.

1
Lire les prédicats de dimension

Le parseur voit deux conditions filtrantes : Level = 'HCSC' et Dept = 'Stat'. Chacune correspond à une entrée ciblée dans un index de jointure.

WHERE Level = 'HCSC' WHERE Dept = 'Stat'
2
Accéder directement aux entrées d'index

La base lit JI₁[HCSC] et JI₂[Stat]. A ce moment, elle ne relit pas Student ni Course pour tester les conditions : elles sont déjà matérialisées dans les index.

JI₁[HCSC] → {1,3,6,7,8,9,14,15,16,24,25,26,27} JI₂[Stat] → {9,16,17,18,23,24,27}
3
Faire l'opération d'ensemble en mémoire

Le moteur combine les deux représentations par un AND bitmap ou, dans ta version simplifiée, par une intersection de RIDs. Le résultat est l'ensemble exact des lignes de Grades utiles.

JI₁[HCSC]
9 16 17 18 23 24 27
JI₂[Stat]
9 16 17 18 23 24 27
Intersection
9 16 ∅ ∅ ∅ 24 27
4
Convertir le résultat en lignes de faits à lire

Après l'AND, la base connaît exactement les RIDs utiles : {9, 16, 24, 27}. C'est seulement maintenant qu'elle va lire les tuples de Grades correspondants.

résultat = {9,16,24,27} → lecture ciblée de 4 lignes Grades
5
Appliquer l'opération finale de la requête

Comme la requête demande COUNT(*), il n'y a même pas besoin d'afficher Student ou Course. Le moteur compte simplement les tuples qualifiants : 4.

RIDs qualifiants = 4 → COUNT(*) = 4
Point clé issu de la doc data warehouse moderne : le gros gain vient du fait que le filtrage logique est fait avant l'accès à la table de faits. Les opérations booléennes sur bitmaps sont rapides, puis la table de faits n'est lue qu'à la fin, et seulement sur les lignes survivantes.

Algorithme de construction

Pour construire JI₁ (Student ⋈ Grades sur Level), on effectue une boucle sur chaque tuple de Grades :

Pseudo-code

JI₁ = {}  // dictionnaire vide

POUR CHAQUE tuple t DE Grades :
    s = Student WHERE s.StudId = t.StudId
    level = s.Level
    JI₁[level].append(t.RID)

// Résultat :
// JI₁["HCSC"] = [1, 3, 6, 7, 8, 9, ...]
// JI₁["MCSC"] = [2, 4, 10, 11, ...]
// JI₁["MSC"]  = [5, 21, 22, 23]

Points clés

  • On parcourt Grades une seule fois (27 tuples)
  • Pour chaque tuple, une jointure avec Student via StudId
  • L'index regroupe les RIDs par valeur de Level
  • Le coût de construction est amorti sur toutes les requêtes futures

Utilisation de l'index : intersection de listes

Pour répondre à une requête avec plusieurs conditions de dimension, l'optimiseur intersecte les listes de RIDs des index correspondants :

Les 4 RIDs en orange sont ceux présents dans les deux listes. Ce sont les seuls tuples de Grades que l'optimiseur devra lire — sans jamais rouvrir les tables Student ou Course.

Q1 4 pts Fondamental

Construire un index de jointure binaire JI1 entre Student et Grades en utilisant l'attribut Level, et un index JI2 entre Course et Grades en utilisant l'attribut Dept.

Écrire les requêtes SQL de création de ces index.

Piège courant : Stocker les paires (Student.RID, Grades.RID) sans les indexer par valeur d'attribut rend l'index inutilisable. L'index doit être organisé par valeur de dimension (ex: HCSC → liste de RIDs Grades).
Pour JI1 : parcourir la table Grades, et pour chaque tuple, retrouver le Level de l'étudiant correspondant dans Student. Regrouper les RIDs de Grades par valeur de Level. (Référez-vous à la trace de construction dans le guide ci-dessus.)

Pour JI2, même méthode avec Course et Dept :

  • Grades.RID 1 : CourId=C306 → Course.Dept=Comp
  • Grades.RID 7 : CourId=C422 → Course.Dept=Comp
  • Grades.RID 9 : CourId=S402 → Course.Dept=Stat
  • …

Exemple pour Dept=Stat : S402 (RID=5 dans Course) est dans Grades aux RIDs {9,16,17,27} et S466 (RID=7) aux RIDs {18,23,24}. Donc Dept=Stat → RIDs Grades = {9,16,17,18,23,24,27}.

Solution — Q1 : Construction des index de jointure binaire

Requêtes SQL de création :

-- JI1 : Index de jointure binaire Student ⋈ Grades sur Level
CREATE BITMAP INDEX JI1_Level
ON Grades (Student.Level)
FROM Student, Grades
WHERE Student.StudId = Grades.StudId;
-- JI2 : Index de jointure binaire Course ⋈ Grades sur Dept
CREATE BITMAP INDEX JI2_Dept
ON Grades (Course.Dept)
FROM Course, Grades
WHERE Course.CourId = Grades.CourId;

Q2 3 pts Intermédiaire

Montrer comment l'optimiseur de requêtes utilise les index JI1 et JI2 pour éviter les accès non nécessaires aux tables lors de l'exécution de la requête suivante : 

SELECT Count(*)
FROM Student, Grades, Course
WHERE Level = 'HCSC'
  AND Dept  = 'Stat'
  AND Student.StudId = Grades.StudId
  AND Course.CourId  = Grades.CourId;
Piège courant : Penser que l'optimiseur doit quand même lire Student et Course pour vérifier les conditions. Avec les BJI, ces vérifications ont déjà été pré-calculées — les RIDs dans l'index sont directement les tuples qualifiants.
Cherchez dans JI1 la liste des RIDs de Grades correspondant à Level='HCSC', puis dans JI2 la liste correspondant à Dept='Stat'. Que fait-on ensuite avec ces deux listes ?
L'intersection des deux listes de RIDs donne exactement les tuples de Grades qui satisfont à la fois les deux conditions de jointure. Référez-vous au schéma d'intersection dans le guide de construction ci-dessus. Aucun accès à Student ou Course n'est nécessaire à ce stade.

Solution — Q2 : Utilisation des BJI par l'optimiseur

Plan d'exécution avec les index de jointure :

  • Chercher dans JI1 l'entrée Level = 'HCSC' → obtenir la liste L1 = {1, 3, 6, 7, 8, 9, 14, 15, 16, 24, 25, 26, 27} (RIDs dans Grades)
  • Chercher dans JI2 l'entrée Dept = 'Stat' → obtenir la liste L2 = {9, 16, 17, 18, 23, 24, 27} (RIDs dans Grades)
  • Calculer l'intersection L1 ∩ L2 = {9, 16, 24, 27}
  • Accéder directement aux 4 tuples de Grades ayant ces RIDs

Avantage clé

L'optimiseur n'accède jamais aux tables Student et Course durant l'exécution : les conditions sur Level et Dept ont déjà été matérialisées dans les index. Seuls les tuples utiles de Grades sont lus.

ÉtapeTable lueTuples accédés
Sans BJIStudent + Course + Grades6 + 7 + 27 = 40 tuples
Avec BJIGrades uniquement4 tuples (intersection des RIDs)

L'intersection calculée en Q2 donne directement les tuples de Grades qualifiants. Il suffit de les compter.

Q4 3 pts Intermédiaire

Quel est le nombre de tuples de la table Grades que l'optimiseur doit scanner lors de l'exécution avec les index de jointure binaire ?

Piège courant : Confondre "lire tous les Grades pour vérifier les conditions" (scan complet = 27 tuples) avec "accéder uniquement aux Grades qualifiants identifiés par les BJI" (accès ciblé).
Après l'intersection des RIDs, l'optimiseur connaît exactement quels tuples de Grades accéder. Combien y en a-t-il dans le résultat de l'intersection ?

Solution — Q4 : Tuples Grades accédés

Grâce aux index de jointure binaire, l'optimiseur calcule l'intersection des RIDs sans lire Grades. Il accède ensuite uniquement aux 4 tuples identifiés (RIDs 9, 16, 24, 27), au lieu de scanner les 27 tuples de la table.

ApprocheTuples Grades lusRatio
Sans index (scan séquentiel)27100 %
Avec BJI (intersection RIDs)4≈ 15 %

En pratique, les BJI opèrent sur des bitmaps : l'intersection est une opération AND bit-à-bit, très rapide en mémoire. Les tables de dimension (Student, Course) ne sont pas du tout accédées.
