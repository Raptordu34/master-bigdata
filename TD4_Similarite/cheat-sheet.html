Mémento 1 : Syntaxe SQL & Indexation Binaire (BJI)

Master Big Data 2026

1. Structure Fondamentale

SELECT [DISTINCT] col AS alias
FROM Table_A [JOIN Table_B ON A.id = B.fk]
WHERE condition (Filtre lignes)
GROUP BY col (Agrégation)
HAVING condition (Filtre groupes)
ORDER BY col [ASC|DESC]
Logique NULL : IS NULL / IS NOT NULL.
COALESCE(val, 0) remplace NULL par 0.
NOT IN (..., NULL) ne renvoie rien.

2. Agrégats & GROUP BY

FonctionComportement
COUNT(*)Toutes les lignes
COUNT(col)Lignes où col ≠ NULL
AVG, SUMIgnorent les NULL
Règle d'or : Toute colonne dans le SELECT non agrégée doit être dans le GROUP BY.

3. Opérateurs de Jointure

INNER JOIN : Intersection stricte.
LEFT JOIN : Tout le parent + enfants si existent.
CROSS JOIN : Produit cartésien (A × B).
Semi-Join (⋉) : Tuples de A ayant un match dans B. Utile en fragmentation dérivée.

4. Index de Jointure Binaire (BJI)

Déf : Indexe une table de faits (ex: Grades) sur des attributs de tables de dimension (ex: Level de Student).

Mécanisme de construction :

1. Joindre Fact et Dimension.
2. Grouper par attribut de dimension.
3. Stocker liste des RID (Record IDs) de Fact.

Exemple conceptuel :

Level='HCSC' → Bitmaps: 100100... (RIDs: {1, 4, ...})

Avantage : Évite la jointure physique à l'exécution. On travaille directement sur les listes de RID / Bitmaps.

5. Résolution par intersection

Pour une requête avec plusieurs critères (Dept='Stat' AND Level='HCSC') :

RID_Final = RID(Stat) ∩ RID(HCSC)

En bitmap : Bitmap_A AND Bitmap_B.

Coût estimé : Nombre de RIDs dans le résultat final (si index parfait).

6. Fonctions Utilitaires

Texte :

UPPER(), LOWER(), TRIM()
SUBSTR(str, start, len)
CONCAT(a, b) ou a || b

Maths :

ROUND(val, 2), ABS(), FLOOR()
MOD(a, b) ou a % b

Dates :

CURRENT_DATE, EXTRACT(YEAR FROM d)
TO_CHAR(d, 'YYYY-MM-DD')

Conditionnel :

CASE WHEN cond THEN val1 
     ELSE val2 END

7. Pièges Classiques (Exam)

Confondre WHERE (filtre avant agrégat) et HAVING (filtre après).
Oublier les alias dans une auto-jointure.
Penser que COUNT(col) compte les NULL.

Master Big Data 2026

8. Partitionnement Horizontal

A. Primaire (PHP) :

Partitionnement d'une table R selon ses propres attributs.

R_i = σ_{pred_i}(R)

B. Dérivé (PHD) :

Partitionnement d'une table de faits F selon les partitions d'une table dimension D.

F_i = F ⋉ D_i (Semi-jointure)
Analogie logistique : On range les "Ventes" directement dans des cartons étiquetés par "Région".

9. Élimination de partitions

Partition Pruning : L'optimiseur ne scanne que les fragments concernés par le WHERE.

Si la requête filtre sur Dept='Stat' et que Grades est fragmenté par Dept, on ne lit qu'un seul fragment.

Coût : Somme des tailles des fragments accédés.

10. Sous-Requêtes Avancées

Scalaire (1 valeur) :

WHERE prix > (SELECT AVG(prix) FROM...)

Liste (IN / ANY / ALL) :

WHERE id IN (SELECT id FROM...)

Corrélée (Exécute par ligne) :

SELECT nom FROM emp e
WHERE salaire > (
  SELECT AVG(salaire) FROM emp 
  WHERE dept = e.dept
)
EXISTS : Plus rapide que IN pour tester la présence (s'arrête au 1er match).

12. Conseils & Pièges TD4

Piège : Fragmenter la table de faits sans passer par les dimensions (PHD).
Piège : Additionner tous les fragments lors du calcul du coût (on ne lit que l'utile !).
ASTUCE EXAM : Toujours vérifier si l'attribut du WHERE correspond à la clé de fragmentation ou à un index BJI existant.
