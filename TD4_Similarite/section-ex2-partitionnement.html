Exercice 2 — Partitionnement horizontal dérivé

Avancé ◎ 14 pts ⏱ 35–50 min
Objectifs
  • Concevoir une fragmentation horizontale dérivée pour optimiser une requête multi-tables
  • Identifier les fragments pertinents et calculer le nombre de tuples accédés
  • Comparer la fragmentation aux index de jointure binaire (similarité conceptuelle)

Prérequis : Exercice 1 — Index de jointure binaire · Cours — Fragmentation horizontale dérivée

Contexte — Même base Scolarité, approche différente

On reprend le même schéma que l'exercice 1 : tables Student, Course et Grades (27 tuples). L'objectif est cette fois d'optimiser la même requête par fragmentation horizontale dérivée plutôt que par des index de jointure binaire.

Requête cible :

SELECT Count(*)
FROM Student, Grades, Course
WHERE Level = 'HCSC'
  AND Dept  = 'Stat'
  AND Student.StudId = Grades.StudId
  AND Course.CourId  = Grades.CourId;

Les tables et leurs populations sont identiques à celles de l'exercice 1.

Fragmentation horizontale primaire : on partitionne une table selon des prédicats sur ses propres attributs.

R_i = σ(prédicat_i)(R)

Fragmentation horizontale dérivée : on partitionne une table de faits en fonction des fragments d'une table de dimension (via la clé de jointure).

F_i = Faits ⋉ D_i   (semi-jointure derivée)
  • D est fragmenté en D1, D2, … selon un prédicat sur un attribut de dimension.
  • F est ensuite fragmenté de façon dérivée : Fi contient les tuples de F qui se jointurent avec Di.
  • Pour deux dimensions, on croise les fragments : Fij = F ⋉ Di ∩ F ⋉ Ej.

Repère

L'optimiseur peut alors ignorer les fragments qui ne peuvent contribuer au résultat : si la requête filtre sur Level='HCSC' et Dept='Stat', seul le fragment FHCSC,Stat est accédé.

Exemple

Fragmenter Student par Level donne 3 fragments : S_HCSC, S_MCSC, S_MSC. La fragmentation dérivée de Grades par Student donne G_HCSC, G_MCSC, G_MSC — chaque fragment contient exactement les inscriptions des étudiants du groupe correspondant.

Analogie

Intuition : c'est comme partitionner un entrepôt en rayons thématiques. Au lieu de chercher un produit dans tout l'entrepôt, on sait directement dans quel rayon aller — et si la requête ne concerne que ce rayon, on ignore tous les autres.

Q1 5 pts Intermédiaire

Proposer une fragmentation de la base de données afin d'optimiser la requête cible. Définir les fragments de Student, Course et Grades, en listant les tuples (RIDs) de chaque fragment.

Piège courant : Fragmenter uniquement Grades directement, sans passer par les tables de dimension. La fragmentation de Grades doit être dérivée de celles de Student et Course — c'est ce qui garantit l'élimination de partitions à l'exécution.
Commencez par fragmenter Student par Level : quels tuples (et RIDs Student) appartiennent à chaque fragment ?
Fragmentez ensuite Course par Dept. Puis, pour chaque combinaison (fragment Student, fragment Course), dérivez le fragment de Grades : ce sont les tuples de Grades dont le StudId est dans le fragment Student ET le CourId est dans le fragment Course.
Pour la requête, seul le fragment GHCSC,Stat est pertinent. Identifiez les StudIds HCSC = {101, 303, 606} et les CourIds Stat = {S402, S466}. Quels RIDs de Grades ont à la fois StudId ∈ {101,303,606} ET CourId ∈ {S402,S466} ?

Q3 2 pts Fondamental

Quel est le nombre de tuples de la table des faits (Grades) que l'optimiseur doit scanner ?

Piège courant : Additionner tous les fragments accédés. La fragmentation permet justement d'isoler un seul fragment cible — l'optimiseur ne consulte pas les autres.
Avec la fragmentation, l'optimiseur sait qu'aucun tuple satisfaisant les deux prédicats ne peut se trouver en dehors du fragment G_HCSC_Stat. Il n'a donc besoin de lire que ce fragment.

Solution — Q3 : Tuples scannés

L'optimiseur effectue un scan complet du seul fragment G_HCSC_Stat, soit 4 tuples sur 27.

Q4 2 pts Intermédiaire

Comparer cette solution (fragmentation) avec celle des index de jointure binaire (exercice 1). Quelles sont les différences pratiques ?

Piège courant : Conclure que l'une est "meilleure" que l'autre de façon absolue. Les deux approches ont des avantages différents selon le contexte (charge de mise à jour, requêtes ad hoc, volume…).
Comparez selon plusieurs critères : stockage (structures auxiliaires vs physique), maintenabilité (coût à l'insert/update), flexibilité (requêtes non prévues), et transparence pour l'optimiseur.

Solution — Q4 : Comparaison BJI vs Fragmentation

CritèreIndex de jointure binaire (Ex. 1)Fragmentation dérivée (Ex. 2)
Résultat Count(*) = 4 Count(*) = 4
Mécanisme Intersection de listes de RIDs (ou bitmaps) en mémoire Routage physique vers le fragment approprié
Stockage Structures auxiliaires supplémentaires (index) Les données sont directement réorganisées physiquement

Q5 3 pts Avancé

Y a-t-il une similarité entre les index de jointure binaire et le partitionnement horizontal dérivé ? Si oui, commenter votre réponse.

Piège courant : Se limiter à "les deux donnent le même résultat". La similarité est plus profonde : elle touche à la structure logique de la sélectivité pré-calculée, pas seulement au résultat final.
Comparez ce que contient un fragment G_HCSC_Stat avec ce que représente l'intersection L1 ∩ L2 dans les BJI. Que décrivent-ils tous les deux ?
Les deux techniques pré-calculent la jointure entre la table de faits et les dimensions, mais l'une le fait de façon logique (RIDs) et l'autre de façon physique (données réorganisées). Peut-on dire que l'une est la version physique de l'autre ?

Solution — Q5 : Similarité entre BJI et fragmentation

Oui, il existe une similarité profonde. Les deux techniques reposent sur le même principe : pré-calculer la sélectivité de la jointure pour éviter les scans inutiles à l'exécution.

Dimension d'analyseBJI (Ex. 1)Fragmentation (Ex. 2)
Ce qui est pré-calculé Pour chaque valeur de dimension, la liste des RIDs de Grades qui satisfont la jointure Pour chaque combinaison de valeurs de dimensions, le sous-ensemble physique de Grades qui satisfait la jointure
Forme logique Structure auxiliaire : listes de RIDs (ou bitmaps) Structure physique : données réorganisées en partitions
Équivalence L'intersection L1(HCSC) ∩ L2(Stat) = {9,16,24,27} est exactement l'ensemble des RIDs du fragment G_HCSC_Stat

Conclusion

La fragmentation horizontale dérivée peut être vue comme la matérialisation physique de ce que les index de jointure binaire représentent de façon logique. Le BJI dit « pour cette valeur, les tuples qualifiants ont ces RIDs » ; la fragmentation dit « pour cette valeur, ces tuples qualifiants sont dans ce fragment sur disque ».

Les deux techniques éliminent les mêmes accès inutiles, mais par des mécanismes différents : le BJI fonctionne par intersection logique de RIDs, la fragmentation par routage physique vers une partition.
