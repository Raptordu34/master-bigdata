Q1.1 – Q1.3 : VM Performances Étudiants

Synthèse

  • Q1.1 — CREATE MATERIALIZED VIEW VM_Performance_Etudiants AS SELECT … LEFT JOIN Inscriptions GROUP BY …
  • Q1.2 — Requête directe = JOIN + GROUP BY + WHERE sur les tables de base. Requête réécrite = simple SELECT * FROM VM_Performance_Etudiants WHERE Niveau = 'Master'.
  • Q1.3 — Avec 50 000 étudiants et 500 000 inscriptions exécutés 100×/jour, la VM supprime 99 recalculs coûteux sur 100. Gain énorme sur le temps CPU et les I/O.

Question 1.1 — Vue des performances étudiantes

Créer une vue matérialisée VM_Performance_Etudiants qui affiche, pour chaque étudiant : son ID, son nom complet (Nom + Prénom), son niveau d'études, le nombre total de cours suivis et sa moyenne générale.

Réponse
CREATE MATERIALIZED VIEW VM_Performance_Etudiants AS
SELECT
    e.ID_etudiant,
    e.Nom || ' ' || e.Prenom          AS Nom_Complet,
    e.Niveau,
    COUNT(i.ID_cours)               AS Nb_Cours_Suivis,
    ROUND(AVG(i.Note), 2)             AS Moyenne_Generale
FROM   Etudiants e
LEFT JOIN Inscriptions i ON e.ID_etudiant = i.ID_etudiant
GROUP BY
    e.ID_etudiant, e.Nom, e.Prenom, e.Niveau;

Comprendre le GROUP BY

GROUP BY regroupe les lignes qui partagent la même valeur dans une ou plusieurs colonnes, afin d'appliquer des fonctions d'agrégation (COUNT, AVG, SUM…) sur chaque groupe séparément.

Sans GROUP BY

SELECT AVG(Note)
FROM Inscriptions;

Retourne une seule ligne : la moyenne de toutes les notes de tous les étudiants confondus.

AVG(Note)
──────────
12.74

Avec GROUP BY e.ID_etudiant

SELECT e.ID_etudiant, AVG(i.Note)
FROM Etudiants e
JOIN Inscriptions i
  ON e.ID_etudiant = i.ID_etudiant
GROUP BY e.ID_etudiant;

Retourne une ligne par étudiant, chacune avec sa propre moyenne.

ID | AVG(Note)
───┼──────────
1 | 14.20
2 | 11.50
3 | 13.80
Règle fondamentale : toute colonne dans le SELECT qui n'est pas une agrégation doit apparaître dans le GROUP BY. C'est pourquoi on écrit GROUP BY e.ID_etudiant, e.Nom, e.Prenom, e.Niveau — ces quatre colonnes sont dans le SELECT sans être encapsulées dans COUNT ou AVG.

Pourquoi pas juste GROUP BY e.ID_etudiant ? En SQL standard, le moteur ne sait pas quelle valeur de Nom retourner pour le groupe — même si logiquement il n'y en a qu'une. PostgreSQL l'accepte quand ID_etudiant est PRIMARY KEY (il déduit la dépendance fonctionnelle), mais écrire toutes les colonnes reste la pratique portable et explicite, valable sur tous les SGBD.
Pourquoi LEFT JOIN ? Un étudiant inscrit à zéro cours doit quand même apparaître dans la vue (avec Nb_Cours_Suivis = 0 et Moyenne_Generale = NULL). Un INNER JOIN exclurait ces étudiants.
Syntaxe Oracle / PostgreSQL : La concaténation || est standard SQL. Sous MySQL on utilise CONCAT(e.Nom, ' ', e.Prenom).

Question 1.2 — Requête directe vs vue matérialisée

Un responsable pédagogique veut les performances de tous les étudiants de niveau Master.

1. Requête directe (sans VM)

-- Accès direct aux tables de base
SELECT
    e.ID_etudiant,
    e.Nom || ' ' || e.Prenom  AS Nom_Complet,
    e.Niveau,
    COUNT(i.ID_cours)     AS Nb_Cours_Suivis,
    ROUND(AVG(i.Note), 2) AS Moyenne_Generale
FROM   Etudiants e
LEFT JOIN Inscriptions i
    ON e.ID_etudiant = i.ID_etudiant
WHERE  e.Niveau = 'Master'
GROUP BY
    e.ID_etudiant,
    e.Nom, e.Prenom,
    e.Niveau;

2. Requête réécrite (avec VM)

-- Lecture directe de la vue pré-calculée
SELECT *
FROM   VM_Performance_Etudiants
WHERE  Niveau = 'Master';

Le SGBD lit directement les lignes stockées dans la VM — aucun JOIN ni GROUP BY à l'exécution.

Question 1.3 — Analyse des avantages

Contexte : 50 000 étudiants · 500 000 inscriptions · requête exécutée 100 fois par jour.
Critère Requête directe Via VM_Performance_Etudiants
Opérations à chaque exécution Scan Etudiants (50 000 lignes) + Scan Inscriptions (500 000) + JOIN + GROUP BY + AVG Scan de la VM (~50 000 lignes précalculées) + filtre WHERE simple
Coût CPU / I/O par exécution Élevé — plusieurs centaines de milliers de lectures disque Très faible — lecture séquentielle d'une seule table
Coût total sur la journée (×100) 100 × (coût JOIN + agrégation) → charge CPU constante 1 rafraîchissement nocturne + 100 lectures légères
Temps de réponse utilisateur Plusieurs secondes (à l'échelle millions de lignes) Millisecondes
Fraîcheur des données Toujours à jour Données du dernier rafraîchissement
Coût de stockage Aucun surcoût ~50 000 lignes supplémentaires stockées

Conclusion

Avec 100 exécutions quotidiennes, le coût d'un JOIN de 550 000 lignes est payé 100 fois par jour en mode direct, contre une seule fois (lors du rafraîchissement) en mode VM. Pour un usage en lecture intensive (tableau de bord, reporting), la vue matérialisée est le choix optimal.

La légère perte de fraîcheur (données décalées d'un cycle de rafraîchissement) est généralement acceptable dans un contexte académique où les notes et inscriptions ne changent pas à la minute.

Réécriture automatique (Query Rewriting) : des SGBD comme Oracle ou PostgreSQL peuvent réécrire automatiquement la requête directe pour utiliser la VM lorsque le planificateur détecte qu'elle est équivalente et plus efficace — sans modifier le code applicatif.
