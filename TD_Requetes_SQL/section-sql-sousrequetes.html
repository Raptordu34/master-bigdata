Niveau 4 — Sous-requêtes

Avancé ⏱ 40–55 min
Objectifs
  • Distinguer sous-requête scalaire, liste et table
  • Utiliser IN, NOT IN et NOT EXISTS
  • Écrire une sous-requête corrélée (référence à la requête externe)
  • Placer une sous-requête dans FROM, WHERE, SELECT ou HAVING

Rappel — Types de sous-requêtes

Type Ce qu'elle retourne Utilisation typique Exemple
Scalaire Une seule valeur (1 ligne × 1 col) WHERE, SELECT, HAVING WHERE salaire > (SELECT AVG(salaire) FROM employes)
Liste Une colonne de valeurs (n lignes × 1 col) IN, NOT IN, EXISTS WHERE id IN (SELECT employe_id FROM affectations)
Table Un ensemble de lignes et colonnes FROM (table dérivée) FROM (SELECT ... GROUP BY ...) AS stats

Sous-requête non-corrélée

Indépendante de la requête externe. Évaluée une seule fois.

-- La sous-requête ne dépend d'aucune table externe
SELECT nom, salaire
FROM employes
WHERE salaire > (SELECT AVG(salaire) FROM employes);

Sous-requête corrélée

Référence une colonne de la requête externe. Réévaluée pour chaque ligne.

-- e2.departement_id référence la ligne courante de e
SELECT e.nom, e.salaire
FROM employes e
WHERE e.salaire > (
    SELECT AVG(e2.salaire)
    FROM employes e2
    WHERE e2.departement_id = e.departement_id
);
Piège classique — NOT IN avec un NULL dans la liste : si la sous-requête retourne ne serait-ce qu'un NULL, NOT IN ne retourne aucune ligne (le prédicat devient inconnu pour toutes les lignes). Utilisez NOT EXISTS dès que la colonne testée peut contenir des NULL.
EXISTS vs IN — lequel choisir ?
EXISTS s'arrête dès qu'une ligne est trouvée (court-circuit) : il est souvent plus rapide quand la sous-requête renvoie beaucoup de lignes. IN est plus lisible pour des listes courtes et connues. Sur des colonnes indexées, les optimiseurs modernes produisent généralement des plans équivalents. En cas de doute sur les NULL, préférez toujours EXISTS/NOT EXISTS.

Positions possibles : WHERE (filtre), FROM (table dérivée — doit avoir un alias), SELECT (colonne calculée), HAVING (filtre après agrégat).

Q1 Intermédiaire

Listez les employés dont le salaire est strictement supérieur à la moyenne de l'ensemble des salaires. Affichez le nom, le prénom et le salaire, triés par salaire décroissant.

Dans le WHERE, comparez salaire à une sous-requête scalaire : (SELECT AVG(salaire) FROM employes). Cette sous-requête est évaluée une seule fois avant le filtrage des lignes.
Ctrl+Entrée pour exécuter

Solution — Q1

SELECT nom, prenom, salaire
FROM employes
WHERE salaire > (SELECT AVG(salaire) FROM employes)
ORDER BY salaire DESC;

La sous-requête scalaire est évaluée une seule fois et retourne une unique valeur (la moyenne globale). Toutes les lignes d'employes sont ensuite comparées à cette valeur — c'est une sous-requête non-corrélée.

Q2 Intermédiaire

Affichez le ou les département(s) ayant le budget le plus élevé (nom et budget). Si deux départements sont ex-aequo, ils doivent tous deux apparaître.

Comparez budget à (SELECT MAX(budget) FROM departements). Contrairement à ORDER BY budget DESC LIMIT 1, cette approche retourne tous les départements qui partagent le budget maximum.
Ctrl+Entrée pour exécuter

Solution — Q2

SELECT nom, budget
FROM departements
WHERE budget = (SELECT MAX(budget) FROM departements);

MAX retourne une valeur scalaire — cette sous-requête est donc non-corrélée et évaluée une fois. L'approche ORDER BY ... LIMIT 1 retournerait arbitrairement un seul gagnant même en cas d'ex-aequo.

Q3 Avancé

Listez les employés qui sont affectés au projet "Alpha". Affichez leur nom et prénom. Utilisez une sous-requête avec IN.

La sous-requête doit retourner les employe_id présents dans affectations pour le projet Alpha. Vous devrez joindre affectations et projets à l'intérieur de la sous-requête pour filtrer par nom de projet.
La sous-requête : SELECT a.employe_id FROM affectations a JOIN projets p ON p.id = a.projet_id WHERE p.nom = 'Alpha'
Ctrl+Entrée pour exécuter

Solution — Q3

SELECT nom, prenom
FROM employes
WHERE id IN (
    SELECT a.employe_id
    FROM affectations a
    JOIN projets p ON p.id = a.projet_id
    WHERE p.nom = 'Alpha'
);

La sous-requête de type liste retourne une colonne d'identifiants. La requête externe filtre ensuite les employés dont l'id appartient à cette liste. On aurait pu obtenir le même résultat avec un INNER JOIN + DISTINCT.

Q4 Avancé

Listez les employés qui ne sont affectés à aucun projet. Écrivez deux versions : une avec NOT IN et une avec NOT EXISTS.

Piège — NOT IN et les NULL : si affectations.employe_id contient un seul NULL, NOT IN retourne zéro ligne car la comparaison avec NULL est toujours inconnue. NOT EXISTS est immunisé contre ce problème et est généralement à préférer.
NOT IN : WHERE id NOT IN (SELECT employe_id FROM affectations)

NOT EXISTS : pour chaque employé e, vérifiez qu'il n'existe aucune ligne dans affectations où a.employe_id = e.id. C'est une sous-requête corrélée : elle référence e.id depuis la requête externe.
Ctrl+Entrée pour exécuter

Solution — Q4

-- Version 1 : NOT IN (simple, mais fragile face aux NULL)
SELECT nom, prenom
FROM employes
WHERE id NOT IN (SELECT employe_id FROM affectations);

-- Version 2 : NOT EXISTS (robuste, recommandée)
SELECT nom, prenom
FROM employes e
WHERE NOT EXISTS (
    SELECT 1
    FROM affectations a
    WHERE a.employe_id = e.id
);

La version NOT EXISTS est une sous-requête corrélée : elle est réévaluée pour chaque ligne de employes. Le SELECT 1 est une convention — seule l'existence d'une ligne importe, pas sa valeur. Sur nos données, les deux versions donnent le même résultat.

Q5 Avancé

Listez les employés dont le salaire est supérieur à la moyenne de salaire de leur propre département. Affichez le nom, le prénom, le salaire et l'identifiant du département, triés par département puis par salaire décroissant.

Il s'agit d'une sous-requête corrélée : la sous-requête calcule la moyenne du département de l'employé en cours d'évaluation. Elle est réévaluée pour chaque ligne.
Donnez un alias à la table externe : FROM employes e. La sous-requête doit calculer AVG(salaire) sur les employés du même département que e.
Dans la sous-requête, filtrez sur WHERE e2.departement_id = e.departement_id (où e2 est l'alias interne et e est la référence externe).
Structure complète : 
WHERE e.salaire > (
    SELECT AVG(e2.salaire)
    FROM employes e2
    WHERE e2.departement_id = e.departement_id
)
Ctrl+Entrée pour exécuter

Solution — Q5

SELECT e.nom, e.prenom, e.salaire, e.departement_id
FROM employes e
WHERE e.salaire > (
    SELECT AVG(e2.salaire)
    FROM employes e2
    WHERE e2.departement_id = e.departement_id
)
ORDER BY e.departement_id, e.salaire DESC;

La sous-requête est dite corrélée car e.departement_id vient de la requête externe. Elle est réévaluée pour chaque ligne de employes — une fois par employé. Certains SGBD peuvent l'optimiser en une seule passe avec un GROUP BY interne, mais la sémantique reste identique.

Q6 Avancé

Listez les projets ayant plus d'affectations que la moyenne du nombre d'affectations par projet. Affichez le nom du projet et son nombre d'affectations.

La sous-requête doit calculer la moyenne des comptages par groupe, ce qui nécessite une sous-requête dans le HAVING. L'ordre d'évaluation SQL est : FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY.
Commencez par grouper affectations par projet_id et compter les lignes avec COUNT(*). Joignez ensuite projets pour afficher le nom.
La condition de filtre va dans le HAVING : HAVING COUNT(*) > (SELECT ...). La sous-requête doit retourner le nombre moyen d'affectations par projet.
Pour la sous-requête : calculez d'abord les comptes par projet avec un GROUP BY, puis prenez la moyenne de ces comptes avec AVG : SELECT AVG(nb) FROM (SELECT COUNT(*) AS nb FROM affectations GROUP BY projet_id) AS stats
Ctrl+Entrée pour exécuter

Solution — Q6

SELECT p.nom, COUNT(*) AS nb_affectations
FROM projets p
JOIN affectations a ON a.projet_id = p.id
GROUP BY p.id, p.nom
HAVING COUNT(*) > (
    SELECT AVG(nb)
    FROM (
        SELECT COUNT(*) AS nb
        FROM affectations
        GROUP BY projet_id
    ) AS stats_par_projet
)
ORDER BY nb_affectations DESC;

La sous-requête imbriquée dans HAVING est non-corrélée et évaluée une seule fois. Elle calcule la moyenne du nombre d'affectations à partir d'une table dérivée (sous-requête dans le FROM) qui regroupe d'abord les affectations par projet.

Q7 Avancé

En utilisant une sous-requête dans le FROM (table dérivée), trouvez les départements dont le salaire moyen dépasse la moyenne globale des salaires. Affichez le nom du département et son salaire moyen arrondi à l'entier, trié par salaire moyen décroissant.

Syntaxe obligatoire : toute sous-requête dans un FROM doit avoir un alias (AS nom_alias), sinon la requête génère une erreur.
Créez une table dérivée dans le FROM qui calcule le salaire moyen par département. Quelque chose comme : FROM (SELECT departement_id, AVG(salaire) AS sal_moyen FROM employes GROUP BY departement_id) AS stats
Joignez cette table dérivée avec departements pour récupérer le nom du département. Filtrez ensuite avec un WHERE comparant sal_moyen à la moyenne globale.
La condition de filtre : WHERE stats.sal_moyen > (SELECT AVG(salaire) FROM employes)
Ctrl+Entrée pour exécuter

Solution — Q7

SELECT d.nom AS departement,
       ROUND(stats.sal_moyen) AS salaire_moyen
FROM (
    SELECT departement_id,
           AVG(salaire) AS sal_moyen
    FROM employes
    GROUP BY departement_id
) AS stats
JOIN departements d ON d.id = stats.departement_id
WHERE stats.sal_moyen > (SELECT AVG(salaire) FROM employes)
ORDER BY salaire_moyen DESC;

La table dérivée (stats) pré-agrège les salaires par département avant que la requête externe ne filtre et n'affiche les résultats. Cette approche évite de répéter le GROUP BY et facilite la lecture. Les CTE (WITH ... AS) offrent une syntaxe encore plus lisible pour ce type de construction.
