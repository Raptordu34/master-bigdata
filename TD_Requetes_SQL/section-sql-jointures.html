Niveau 3 — Jointures

Intermédiaire → Avancé ⏱ 40–50 min
Objectifs
  • Comprendre et distinguer INNER JOIN, LEFT JOIN, RIGHT JOIN et CROSS JOIN
  • Utiliser des alias de table pour lever l'ambiguïté sur les colonnes
  • Enchaîner des jointures sur 3 tables ou plus
  • Combiner jointures avec GROUP BY, COUNT et SUM

Rappel de cours — Les jointures SQL

Une jointure combine des lignes de deux tables (ou plus) en se basant sur une condition logique, généralement l'égalité entre une clé étrangère et une clé primaire. Le schéma de notre base :

employes     (id, nom, prenom, salaire, departement_id, date_embauche, poste)
departements (id, nom, budget, localisation)
projets      (id, nom, budget, date_debut, date_fin, departement_id)
affectations (employe_id, projet_id, role, heures)

Tableau des types de jointures

INNER JOIN — Intersection

Retourne uniquement les lignes qui ont une correspondance dans les deux tables. Toute ligne sans correspondance de l'un ou l'autre côté est exclue.

SELECT e.nom, d.nom AS dept
FROM employes e
INNER JOIN departements d ON e.departement_id = d.id;

Résultat : seuls les employés ayant un département existant.

LEFT JOIN — Tout A + correspondances B

Retourne toutes les lignes de la table de gauche (A), et les colonnes de la table de droite (B) valent NULL quand il n'y a pas de correspondance.

SELECT e.nom, d.nom AS dept
FROM employes e
LEFT JOIN departements d ON e.departement_id = d.id;

Résultat : tous les employés, même sans département.

RIGHT JOIN — Tout B + correspondances A

Symétrique du LEFT JOIN : retourne toutes les lignes de la table de droite (B), les colonnes de A valent NULL si pas de correspondance.

SELECT e.nom, d.nom AS dept
FROM employes e
RIGHT JOIN departements d ON e.departement_id = d.id;

Résultat : tous les départements, même sans employés. Équivalent à un LEFT JOIN en inversant les tables.

CROSS JOIN — Produit cartésien

Combine chaque ligne de A avec chaque ligne de B sans condition. Si A a n lignes et B m lignes, le résultat a n × m lignes.

SELECT e.nom, d.nom AS dept
FROM employes e
CROSS JOIN departements d;

Rarement utile en pratique sauf pour générer des combinaisons exhaustives.

La règle de la clé de jointure

La condition ON relie généralement une clé étrangère d'une table à la clé primaire de l'autre :

-- Syntaxe générale
FROM table1 t1
JOIN table2 t2 ON t1.cle_etrangere = t2.cle_primaire

-- Exemple concret
FROM employes e
JOIN departements d ON e.departement_id = d.id
--                     ^^^^^^^^^^^^^^^^^   ^^^^
--                     clé étrangère       clé primaire

Alias de table : indispensables

Les alias (e, d, p, a) sont déclarés juste après le nom de table et permettent de préfixer toute référence à une colonne :

FROM employes e          -- e = alias pour employes
JOIN departements d ...  -- d = alias pour departements
JOIN projets p ...       -- p = alias pour projets
JOIN affectations a ...  -- a = alias pour affectations
INNER est implicite : JOIN seul est équivalent à INNER JOIN. Ces deux requêtes sont identiques : 
FROM employes e INNER JOIN departements d ON e.departement_id = d.id
FROM employes e JOIN departements d ON e.departement_id = d.id
Par convention, on écrit souvent JOIN pour l'inner join et on est explicite pour les autres (LEFT JOIN, RIGHT JOIN).
Erreur classique — Jointure sans condition ON : Oublier la clause ON produit un produit cartésien silencieux. Avec 100 employés et 10 départements, vous obtiendrez 1 000 lignes au lieu de 100 ! 
-- MAUVAIS : produit cartésien explosif
SELECT * FROM employes, departements;

-- CORRECT : toujours spécifier la condition de jointure
SELECT * FROM employes e JOIN departements d ON e.departement_id = d.id;
Colonnes ambiguës : Quand deux tables ont une colonne avec le même nom (ex. nom dans employes et departements), il est obligatoire de la préfixer avec l'alias. Sans préfixe, SQL renvoie une erreur "ambiguous column name". 
-- MAUVAIS
SELECT nom FROM employes e JOIN departements d ON e.departement_id = d.id;
-- Erreur : nom est ambigu !

-- CORRECT
SELECT e.nom, d.nom AS nom_departement
FROM employes e JOIN departements d ON e.departement_id = d.id;
Q1 Fondamental

Affichez le nom et le prénom de chaque employé ainsi que le nom de son département. N'affichez que les employés qui ont un département.

Joignez employes et departements sur employes.departement_id = departements.id. Utilisez des alias de table (e et d) et renommez la colonne d.nom avec AS departement pour éviter l'ambiguïté.
Ctrl+Entrée pour exécuter

Solution — Q1

SELECT e.nom, e.prenom, d.nom AS departement
FROM employes e
INNER JOIN departements d ON e.departement_id = d.id
ORDER BY e.nom;

Les alias e et d sont indispensables ici : les deux tables ont une colonne nom. Sans préfixe, SQL ne saurait pas laquelle choisir et renverrait une erreur.

Q2 Fondamental

Affichez tous les employés avec le nom de leur département. Les employés sans département doivent aussi apparaître, avec NULL pour le nom du département.

Piège courant : Un INNER JOIN exclut silencieusement les lignes sans correspondance. Si vous attendez N lignes (autant que d'employés) et n'en obtenez que moins, c'est probablement parce que certains employés n'ont pas de département — l'INNER JOIN les a supprimés sans avertissement.
Remplacez INNER JOIN par LEFT JOIN. La table de gauche doit être employes pour que tous les employés soient conservés, même ceux sans département.
Ctrl+Entrée pour exécuter

Solution — Q2

SELECT e.nom, e.prenom, d.nom AS departement
FROM employes e
LEFT JOIN departements d ON e.departement_id = d.id
ORDER BY e.nom;

Dans notre base de données de démonstration, tous les employés ont un département, donc le résultat est identique à Q1. Mais en contexte réel, le LEFT JOIN révèle les lignes "orphelines" — précieux pour auditer la qualité des données.

Q3 Intermédiaire

Affichez tous les départements avec le nom et prénom de leurs employés. Les départements sans aucun employé doivent aussi apparaître (avec NULL pour les colonnes de l'employé). Triez par nom de département.

Cette fois, c'est departements qui doit être la table de gauche dans le LEFT JOIN. Comparez avec Q2 : on a inversé les rôles des deux tables.
Ctrl+Entrée pour exécuter

Solution — Q3

SELECT d.nom AS departement, e.nom, e.prenom
FROM departements d
LEFT JOIN employes e ON e.departement_id = d.id
ORDER BY d.nom;

Ici departements est la table de gauche — on garde tous les départements, même vides. C'est l'exact symétrique de Q2 : on a simplement échangé quelle table est "protégée" par le LEFT JOIN. Un département vide apparaîtra avec NULL dans les colonnes e.nom et e.prenom.

Q4 Intermédiaire

Affichez le nom et prénom de chaque employé, le nom du projet auquel il est affecté et le nombre d'heures d'affectation — en joignant 3 tables : employes, affectations et projets.

La table de liaison est affectations : elle contient employe_id (lien vers employes) et projet_id (lien vers projets). Partez de employes, puis joignez affectations, puis projets.
Structure de la clause FROM : 
FROM employes e
JOIN affectations a ON a.employe_id = e.id
JOIN projets p      ON p.id = a.projet_id
Il ne reste qu'à compléter le SELECT avec les colonnes voulues.
Ctrl+Entrée pour exécuter

Solution — Q4

SELECT e.nom, e.prenom, p.nom AS projet, a.heures
FROM employes e
JOIN affectations a ON a.employe_id = e.id
JOIN projets p      ON p.id = a.projet_id
ORDER BY e.nom, p.nom;

On enchaîne deux JOIN : SQL traite le résultat de la première jointure comme une table intermédiaire, sur laquelle on applique le second JOIN. Les alias (e, a, p) sont essentiels pour distinguer les colonnes homonymes entre les 3 tables.

Q5 Intermédiaire

Pour chaque département, affichez son nom, son budget et le nombre d'employés qu'il contient. Tous les départements doivent apparaître, y compris ceux sans aucun employé (qui afficheront 0). Triez par nombre d'employés décroissant.

Utilisez un LEFT JOIN (pour garder les départements sans employés) + GROUP BY sur le département + COUNT(e.id). Attention : COUNT(e.id) renvoie 0 quand e.id est NULL, ce qui n'est pas le cas de COUNT(*).
Ctrl+Entrée pour exécuter

Solution — Q5

SELECT d.nom, d.budget, COUNT(e.id) AS nb_employes
FROM departements d
LEFT JOIN employes e ON e.departement_id = d.id
GROUP BY d.id, d.nom, d.budget
ORDER BY nb_employes DESC;

COUNT(e.id) compte les valeurs non-NULL de e.id : pour un département sans employés, toutes les valeurs e.id seront NULL (issues du LEFT JOIN), donc COUNT retourne 0. Avec COUNT(*), on obtiendrait 1 (la ligne NULL est quand même comptée).

Q6 Intermédiaire

Affichez le nom de chaque projet, son budget, sa date de début et le nom du département porteur. Tous les projets doivent apparaître, même ceux sans département associé. Triez par budget de projet décroissant.

La table projets possède une colonne departement_id qui référence departements.id. Joignez ces deux tables. Pour garder tous les projets (même sans département), utilisez un LEFT JOIN depuis projets.
Ctrl+Entrée pour exécuter

Solution — Q6

SELECT p.nom AS projet, p.budget, p.date_debut,
       d.nom AS departement_porteur
FROM projets p
LEFT JOIN departements d ON p.departement_id = d.id
ORDER BY p.budget DESC;

Même schéma que Q2/Q3 : projets est la table de gauche, donc tous les projets apparaissent. Si un projet n'a pas de département associé, departement_porteur sera NULL.

Q7 Avancé

Affichez le nom, prénom de chaque employé et son nombre de projets affectés (nb_projets). Les employés sans aucune affectation doivent apparaître avec 0. Triez par nombre de projets décroissant, puis par nom.

Pour garder les employés sans affectation, utilisez un LEFT JOIN depuis employes vers affectations. Ensuite, un GROUP BY sur l'employé + COUNT sur les affectations permettra de compter les projets.
Utilisez COUNT(a.projet_id) et non COUNT(*) pour obtenir 0 pour les employés sans affectation. Groupez sur e.id, e.nom, e.prenom. 
FROM employes e
LEFT JOIN affectations a ON a.employe_id = e.id
GROUP BY e.id, e.nom, e.prenom
Ctrl+Entrée pour exécuter

Solution — Q7

SELECT e.nom, e.prenom, COUNT(a.projet_id) AS nb_projets
FROM employes e
LEFT JOIN affectations a ON a.employe_id = e.id
GROUP BY e.id, e.nom, e.prenom
ORDER BY nb_projets DESC, e.nom;

Le LEFT JOIN garantit que les employés sans affectation apparaissent dans le résultat. Pour ces employés, a.projet_id est NULL et COUNT(a.projet_id) retourne 0. On groupe sur e.id en plus de e.nom et e.prenom pour distinguer deux employés qui auraient le même nom et prénom.

Q8 Avancé

Affichez le top 3 des employés ayant le plus grand nombre d'heures totales affectées à des projets. Affichez leur nom, prénom et la somme de leurs heures (total_heures). Seuls les employés ayant au moins une affectation doivent apparaître.

Pour sommer les heures, utilisez SUM(a.heures) après un JOIN entre employes et affectations. Ici, un INNER JOIN suffit puisqu'on ne veut que les employés avec affectations.
Après le JOIN, groupez par employé (GROUP BY e.id, e.nom, e.prenom) et calculez SUM(a.heures) AS total_heures. Triez ensuite par total_heures DESC.
Pour limiter à 3 résultats, ajoutez LIMIT 3 à la fin de la requête. 
...
ORDER BY total_heures DESC
LIMIT 3;
Ctrl+Entrée pour exécuter

Solution — Q8

SELECT e.nom, e.prenom, SUM(a.heures) AS total_heures
FROM employes e
JOIN affectations a ON a.employe_id = e.id
GROUP BY e.id, e.nom, e.prenom
ORDER BY total_heures DESC
LIMIT 3;

L'ordre des clauses est essentiel : JOIN → GROUP BY → ORDER BY → LIMIT. SQL évalue d'abord la jointure (toutes les lignes d'affectations par employé), puis agrège par employé avec SUM, puis trie, puis coupe à 3 résultats. Un INNER JOIN est ici correct car les employés sans aucune affectation ne peuvent pas figurer dans le classement.

Pour aller plus loin
  • SELF JOIN : Listez les paires d'employés travaillant dans le même département. Un SELF JOIN joint une table avec elle-même en utilisant deux alias différents : FROM employes e1 JOIN employes e2 ON e1.departement_id = e2.departement_id AND e1.id < e2.id. La condition e1.id < e2.id évite les doublons (A-B et B-A) et les paires avec soi-même.
  • CROSS JOIN illustratif : Générez toutes les combinaisons possibles entre employés et projets (même ceux sans affectation réelle) pour construire une matrice de disponibilité : FROM employes e CROSS JOIN projets p. Attention au volume : n employés × m projets lignes.
  • Jointure + agrégat + filtre HAVING : Trouvez les employés affectés à plus de 2 projets en ajoutant HAVING COUNT(a.projet_id) > 2 après le GROUP BY de Q7.
  • Jointure sur 4 tables : Affichez le nom de l'employé, le nom de son département, le nom du projet et ses heures en enchaînant 4 tables : employes → affectations → projets → departements (via projets.departement_id).
  • Budget moyen par département : Quel département porte les projets les plus coûteux en moyenne ? Joignez projets et departements, groupez par département et calculez AVG(p.budget) AS budget_moyen_projets, puis triez par ordre décroissant.
