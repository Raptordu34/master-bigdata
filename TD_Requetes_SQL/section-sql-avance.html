Niveau 5 — SQL Avancé

Avancé ⏱ 50–70 min
Objectifs
  • Combiner des résultats avec UNION, UNION ALL, EXCEPT, INTERSECT
  • Créer des colonnes et des tris conditionnels avec CASE WHEN
  • Calculer des rangs et des agrégats par fenêtre : RANK(), ROW_NUMBER(), SUM() OVER
  • Structurer une requête complexe avec une CTE (WITH … AS)

Rappel — Opérateurs ensemblistes

UNION / UNION ALL

Empile les résultats de deux SELECT. Les colonnes doivent être en même nombre et de types compatibles.

  • UNION — élimine les doublons (coût supplémentaire)
  • UNION ALL — conserve tous les doublons, plus rapide

EXCEPT / INTERSECT

EXCEPT retourne les lignes du premier SELECT absentes du second.
INTERSECT retourne les lignes communes aux deux.

SELECT ville FROM clients
EXCEPT
SELECT ville FROM fournisseurs

Rappel — CASE WHEN

Syntaxe

CASE
    WHEN condition1 THEN valeur1
    WHEN condition2 THEN valeur2
    ELSE valeur_defaut
END AS alias

Usages

  • Dans SELECT — créer une colonne calculée
  • Dans ORDER BY — trier selon une logique personnalisée
  • Dans WHERE — filtre conditionnel

Les conditions sont évaluées dans l'ordre ; la première vraie gagne.

Erreur fréquente : oublier le mot-clé END à la fin du bloc CASE provoque une erreur de syntaxe. Si tous les cas possibles ne sont pas couverts et qu'il n'y a pas de ELSE, les lignes non correspondantes retournent NULL.

Rappel — Fonctions de fenêtre

Les fonctions de fenêtre (window functions) calculent une valeur pour chaque ligne en tenant compte d'un ensemble de lignes voisines — sans réduire le nombre de lignes (contrairement à GROUP BY).

fonction() OVER (PARTITION BY colonne ORDER BY colonne2)

Fonctions de rang

Fonction Comportement ex-aequo
RANK()1, 1, 3 — saute le rang suivant
DENSE_RANK()1, 1, 2 — ne saute pas
ROW_NUMBER()1, 2, 3 — toujours unique

Fonctions d'accès et d'agrégat

Fonction Usage
LAG(col)Valeur de la ligne précédente
LEAD(col)Valeur de la ligne suivante
SUM/AVG() OVERAgrégat cumulatif (avec ORDER BY)
ORDER BY dans OVER vs ORDER BY final : le ORDER BY placé à l'intérieur de OVER(…) contrôle l'ordre de calcul de la fenêtre (ex. pour un cumul). Il est indépendant du ORDER BY de fin de requête qui contrôle l'ordre d'affichage des lignes dans le résultat.

Rappel — CTE (Common Table Expression)

Syntaxe

WITH nom_cte AS (
    SELECT ...
    FROM ...
)
SELECT *
FROM nom_cte
WHERE ...;

Avantages

  • Améliore la lisibilité en nommant les étapes intermédiaires
  • Permet de réutiliser un sous-résultat plusieurs fois dans la même requête
  • Équivalent à une sous-requête dans FROM, mais plus clair
Q1 Intermédiaire

Créez une liste unifiée en une seule colonne valeur contenant, sans doublons, les noms des employés et les localisations des départements. Triez le résultat par ordre alphabétique.

Ctrl+Entrée pour exécuter

Solution — Q1

SELECT nom AS valeur FROM employes
UNION
SELECT localisation AS valeur FROM departements
ORDER BY valeur;

UNION déduplique automatiquement. Si "Paris" apparaît dans plusieurs départements, il n'apparaît qu'une fois dans le résultat. L'alias de colonne (AS valeur) se définit dans le premier SELECT ; le second n'a pas besoin d'alias.

Q2 Intermédiaire

Reprenez la même liste unifiée, mais cette fois avec UNION ALL pour conserver les doublons. Comptez ensuite le nombre total de lignes retournées dans chaque cas et comparez avec UNION.

Astuce : encapsulez la requête dans un SELECT COUNT(*) FROM (…) pour compter.

Ctrl+Entrée pour exécuter

Solution — Q2

-- Avec doublons
SELECT nom AS valeur FROM employes
UNION ALL
SELECT localisation AS valeur FROM departements
ORDER BY valeur;

-- Compter avec UNION ALL
SELECT COUNT(*) AS total_union_all FROM (
    SELECT nom AS valeur FROM employes
    UNION ALL
    SELECT localisation AS valeur FROM departements
);

-- Compter avec UNION (sans doublons)
SELECT COUNT(*) AS total_union FROM (
    SELECT nom AS valeur FROM employes
    UNION
    SELECT localisation AS valeur FROM departements
);

UNION ALL est systématiquement plus rapide que UNION car il ne réalise pas de tri pour dédupliquer. À préférer quand les doublons sont impossibles ou sans importance.

Q3 Intermédiaire

Affichez le nom, prénom et salaire de chaque employé avec une colonne categorie calculée selon les règles suivantes :

  • Salaire ≥ 65 000 → 'Senior'
  • Salaire ≥ 45 000 → 'Confirmé'
  • Sinon → 'Junior'

Triez par salaire décroissant.

Erreur fréquente : inverser l'ordre des conditions dans CASE WHEN. Si vous mettez WHEN salaire >= 45000 avant WHEN salaire >= 65000, tous les salaires ≥ 65 000 seront classés "Confirmé" et jamais "Senior", car la première condition vraie s'applique.
Ctrl+Entrée pour exécuter

Solution — Q3

SELECT
    nom,
    prenom,
    salaire,
    CASE
        WHEN salaire >= 65000 THEN 'Senior'
        WHEN salaire >= 45000 THEN 'Confirmé'
        ELSE 'Junior'
    END AS categorie
FROM employes
ORDER BY salaire DESC;

Les conditions sont évaluées de haut en bas. Dès qu'une est vraie, les suivantes sont ignorées. Le END ferme obligatoirement le bloc CASE.

Q4 Avancé

Affichez tous les employés (nom, prénom, salaire) triés d'abord par catégorie de salaire (Senior en premier, puis Confirmé, puis Junior), puis par nom alphabétique au sein de chaque catégorie. Utilisez un CASE WHEN directement dans la clause ORDER BY.

Ctrl+Entrée pour exécuter

Solution — Q4

SELECT nom, prenom, salaire
FROM employes
ORDER BY
    CASE
        WHEN salaire >= 65000 THEN 1
        WHEN salaire >= 45000 THEN 2
        ELSE 3
    END,
    nom ASC;

On attribue une valeur numérique à chaque catégorie pour piloter l'ordre. Le tri alphabétique secondaire (nom ASC) s'applique au sein de chaque groupe numérique.

Q5 Avancé

Pour chaque employé, calculez son rang par salaire décroissant au sein de son département (colonne rang). Affichez le nom, prénom, departement_id, salaire et rang, trié par département puis par rang.

Erreur fréquente : confondre RANK() et ROW_NUMBER(). Si deux employés ont le même salaire, RANK() leur attribue le même rang (1, 1, 3), tandis que ROW_NUMBER() attribue un numéro arbitraire unique (1, 2, 3).
La fonction de fenêtre se place dans le SELECT : RANK() OVER (PARTITION BY … ORDER BY …) AS rang
Syntaxe complète : RANK() OVER (PARTITION BY departement_id ORDER BY salaire DESC) AS rang
Ctrl+Entrée pour exécuter

Solution — Q5

SELECT
    nom,
    prenom,
    departement_id,
    salaire,
    RANK() OVER (
        PARTITION BY departement_id
        ORDER BY salaire DESC
    ) AS rang
FROM employes
ORDER BY departement_id, rang;

PARTITION BY departement_id remet le compteur à 1 pour chaque département. Le ORDER BY final trie l'affichage ; le ORDER BY dans OVER détermine le calcul du rang.

Q6 Avancé

Numérotez les employés par date d'embauche croissante au sein de chaque département (colonne num_embauche). Affichez le nom, prénom, departement_id, date_embauche et ce numéro.

ROW_NUMBER() ne prend pas d'argument, contrairement à SUM(col). Utilisez ROW_NUMBER() OVER (PARTITION BY … ORDER BY …).
ROW_NUMBER() OVER (PARTITION BY departement_id ORDER BY date_embauche ASC) AS num_embauche
Ctrl+Entrée pour exécuter

Solution — Q6

SELECT
    nom,
    prenom,
    departement_id,
    date_embauche,
    ROW_NUMBER() OVER (
        PARTITION BY departement_id
        ORDER BY date_embauche ASC
    ) AS num_embauche
FROM employes
ORDER BY departement_id, num_embauche;

ROW_NUMBER() garantit un numéro unique même en cas d'ex-aequo sur date_embauche. Le premier employé embauché dans chaque département reçoit le numéro 1.

Q7 Avancé

Pour chaque affectation, affichez le projet_id, l'employe_id, les heures et le cumul des heures par projet (somme courante, triée par employe_id croissant au sein de chaque projet). Nommez cette colonne cumul_heures.

Avec ORDER BY dans OVER, SUM() devient une somme cumulée ligne par ligne. Sans ORDER BY, elle retourne le total complet de la partition.
Commencez par SUM(heures) OVER (PARTITION BY projet_id …) AS cumul_heures.
Syntaxe complète : SUM(heures) OVER (PARTITION BY projet_id ORDER BY employe_id ASC) AS cumul_heures
Ctrl+Entrée pour exécuter

Solution — Q7

SELECT
    projet_id,
    employe_id,
    heures,
    SUM(heures) OVER (
        PARTITION BY projet_id
        ORDER BY employe_id ASC
    ) AS cumul_heures
FROM affectations
ORDER BY projet_id, employe_id;

La dernière ligne de chaque projet affiche le total de toutes les heures du projet. On peut vérifier avec un SELECT projet_id, SUM(heures) FROM affectations GROUP BY projet_id.

Q8 Avancé

Utilisez une CTE pour calculer d'abord le salaire moyen par département, puis sélectionnez les employés dont le salaire est strictement supérieur à la moyenne de leur propre département. Affichez le nom, prénom, salaire, departement_id et le salaire moyen du département.

Structurez votre requête avec WITH moy_dept AS (SELECT departement_id, AVG(salaire) AS salaire_moyen FROM employes GROUP BY departement_id).
Après la CTE, faites une jointure : SELECT e.nom, … FROM employes e JOIN moy_dept m ON e.departement_id = m.departement_id WHERE …
Condition de filtre : WHERE e.salaire > m.salaire_moyen
Ctrl+Entrée pour exécuter

Solution — Q8

WITH moy_dept AS (
    SELECT
        departement_id,
        AVG(salaire) AS salaire_moyen
    FROM employes
    GROUP BY departement_id
)
SELECT
    e.nom,
    e.prenom,
    e.salaire,
    e.departement_id,
    ROUND(m.salaire_moyen, 2) AS salaire_moyen_dept
FROM employes e
JOIN moy_dept m ON e.departement_id = m.departement_id
WHERE e.salaire > m.salaire_moyen
ORDER BY e.departement_id, e.salaire DESC;

La CTE moy_dept est évaluée une seule fois, puis réutilisée comme une table virtuelle. Sans CTE, il faudrait une sous-requête dans le FROM ou une sous-requête corrélée dans le WHERE, moins lisibles.

Pour aller plus loin
  • Utilisez LAG(salaire, 1) OVER (PARTITION BY departement_id ORDER BY salaire DESC) pour afficher, à côté de chaque employé, le salaire de l'employé mieux payé juste au-dessus dans son département.
  • Utilisez LEAD(salaire, 1) OVER (…) pour afficher le salaire de l'employé juste en dessous.
  • Répartissez les employés en quartiles de salaire avec NTILE(4) OVER (ORDER BY salaire DESC) AS quartile.
  • Explorez la clause ROWS BETWEEN pour contrôler précisément la fenêtre de calcul : par exemple SUM(heures) OVER (PARTITION BY projet_id ORDER BY employe_id ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) calcule la somme glissante sur les 3 dernières lignes.
