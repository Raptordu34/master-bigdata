Exercice 1 — Sélection d'index

Intermédiaire ◎ 12 pts ⏱ 25–40 min
Objectifs
  • Identifier les attributs candidats à l'indexation à partir d'un ensemble de requêtes
  • Distinguer les types d'index B-tree, Bitmap et Hash selon les prédicats et la cardinalité
  • Sélectionner et prioriser les 3 meilleurs index à créer sur un schéma donné

Prérequis : Cours — Structures d'accès physique et index · Cours — Optimisation de requêtes

Contexte — Base de données RH d'entreprise

Vous travaillez sur la base de données d'une entreprise. Le DBA constate que plusieurs requêtes fréquentes sont lentes, et vous demande d'identifier les index à créer pour améliorer les performances. Le schéma logique est le suivant (le préfixe # désigne une clé étrangère) :

TableAttributs
Employe NSS (PK), Nom, Prénom, DateNaissance, Adresse, Salaire, #N°Dep, #Supérieur
Projet N°Pro (PK), NomP, #N°Dep, Directeur
Travaille #NSS, #N°Pro (PK composite), Heures

Les 7 requêtes suivantes sont exécutées régulièrement :

#Description de la requête
R1Nom et adresse de Juliette Rochat
R2Nom et adresse des employés qui travaillent au département de recherche
R3Nom et prénom des employés dont le supérieur est Juliette Rochat
R4Nom des projets sur lesquels travaillent, plus de 10 heures, les employés habitant à Genève
R5Nom et prénom des employés qui travaillent sur le même projet que Jean Müller et Annie Grandjean
R6Nom des projets sur lesquels travaillent à la fois Jean Müller et Annie Grandjean
R7Nom et prénom des employés qui ne travaillent sur aucun projet
Note : Aucun index secondaire n'existe pour l'instant — seules les clés primaires sont indexées automatiquement par le SGBD. Toutes les requêtes reposent donc actuellement sur des parcours séquentiels (Seq Scan) sur les tables concernées.
Un index est une structure de données auxiliaire construite sur un ou plusieurs attributs d'une table. Il permet au moteur de localiser les tuples correspondant à un prédicat en O(log n) au lieu de O(n), sans parcourir toutes les lignes.
  • Un index sur Employe(Nom) permet de trouver "Rochat" en quelques comparaisons, quelle que soit la taille de la table.
  • Un index a un coût en écriture : chaque INSERT, UPDATE ou DELETE doit mettre à jour toutes les structures d'index associées à la table.
  • Les clés primaires sont automatiquement indexées — il est inutile de recréer un index dessus.

B-tree (arbre B+) : structure arborescente équilibrée, type par défaut dans presque tous les SGBD. Efficace pour les filtres d'égalité (=), d'intervalle (<, >, BETWEEN) et les tris (ORDER BY). Convient à tout type de cardinalité.

Bitmap : représente chaque valeur distincte par un vecteur de bits — un bit par tuple indique si ce tuple possède cette valeur. Très compact et efficace pour les attributs à faible cardinalité (peu de valeurs distinctes : département, ville, statut). Peu adapté aux mises à jour fréquentes car reconstruire un bitmap est coûteux.

Hash : calcule une fonction de hachage sur la valeur clé et stocke le résultat dans des buckets. Optimal uniquement pour les tests d'égalité stricte (=). Complètement inutilisable pour les plages (>, BETWEEN) ou les tris.

Règle de décision

Pour choisir le type, posez deux questions : (1) Quel prédicat porte sur cet attribut ? — égalité seule → Hash ou B-tree ; plage ou tri → B-tree obligatoire. (2) Quelle est la cardinalité ? — peu de valeurs distinctes → Bitmap avantageux.

Exemple

Pour R1 ("Nom et adresse de Juliette Rochat"), sans index, le moteur parcourt chaque ligne de Employe. Un index sur (Nom, Prénom) réduit l'accès à quelques nœuds d'un arbre B+, indépendamment du volume.

Analogie

Intuition : un index sur Employe(Nom) ressemble à un annuaire téléphonique trié par nom. Sans annuaire, il faut appeler tous les abonnés un par un. Avec l'annuaire, on ouvre directement à la lettre R.

Q1 4 pts Fondamental

Pour chacune des requêtes R1 à R7, identifiez les attributs candidats à l'indexation. Pour chaque candidat, précisez la ou les requêtes qui en bénéficieraient et le type de prédicat impliqué (égalité, intervalle, jointure, existence).

Piège courant : Proposer un index sur tous les attributs de toutes les tables. Un index ralentit les INSERT, UPDATE et DELETE (la structure doit être mise à jour à chaque modification). Ne retenir que les attributs réellement utilisés dans WHERE, JOIN … ON, ORDER BY ou dans un test d'existence (NOT EXISTS).
Analysez chaque requête une par une et notez l'attribut impliqué dans le prédicat :
  • R1 filtre sur Nom et Prénom dans Employe.
  • R2 fait une jointure entre Employe et un département via N°Dep.
  • R3 filtre sur Supérieur dans Employe (auto-jointure ou sous-requête).
  • R7 vérifie l'absence de tuple dans Travaille pour un employé donné — un index sur Travaille(NSS) accélère cette vérification (NOT EXISTS).
Synthèse des candidats les plus fréquents :
  • Employe(Nom, Prénom) → R1, R3, R5
  • Employe(N°Dep) → R2
  • Employe(Supérieur) → R3 (FK vers NSS du supérieur)
  • Employe(Adresse) → R4 (filtre sur ville)
  • Travaille(NSS) → R5, R6, R7 (jointure et NOT EXISTS)
  • Travaille(N°Pro) → R4, R5, R6 (jointure)
  • Travaille(Heures) → R4 (filtre d'intervalle > 10)
  • Projet(N°Dep) → R2 (jointure côté Projet)

Solution — Q1

Employe(N°Dep) R2 Égalité / jointure avec Département
Employe(Supérieur) R3 Égalité — FK vers NSS du supérieur

Les clés primaires (NSS, N°Pro, la PK composite de Travaille) sont déjà indexées automatiquement — inutile de les recréer explicitement.

Q2 3 pts Intermédiaire

Parmi les candidats identifiés en Q1, précisez pour chacun le type d'index le plus adapté : B-tree, Bitmap ou Hash. Justifiez votre choix en vous appuyant sur le type de prédicat et la cardinalité estimée de l'attribut.

Piège courant : Choisir Bitmap pour un attribut à haute cardinalité (comme Nom ou NSS). Un index Bitmap est efficace uniquement si le nombre de valeurs distinctes est faible (généralement moins de quelques centaines). Pour Nom, chaque employé a un nom différent : un Bitmap générerait autant de vecteurs que d'individus, occupant plus d'espace que la table elle-même.
Deux questions suffisent pour décider :
  1. Prédicat : le filtre est-il un intervalle ou utilisé dans un tri ? Si oui, B-tree obligatoire (Heures > 10 → B-tree).
  2. Cardinalité : combien de valeurs distinctes ? Peu de valeurs (quelques dizaines de départements, quelques villes) → Bitmap possible. Beaucoup de valeurs (noms, NSS, heures continues) → B-tree.
Employe(N°Dep) a peu de valeurs distinctes → Bitmap. Employe(Nom, Prénom) a une cardinalité très haute → B-tree.

Note moteur : PostgreSQL ne supporte pas les index Bitmap créables explicitement — il les utilise en interne lors de la combinaison de plusieurs index B-tree (Bitmap AND/OR). En Oracle ou dans les entrepôts de données (Redshift, Hive, Teradata), les BITMAP INDEX sont explicitement disponibles et très utilisés.
Q3 5 pts Avancé

Parmi tous les candidats identifiés, sélectionnez les 3 meilleurs index à créer en priorité. Pour chaque index retenu, justifiez le choix en indiquant :

  • le nombre de requêtes couvertes (R1–R7),
  • la sélectivité estimée (proportion de tuples conservés après filtrage),
  • la commande DDL CREATE INDEX correspondante.
Piège courant : Retenir un index parce qu'il intervient dans une seule requête peu fréquente, au détriment d'un autre candidat couvrant plus de requêtes. La priorité va aux index à fort impact global : un index qui accélère 3 requêtes sur 7 est presque toujours préférable à un index qui n'accélère qu'une seule requête rare.
Comptez pour chaque candidat le nombre de requêtes (R1–R7) dans lesquelles il intervient :
  • Employe(Nom, Prénom) : R1, R3, R5 → 3 requêtes
  • Travaille(NSS) : R5, R6, R7 → 3 requêtes
  • Travaille(N°Pro) : R4, R5, R6 → 3 requêtes
Ces trois candidats couvrent chacun 3 requêtes — c'est le maximum observable ici.
La sélectivité affine le classement entre candidats de même couverture :
  • Employe(Nom, Prénom) : très sélectif — une recherche par nom retourne 1 ou 2 tuples sur des milliers d'employés.
  • Travaille(NSS) : sélectif si on cherche les projets d'un employé précis ; indispensable pour R7 (test NOT EXISTS par employé).
  • Pour R4, un index composite (N°Pro, Heures) est préférable à deux index séparés : la colonne d'égalité (N°Pro) en premier, puis la plage (Heures), pour que le moteur puisse balayer uniquement les entrées pertinentes d'un seul parcours B-tree.

Méthode — Sélectionner les meilleurs index

  • Lister tous les candidats et attribuer à chacun un score de couverture (nombre de requêtes où il intervient).
  • Évaluer la sélectivité : un filtre sur Nom = 'Rochat' élimine presque tous les tuples → très sélectif → gain maximal.
  • Repérer les candidats qui cumulent couverture élevée ET sélectivité élevée → priorité maximale.
  • Vérifier si deux candidats proches (même table, mêmes requêtes) peuvent être fusionnés en un index composite pour économiser une structure.
  • Limiter le nombre total d'index pour ne pas pénaliser les INSERT et UPDATE fréquents.

Solution — Q3

Index 1 — Recherche par nom dans Employe

-- Couvre R1, R3, R5 — sélectivité très haute (1 ou 2 tuples attendus)
CREATE INDEX idx_employe_nom_prenom
    ON Employe (Nom, Prénom);

Index 2 — Jointure et NOT EXISTS sur Travaille(NSS)

-- Couvre R5, R6, R7 — jointure fréquente + test NOT EXISTS pour R7
CREATE INDEX idx_travaille_nss
    ON Travaille (NSS);

Index 3 — Index composite sur Travaille(N°Pro, Heures)

-- Couvre R4 (filtre Heures > 10) et R5/R6 (jointure sur N°Pro)
-- N°Pro en premier (prédicat d'égalité) puis Heures (plage ≥ 10)
CREATE INDEX idx_travaille_npro_heures
    ON Travaille (N°Pro, Heures);
Index retenuRequêtes couvertesSélectivitéGain attendu
Employe(Nom, Prénom) R1, R3, R5 Très haute Seq Scan → Index Scan ; 1–2 tuples retournés
Travaille(NSS) R5, R6, R7 Haute Jointure en O(log n) ; NOT EXISTS rapide par employé
Travaille(N°Pro, Heures) R4, R5, R6 Haute sur N°Pro Évite le Seq Scan sur Travaille pour filtrer Heures > 10

Contre-exemple

Un index sur Employe(Salaire) n'intervient dans aucune des 7 requêtes. Le créer alourdit chaque INSERT et UPDATE sur Employe sans aucun bénéfice en lecture pour ces requêtes.

Checklist avant validation :
  • Chaque index est-il utilisé dans au moins une des requêtes cibles ?
  • L'ordre des colonnes dans les index composites est-il justifié (égalité avant plage) ?
  • Le nombre total d'index reste-t-il raisonnable par rapport au volume d'écritures ?
