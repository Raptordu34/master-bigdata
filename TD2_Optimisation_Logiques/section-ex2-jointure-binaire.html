Exercice 2 — Index de jointure binaire

Intermédiaire ◎ 16 pts ⏱ 40–55 min
Objectifs
  • Distinguer table de faits et tables de dimension dans un schéma en étoile
  • Calculer les facteurs de sélectivité des prédicats de sélection
  • Concevoir des index de jointure binaire (IJB / Bitmap Join Index)
  • Décrire la séquence d'opérations bit-à-bit qui remplace un parcours de table

Prérequis : Optimisation logique · Index et structures d'accès

Requête Q à optimiser : 
SELECT Count(*)
FROM   Student, Grades, Course
WHERE  Level     = 'MSC'
  AND  Student.StudentId = Grades.StudentId
  AND  Course.CourId     = Grades.CourId
  AND  Dept      = 'Stat';

Principe : un index de jointure binaire (IJB, ou Bitmap Join Index) est construit sur la table de faits mais indexe des attributs issus d'une table de dimension. Il pré-calcule le résultat de la jointure et l'encode sous forme de bitmap — un vecteur de bits indexé sur les RID de la table de faits.

Structure : pour chaque valeur distincte de l'attribut de dimension (ex. Level='MSC'), l'index stocke un vecteur de n bits où n = nombre de tuples dans la table de faits. Le bit i vaut 1 si le tuple de rang i dans Grades est lié (via la jointure) à un tuple de dimension satisfaisant le prédicat.

  • Avantage clé : les opérations AND/OR entre bitmaps sont ultrarapides — gérées nativement par le CPU.
  • Cas d'usage : entrepôts de données (Data Warehouses), requêtes analytiques avec plusieurs filtres sur dimensions.
  • Limite : coût élevé en mise à jour — chaque INSERT/UPDATE/DELETE invalide les bitmaps concernés.

Repère

Un IJB est inter-table : il encode une jointure, pas seulement une colonne locale. C'est ce qui le distingue d'un simple index bitmap sur une colonne de la table de faits.

Exemple

Si Grades contient 26 tuples (RID 1..26) et que 15 correspondent à des étudiants MSC, le bitmap IJB[Level='MSC'] contient 15 bits à 1 et 11 bits à 0.

Pour compter les étudiants MSC inscrits en cours Stat, il suffit de faire IJB[Level='MSC'] AND IJB[Dept='Stat'] et de compter les bits à 1 — sans toucher à la table Grades.

Analogie

Image mentale : imaginez une feuille de pointage par critère. La feuille "MSC" coche les lignes d'inscription des étudiants MSC. La feuille "Stat" coche les inscriptions dans un cours Stat.

Superposer les deux feuilles en lumière (AND logique) révèle immédiatement les cases cochées sur les deux — sans relire les dossiers originaux.

Q1 2 pts Fondamental

Identifiez dans la requête Q : (a) la table de faits, (b) les tables de dimension, (c) les prédicats de sélection (filtre sur une valeur scalaire) et (d) les prédicats de jointure.

Piège courant : confondre un prédicat de jointure (A.id = B.id) avec un prédicat de sélection (colonne = valeur). Seuls les prédicats de sélection portant sur des tables de dimension peuvent servir de clé dans un IJB.
La table de faits est celle qui contient les mesures (ici : la note). Les tables de dimension décrivent les axes d'analyse (qui a étudié quoi). Les prédicats Level = 'MSC' et Dept = 'Stat' portent chacun sur la colonne d'une table de dimension différente.

Q2 3 pts Fondamental

Calculez le facteur de sélectivité de chacun des deux prédicats de sélection : Level = 'MSC' et Dept = 'Stat'. Appuyez-vous sur les données fournies.

Rappel : fs(p) = nombre de tuples satisfaisant p / nombre total de tuples dans la table source.

Piège courant : calculer le facteur de sélectivité sur la table de faits (Grades) plutôt que sur la table de dimension concernée. Le prédicat Level = 'MSC' se mesure sur Student, et non sur Grades.
Comptez combien d'étudiants ont Level = 'MSC' dans la table Student, puis divisez par le nombre total d'étudiants. Faites de même pour Dept = 'Stat' dans Course.
Dans Student : Carol (303), Ahmad (404), Philip (505) ont Level='MSC' — soit 3 étudiants sur 6. Dans Course : M444 et S466 ont Dept='Stat' — soit 2 cours sur 7.

Solution — Q2

PrédicatTable sourceTuples satisfaisantsTotalFacteur fs

Estimation du nombre de tuples de Grades qualifiants (en supposant les prédicats indépendants) :

|Grades| × fs(MSC) × fs(Stat) = 26 × 1/2 × 2/7 ≈ 3,7 tuples

L'estimation donne ≈ 4 ; la valeur exacte est 2. L'écart est normal : l'hypothèse d'indépendance est une approximation.

Q3 3 pts Intermédiaire

Proposez un ou plusieurs index de jointure binaire (IJB) pour optimiser la requête Q. Pour chaque index, précisez : la table d'ancrage, l'attribut de dimension indexé et la jointure encodée.

Piège courant : créer un index ordinaire sur Student.Level ou Course.Dept. Un tel index ne traverse pas la jointure — il faut un IJB ancré sur la table de faits qui pré-calcule la jointure.
Il faut un IJB par prédicat de sélection, chacun ancré sur Grades. Le premier encode la jointure Grades.StudentId = Student.StudentId et indexe Student.Level. Le second encode Grades.CourId = Course.CourId et indexe Course.Dept.

Solution — Q3

Deux IJB suffisent pour couvrir les deux prédicats de la requête Q :

-- IJB 1 : ancré sur Grades, encode la jointure avec Student, indexe Level
CREATE BITMAP INDEX ijb_grade_level
    ON Grades (Student.Level)
    FROM Grades g, Student s
    WHERE g.StudentId = s.StudentId;

-- IJB 2 : ancré sur Grades, encode la jointure avec Course, indexe Dept
CREATE BITMAP INDEX ijb_grade_dept
    ON Grades (Course.Dept)
    FROM Grades g, Course c
    WHERE g.CourId = c.CourId;
IndexTable d'ancrageAttribut indexéJointure encodée
ijb_grade_levelGradesStudent.LevelGrades.StudentId = Student.StudentId
ijb_grade_deptGradesCourse.DeptGrades.CourId = Course.CourId

La syntaxe CREATE BITMAP INDEX … FROM … WHERE est propre à Oracle. D'autres SGBD proposent des mécanismes équivalents via des index sur colonnes calculées ou des index de couverture.

Q6 1 pt Avancé

Quel est le nombre de tuples de la table des faits Grades que l'optimiseur doit physiquement lire pour répondre à la requête Q grâce aux IJB ? Comparez à ce qu'aurait coûté une exécution naïve sans index.

Piège courant : répondre "2 tuples" parce qu'il y a 2 lignes qualifiantes. Avec un COUNT(*) et des IJB, l'optimiseur compte les bits à 1 dans le vecteur résultat sans accéder aux tuples de la table.
Pour un COUNT(*), l'optimiseur n'a besoin d'aucune valeur de colonne de Grades — il lui suffit de compter les bits à 1 dans le bitmap AND résultat. Les IJB ont déjà tout résolu.

Solution — Q6

Avec IJB
Tuples de Grades lus physiquement : 0.
L'optimiseur consulte uniquement les bitmaps en mémoire. Pour COUNT(*), il suffit de compter les bits à 1 dans le vecteur AND — aucun accès aux blocs de données de Grades.

L'optimiseur parcourt les 26 tuples de Grades, effectue les jointures avec Student (6 tuples) et Course (7 tuples), puis applique les filtres. Lectures minimales : 26 + 6 + 7 = 39 tuples.
