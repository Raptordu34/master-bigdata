Exercice 3 — Rule Based Approach (RBA)

Intermédiaire ◎ 20 pts ⏱ 45–60 min
Objectifs
  • Traduire une requête SQL en expression d'algèbre relationnelle naïve
  • Appliquer les règles de réécriture du RBA pas à pas (cascade, push-down, synthèse de jointure)

Les quatre requêtes à optimiser sont les suivantes. Elles partagent le même schéma et illustrent des situations de complexité croissante. 

-- Requête 1 : jointure binaire + une sélection
SELECT Name, Grade
FROM   Student, Grades
WHERE  Student.StudId = Grades.StudId
  AND  Level = 'MSC';

-- Requête 2 : triple jointure + une sélection
SELECT Name, Dept, Grade
FROM   Student, Grades, Course
WHERE  Student.StudId = Grades.StudId
  AND  Grades.CourId  = Course.CourId
  AND  Level = 'MSC';

-- Requête 3 : triple jointure + deux sélections (tables différentes)
SELECT Name, Dept, Grade
FROM   Student, Grades, Course
WHERE  Student.StudId = Grades.StudId
  AND  Grades.CourId  = Course.CourId
  AND  Level = 'MSC'
  AND  Dept  = 'Math';

-- Requête 4 : agrégat + deux sélections + triple jointure
SELECT Count(*)
FROM   Student, Grades, Course
WHERE  Student.StudId = Grades.StudId
  AND  Grades.CourId  = Course.CourId
  AND  Level = 'MSC'
  AND  Dept  = 'Stat';

Le Rule Based Approach (approche dirigée par des règles) optimise une expression d'algèbre relationnelle en appliquant des équivalences algébriques dans un ordre prescrit, sans tenir compte des statistiques réelles. L'objectif est de réduire la taille des relations intermédiaires le plus tôt possible.

Les règles principales, dans l'ordre d'application habituel :

  • R1 — Cascade de sélections : σC₁ ∧ C₂(E) ≡ σC₁(σC₂(E)) — on scinde une conjonction en sélections imbriquées.
  • R2 — Commutativité des sélections : σC₁(σC₂(E)) ≡ σC₂(σC₁(E)) — on peut réordonner les sélections.
  • R3 — Push-down sur produit cartésien : Si C ne porte que sur R : σC(R × S) ≡ σC(R) × S — on descend la sélection avant le produit.
  • R4 — Synthèse de jointure : σR.A=S.B(R × S) ≡ R ⋈R.A=S.B S — un produit cartésien suivi d'une sélection d'équi-jointure devient une jointure.
  • R5 — Push-down sur jointure : Si C ne porte que sur R : σC(R ⋈ S) ≡ σC(R) ⋈ S — on applique la sélection avant la jointure.
  • R6 — Commutativité des jointures : R ⋈ S ≡ S ⋈ R — on peut choisir l'ordre de la jointure.
  • R7 — Associativité des jointures : (R ⋈ S) ⋈ T ≡ R ⋈ (S ⋈ T) — on peut regrouper différemment.
  • R8 — Cascade de projections : πL₁(πL₂(E)) ≡ πL₁(E) si L₁ ⊆ L₂ — les projections redondantes se suppriment.

Heuristique principale

Toujours appliquer les sélections aussi tôt que possible (avant les jointures ou produits cartésiens). Plus une table est petite avant la jointure, moins la jointure coûte cher.

Pour chaque requête, suivre ces cinq étapes dans l'ordre :

  • Écriture naïve : traduire le FROM (tables) en produits cartésiens, puis le WHERE en une seule grande sélection conjonctive.
  • Cascade (R1) : décomposer la sélection conjonctive en sélections imbriquées individuelles.
  • Push-down (R2 + R3) : faire descendre chaque sélection qui porte sur une seule table jusqu'à ce qu'elle soit directement au-dessus de cette table.
  • Synthèse de jointure (R4) : remplacer chaque produit cartésien + sélection d'égalité de colonnes par une jointure naturelle ou conditionnelle.
  • Push projections (R8) : si des colonnes ne sont plus nécessaires après la jointure, ajouter des projections intermédiaires pour réduire la largeur des tuples.

Analogie

Cuisiner vs préparer : lire une recette naïvement, c'est comme mettre tous les ingrédients dans la casserole avant de les éplucher. Le push-down, c'est éplucher et sélectionner les bons légumes avant de les ajouter — le volume dans la casserole diminue dès le départ.

Exemple — lecture d'une expression naïve

Pour SELECT Name FROM Student WHERE Level = 'MSC', la traduction naïve est :

πName( σLevel='MSC'(Student) )

Cette requête simple est déjà optimale : il n'y a qu'une table et une sélection. Le RBA devient utile dès qu'il y a plusieurs tables, car le produit cartésien est coûteux.

Analogie — volume intermédiaire

Imaginez : filtrer 26 bulletins de notes pour ne garder que les étudiants MSC avant de chercher leur nom dans l'annuaire. Sans push-down, on croise d'abord les 26 bulletins avec les 6 étudiants (= 156 paires) avant de filtrer. Avec push-down, on filtre d'abord les 3 étudiants MSC, puis on croise (= 78 paires seulement).

Q1 4 pts Fondamental

Optimisez la Requête 1 par le RBA :

SELECT Name, Grade
FROM   Student, Grades
WHERE  Student.StudId = Grades.StudId
  AND  Level = 'MSC';
a) Écrivez l'expression d'algèbre relationnelle naïve (produit cartésien + une seule sélection conjonctive).
b) Appliquez les règles R1, R3, R4 pour obtenir l'expression optimisée. Justifiez chaque étape.
c) Calculez le nombre de tuples intermédiaires avant et après optimisation.
Piège courant : Appliquer directement R4 (synthèse de jointure) sans passer par R1 (cascade). Si la sélection mélange une condition de jointure (StudId = StudId) et une condition de filtre (Level = 'MSC'), il faut d'abord les séparer avec R1 avant de descendre la condition de filtre.
Expression naïve : πName,Grade( σLevel='MSC' ∧ Student.StudId=Grades.StudId(Student × Grades) ).
La conjonction dans la sélection est le point de départ de R1.
Après R1 : σLevel='MSC'( σStudent.StudId=Grades.StudId(Student × Grades) ).
R3 : Level='MSC' ne porte que sur Student → descendre avant le ×.
R4 : la sélection d'égalité restante + × → jointure naturelle.

Solution — Q1

a) Expression naïve

On traduit le FROM Student, Grades en produit cartésien et le WHERE en une sélection conjonctive :

πName,Grade( σLevel='MSC' ∧ Student.StudId=Grades.StudId(Student × Grades) )

À ce stade, le produit cartésien génère 6 × 26 = 156 tuples avant tout filtrage.

b) Application des règles

  • R1 — Cascade : on scinde la conjonction en deux sélections imbriquées.
    πName,Grade( σLevel='MSC'( σStudent.StudId=Grades.StudId(Student × Grades) ) )
  • R3 — Push-down : Level='MSC' ne référence que Student → on la descend au-dessus de Student seul, avant le produit cartésien.
    πName,Grade( σStudent.StudId=Grades.StudId( σLevel='MSC'(Student) × Grades ) )
  • R4 — Synthèse de jointure : la sélection d'égalité sur les deux colonnes StudId combinée au produit cartésien se réécrit en jointure.
    πName,Grade( σLevel='MSC'(Student) ⋈StudId Grades )

c) Volumes intermédiaires

ÉtapeOpérationTuples produits
NaïfStudent × Grades (6 × 26)156
Après push-downσMSC(Student) → 3 tuples MSC, puis ⋈ Grades≤ 78 (3 × 26, avant jointure)
Résultat jointureSeuls les tuples de Grades dont StudId ∈ {404,505,606}≈ 13 tuples utiles

Le push-down réduit dès le départ la table d'entrée gauche de 6 à 3 tuples — le coût de la jointure est divisé par 2 avant même de l'effectuer.

Q2 4 pts Fondamental

Optimisez la Requête 2 par le RBA :

SELECT Name, Dept, Grade
FROM   Student, Grades, Course
WHERE  Student.StudId = Grades.StudId
  AND  Grades.CourId  = Course.CourId
  AND  Level = 'MSC';
a) Écrivez l'expression naïve (deux produits cartésiens, une sélection à quatre prédicats conjonctifs).
b) Appliquez les règles R1, R3, R4 dans l'ordre. Quel est le nombre d'étapes nécessaires ?
Piège courant : Oublier que deux produits cartésiens sont associatifs mais pas interchangeables sans vérification. Le FROM A, B, C se traduit en (A × B) × C par défaut, mais R7 (associativité) permet de regrouper différemment — il faut choisir l'ordre qui minimise le résultat intermédiaire.
Expression naïve : πName,Dept,Grade( σP₁ ∧ P₂ ∧ P₃((Student × Grades) × Course) )
où P₁ = Level='MSC', P₂ = Student.StudId=Grades.StudId, P₃ = Grades.CourId=Course.CourId.
Appliquez R1 trois fois pour isoler chaque prédicat.
P₁ porte sur Student seul → push-down avec R3 jusqu'au-dessus de Student.
P₂ porte sur Student et Grades → synthèse de jointure (R4) entre ces deux tables.
P₃ porte sur Grades et Course → synthèse de jointure (R4) entre le résultat et Course.

Solution — Q2

a) Expression naïve

πName,Dept,Grade( σLevel='MSC' ∧ Student.StudId=Grades.StudId ∧ Grades.CourId=Course.CourId((Student × Grades) × Course) )

Volume avant filtrage : 6 × 26 × 7 = 1 092 tuples.

b) Séquence de règles

  • R1 ×3 — Cascade : on isole les trois prédicats conjonctifs en sélections imbriquées.
    σLevel='MSC'( σStudent.StudId=Grades.StudId( σGrades.CourId=Course.CourId((Student × Grades) × Course) ) )
  • R2 — Réordonnancement : on remonte la sélection la plus restrictive (Level='MSC') au plus près de la feuille Student.
  • R3 — Push-down de Level='MSC' : ne porte que sur Student → on la descend avant le premier produit cartésien.
    σStudent.StudId=Grades.StudId( σGrades.CourId=Course.CourId( (σLevel='MSC'(Student) × Grades) × Course ) )
  • R4 — Synthèse jointure Student ⋈ Grades :
    σGrades.CourId=Course.CourId( (σLevel='MSC'(Student) ⋈StudId Grades) × Course )
  • R4 — Synthèse jointure (…) ⋈ Course : expression finale.
    πName,Dept,Grade( (σLevel='MSC'(Student) ⋈StudId Grades) ⋈CourId Course )
Lecture de l'arbre d'exécution :
  • Feuille gauche : filtrer Student → 3 tuples MSC.
  • Jointure 1 : 3 tuples ⋈ 26 tuples Grades → ≈ 13 tuples (seuls les bulletins des étudiants MSC).
  • Jointure 2 : 13 tuples ⋈ 7 tuples Course → ≈ 13 tuples enrichis du Dept.
  • Projection finale sur Name, Dept, Grade.
Q3 6 pts Intermédiaire

Optimisez la Requête 3 par le RBA :

SELECT Name, Dept, Grade
FROM   Student, Grades, Course
WHERE  Student.StudId = Grades.StudId
  AND  Grades.CourId  = Course.CourId
  AND  Level = 'MSC'
  AND  Dept  = 'Math';
a) Écrivez l'expression naïve.
b) Appliquez le push-down complet : les deux sélections de filtre portent sur des tables différentes. Identifiez sur quelle table chacune doit être descendue.
c) Comparez les volumes intermédiaires pour l'ordre (Student ⋈ Grades) ⋈ Course versus l'ordre (Course ⋈ Grades) ⋈ Student après push-down. Lequel est préférable ?
Piège courant : Fusionner les deux sélections de filtre en une seule. Level='MSC' ne peut pas descendre au-dessus de Course, et Dept='Math' ne peut pas descendre au-dessus de Student. Chaque sélection doit rester au-dessus de sa propre table source.
Après cascade et push-down séparé :
— σLevel='MSC'(Student) → 3 tuples
— σDept='Math'(Course) → 2 tuples (M330 et M444)
Grades reste intouché (aucune condition ne porte sur lui seul).
Pour comparer les ordres, estimez les intermédiaires :
— Ordre A : σMSC(Student) ⋈ Grades → ≈ 13 tuples, puis ⋈ σMath(Course) → peu de tuples Math.
— Ordre B : σMath(Course) ⋈ Grades → ≈ peu de tuples, puis ⋈ σMSC(Student).
L'ordre qui produit le plus petit intermédiaire en premier est préférable.

Solution — Q3

a) Expression naïve

πName,Dept,Grade( σLevel='MSC' ∧ Dept='Math' ∧ Student.StudId=Grades.StudId ∧ Grades.CourId=Course.CourId((Student × Grades) × Course) )

Volume naïf : 6 × 26 × 7 = 1 092 tuples avant tout filtrage.

b) Push-down des deux sélections

  • R1 ×4 : cascade → quatre sélections imbriquées (une par prédicat).
  • R3 — Push Level='MSC' : porte sur Student seul → descend jusqu'à la feuille Student.
  • R3 — Push Dept='Math' : porte sur Course seul → descend jusqu'à la feuille Course.
  • R4 ×2 : synthèse des deux jointures.

Expression optimisée :

πName,Dept,Grade( (σLevel='MSC'(Student) ⋈StudId Grades) ⋈CourId σDept='Math'(Course) )

c) Comparaison des ordres de jointure

OrdreIntermédiaire 1Intermédiaire 2Verdict
A : (σMSC(Stu) ⋈ Grades) ⋈ σMath(Cou) 3 ⋈ 26 ≈ 13 tuples 13 ⋈ 2 ≈ quelques tuples Math/MSC Acceptable
B : (σMath(Cou) ⋈ Grades) ⋈ σMSC(Stu) 2 ⋈ 26 ≈ bulletins Math seulement ≈ 10 10 ⋈ 3 ≈ quelques tuples Math/MSC Préférable

L'ordre B est légèrement préférable : commencer par σDept='Math'(Course) (2 cours sur 7) produit un intermédiaire encore plus petit que l'ordre A avant la dernière jointure. Dans les deux cas, le push-down élimine l'essentiel des tuples parasites.

Q4 6 pts Avancé

Optimisez la Requête 4 par le RBA :

SELECT Count(*)
FROM   Student, Grades, Course
WHERE  Student.StudId = Grades.StudId
  AND  Grades.CourId  = Course.CourId
  AND  Level = 'MSC'
  AND  Dept  = 'Stat';
a) Écrivez l'expression naïve complète.
b) Appliquez le RBA complet et donnez l'expression optimisée finale.
c) Donnez le résultat numérique en vous appuyant sur les données des tables (Student et Course fournies dans le contexte). Combien de tuples Grades l'optimiseur doit-il lire au minimum ?
Piège courant : Penser que Count(*) supprime la possibilité de push-down. L'agrégat s'applique après toutes les jointures et sélections — l'optimiseur descend d'abord les sélections, puis agrège le résultat final. L'expression optimisée place l'agrégat au sommet de l'arbre, pas en bas.
Pour le Count(*), la projection finale devient un agrégat : γCount(*) (ou notez simplement Count en haut de l'arbre). Le reste du traitement est identique à Q3 avec Dept='Stat' à la place de Dept='Math'.
σDept='Stat'(Course) retient S402 et S466 → 2 cours Stat.
σLevel='MSC'(Student) retient Ambar (404), Philip (505), Nicolas (606) → 3 étudiants MSC.
Comptez dans Grades les entrées dont CourId ∈ {S402, S466} ET StudId ∈ {404,505,606}.

Solution — Q4

a) Expression naïve

γCount(*)( σLevel='MSC' ∧ Dept='Stat' ∧ Student.StudId=Grades.StudId ∧ Grades.CourId=Course.CourId((Student × Grades) × Course) )

Volume naïf : 6 × 26 × 7 = 1 092 tuples, dont la quasi-totalité sera éliminée par les sélections.

b) Expression optimisée (RBA complet)

  • R1 ×4 : cascade des quatre prédicats en sélections imbriquées.
  • R3 ×2 : push-down de Level='MSC' au-dessus de Student, et de Dept='Stat' au-dessus de Course.
  • R4 ×2 : synthèse des deux jointures (Student ⋈ Grades, puis ⋈ Course).

Expression finale :

γCount(*)( (σLevel='MSC'(Student) ⋈StudId Grades) ⋈CourId σDept='Stat'(Course) )

c) Résultat numérique

Après push-down :

  • σLevel='MSC'(Student) → 3 étudiants : Ambar (404), Philip (505), Nicolas (606).
  • σDept='Stat'(Course) → 2 cours Stat : S402 (BioS), S466 (Stat).
  • L'optimiseur lit uniquement les tuples de Grades correspondant à ces contraintes (après jointure, il cherche des lignes avec StudId ∈ {404,505,606} et CourId ∈ {S402,S466}).
ÉtapeVolumeRatio vs naïf
Naïf (Student × Grades × Course)1 092 tuplesréférence
Après push-down sur Student (×3)3 × 26 × 7 = 546÷ 2
Après push-down sur Course (×2)3 × 26 × 2 = 156÷ 7
Après jointures (tuples utiles)≤ quelques unités÷ >100

Le RBA garantit que chaque sélection est appliquée au plus tôt, réduisant dramatiquement le volume traité. Sans push-down, on aurait traversé 1 092 paires pour n'en conserver que quelques-unes. Le Count(*) final s'applique sur un résultat déjà minimal.
