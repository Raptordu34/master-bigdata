Algèbre Relationnelle (fondations)
  • σC(R) : Sélection (filtre lignes).
  • πA,B(R) : Projection (garde colonnes).
  • R ⋈C S : Jointure (produit + filtre).
  • ρS(A,B)(R) : Renommage.
  • R ⋉ S : Semi-jointure (tuples de R ayant correspondance dans S).
Règle d'or : Early Selection. Toujours pousser les sélections (σ) vers le bas de l'arbre pour réduire les tailles intermédiaires.
Ordre Logique d'Exécution SQL
  1. FROM / JOIN (Accès données)
  2. WHERE (Filtrage tuples)
  3. GROUP BY (Regroupement)
  4. HAVING (Filtrage groupes)
  5. SELECT (Projection & Calculs)
  6. DISTINCT / ORDER BY / LIMIT
Accès & Modèles de Coût
  • Full Table Scan (FTS) : Lit 100% des blocs. Efficace pour gros volumes (>10% table).
  • Index Range Scan : Parcours B-Tree pour plage de valeurs.
  • Table Access by ROWID : Récupère la ligne physique après accès index.

Hachage (Hashing)
Hachage Statique : k seaux fixes. Risque de collisions/débordements. Hachage Dynamique : Utilise un répertoire/arbre pour gérer la croissance.
Coût insertion : lit répertoire + lit bloc + écrit bloc.
Linear Hashing : Division graduelle des seaux sans répertoire global.

Agrégats & Filtrage
  • COUNT(*) compte tout (null inclus).
  • COUNT(col) ignore les NULL.
  • WHERE vs HAVING : WHERE filtre avant l'agrégation, HAVING après.
SELECT dpt, AVG(sal) FROM emp GROUP BY dpt HAVING AVG(sal) > 2000;
Lecture Plan Oracle (EXPLAIN)
  • Lecture : Du bas vers le haut, de droite à gauche (les plus indentés d'abord).
  • NESTED LOOPS : Boucle externe (petite) et boucle interne (indexée).
  • SORT JOIN : Tri nécessaire avant MERGE JOIN.
  • INDEX UNIQUE SCAN : 1 seule ligne (recherche PK).
Performance : Préférer les index si la requête est sélective (<5-10% de la table). Sinon, préférer le Full Scan (E/S séquentielles).
Pièges & Règles CBA
  • Sargability : WHERE YEAR(date) = 2024 tue l'index. Utiliser date >= '01/01/2024'.
  • Nulls : Un index standard ne stocke pas les valeurs NULL (attention aux `NOT IN`).
  • Sélectivité : Nb valeurs distinctes / Nb tuples total. Plus elle est faible (proche de 0), mieux c'est pour l'index.

Algorithmes de Jointure (Crucial)
1. Nested Loops (NLJ) :
Coût = |R| + (|R| / (M-2)) × |S|
  • Optimal si R tient en RAM : |R| + |S|
  • Mémoire min : M ≥ 3 (1 R, 1 S, 1 Out).
  • Usage : Petites relations ou index interne.
2. Sort-Merge Join (SMJ) :
Coût = 3(|R| + |S|) [si 2 passes]
  • Usage : Données déjà triées ou opérateur non équi-jointure (<, >).
3. Grace Hash Join (HJ) :
Coût = 3(|R| + |S|) [si 2 passes]
  • Phase 1 : Partitionnement (2(|R|+|S|)).
  • Phase 2 : Sondage/Join (|R|+|S|).
  • Condition : M ≥ √|R| + 1.
Semi-Jointure & Full Reducer
Principe (⋉) : Réduire S avant le Join.
  1. Projeter R sur l'attribut de join : πA(R)
  2. Envoyer à S, filtrer : S' = S ⋉ πA(R)
  3. Joindre final : R ⋈ S'
Rentable si : |πA(R)| + |S'| < |S|.
Full Reducer (n relations) : Une passe aller (R1→Rn) + une passe retour (Rn→R1) de semi-jointures. Garanti de ne garder que les tuples participants.
Estimations de Cardinalité
  • f(Col = val) = 1 / NDV(Col)
  • f(Col > val) = (Max - val) / (Max - Min)
  • f(A AND B) = f(A) × f(B)
  • f(A OR B) = f(A) + f(B) - (f(A)×f(B))
  • |R ⋈ S| = (|R| × |S|) / MAX(NDV(R.A), NDV(S.A))
Fonctions de Fenêtrage (OLAP)
F(...) OVER (PARTITION BY ... ORDER BY ...)
  • ROW_NUMBER() : 1, 2, 3, 4 (unique)
  • RANK() : 1, 1, 3, 4 (saute si ex-æquo)
  • DENSE_RANK() : 1, 1, 2, 3 (ne saute pas)
  • LAG(col, 1) / LEAD : Accès ligne préc/suiv.
  • SUM(x) OVER(ORDER BY t) : Somme mobile.
RBO vs CBO
  • RBO (Rule-Based) : Utilise des priorités fixes (ex: Index > Full Scan). Obsolète.
  • CBO (Cost-Based) : Calcule le coût I/O et CPU via les statistiques.
  • Statistiques : Min, Max, NDV, Histogrammes. Nécessitent des `ANALYZE` réguliers.
Exemple CASE & Window
SELECT nom, sal,
  CASE WHEN sal > 3000 THEN 'H' ELSE 'L' END,
  RANK() OVER(PARTITION BY dpt ORDER BY sal DESC)
FROM emp;
