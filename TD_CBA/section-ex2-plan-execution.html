Exercice 2 — Plan d'exécution de requêtes

Intermédiaire ◎ 10 pts ⏱ 25–35 min
Objectifs
  • Construire l'expression algébrique d'une requête SQL multi-tables
  • Lire et interpréter un plan EXPLAIN Oracle (structures d'accès, types de jointure)
  • Réécrire une requête avec et sans structures redondantes (index)

Prérequis : Cours — Algèbre relationnelle · Cours — Plans d'exécution

Contexte — Films d'Hitchcock

La base contient trois tables : FILM, ARTISTE et SEANCE. La requête suivante retourne les titres des films réalisés par Hitchcock projetés après 20h00 : 

SELECT F.TITRE
FROM   SEANCE S, FILM F, ARTISTE A
WHERE  A.NOM = 'Hitchcock'
  AND  F.IDREALISATEUR = A.IDREALISATEUR
  AND  S.IDFILM = F.IDFILM
  AND  S.HEURE_DEBUT > '20:00'

Sous Oracle, l'outil EXPLAIN produit le plan d'exécution suivant :

0  SELECT STATEMENT
  1  MERGE JOIN
    2  SORT Join
      3  NESTED LOOPS
        4  TABLE ACCESS FULL ARTISTE
        5  TABLE ACCESS BY ROWID FILM
          6  INDEX RANGE SCAN IDX-ARTISTE-ID
    7  SORT Join
      8  TABLE ACCESS FULL SEANCE
TableColonnes clés
ARTISTEIDREALISATEUR, NOM
FILMIDFILM, TITRE, IDREALISATEUR
SEANCEIDSEANCE, IDFILM, HEURE_DEBUT

Opérateurs de base :

  • σ (sélection) : filtre les lignes selon un prédicat — σ_{condition}(R)
  • π (projection) : garde certaines colonnes — π_{cols}(R)
  • ⋈ (jointure naturelle / θ-jointure) : combine deux relations — R ⋈_{condition} S

Règle d'or : pousser les sélections le plus bas possible dans l'arbre (push-down) pour réduire la taille des relations intermédiaires avant les jointures.

Un plan Oracle se lit des feuilles vers la racine (de bas en haut, de droite à gauche) :

  • Les nœuds feuilles (ex : nœuds 4, 6, 8) sont les opérations d'accès aux données brutes.
  • Les nœuds parents combinent ou transforment ces données (SORT, NESTED LOOPS, MERGE JOIN).
  • La racine (nœud 0) est le résultat final retourné au client.
Nœud OracleSignification
TABLE ACCESS FULLParcours séquentiel complet (scan)
INDEX RANGE SCANParcours d'une plage d'un index B-tree
TABLE ACCESS BY ROWIDAccès à la ligne via son adresse physique (ROWID)
NESTED LOOPSJointure par boucles imbriquées
MERGE JOINJointure par fusion (les deux côtés doivent être triés)
SORT JoinTri préalable d'une relation pour le Merge Join
Q1 3 pts Fondamental

Donner l'expression algébrique correspondante à la requête SQL et proposer un arbre algébrique optimal.

Piège courant : Écrire les sélections au sommet de l'arbre (comme en SQL). En algèbre, l'optimisation consiste à les descendre au plus près des feuilles pour limiter le volume des résultats intermédiaires.
Identifiez d'abord les sélections : A.NOM = 'Hitchcock' (très sélectif — une seule ligne ?) et S.HEURE_DEBUT > '20:00' (filtre une partie de SEANCE). Appliquez-les en premier sur chaque table concernée avant toute jointure.
Ordre recommandé pour les jointures : commencer par le résultat le plus petit (σ(ARTISTE) — très sélectif sur NOM), puis joindre FILM, puis SEANCE. La projection π sur TITRE se place au sommet.

Solution — Q1

Expression algébrique (non optimisée) :

πF.TITRE( σA.NOM='Hitchcock' ∧ S.HEURE_DEBUT>'20:00'( SEANCE ⋈S.IDFILM=F.IDFILM FILM ⋈F.IDREALISATEUR=A.IDREALISATEUR ARTISTE ) )

Arbre algébrique optimal (sélections poussées vers les feuilles) :

                     π F.TITRE
                         │
                    ⋈ S.IDFILM = F.IDFILM
                   ╱                    ╲
    σ HEURE_DEBUT > '20:00'      ⋈ F.IDREALISATEUR = A.IDREALISATEUR
              │                 ╱                              ╲
           SEANCE           FILM                    σ NOM = 'Hitchcock'
                                                           │
                                                        ARTISTE

Justification de l'ordre des jointures :

  • σ_{NOM='Hitchcock'}(ARTISTE) — sélection très sélective, produit probablement 1 ligne.
  • Jointure avec FILM via IDREALISATEUR — résultat réduit (films d'Hitchcock seulement).
  • σ_{HEURE_DEBUT>'20:00'}(SEANCE) — réduit SEANCE avant la jointure finale.
  • Jointure finale SEANCE ⋈ (FILM ⋈ ARTISTE) sur IDFILM.

L'arbre optimal minimise la taille des résultats intermédiaires en appliquant les sélections le plus tôt possible (early selection).

Q2 3 pts Intermédiaire

Commenter le plan EXPLAIN Oracle en identifiant les structures d'accès aux tables et le type de jointure utilisé.

Piège courant : Lire le plan de haut en bas. Oracle EXPLAIN se lit de bas en haut et de droite à gauche — les nœuds les plus indentés s'exécutent en premier.
Identifiez d'abord les feuilles (nœuds 4, 6, 8) : comment chaque table est-elle accédée (scan séquentiel ou index) ? Puis remontez vers les nœuds 3 et 1 pour identifier les types de jointure.

Solution — Q2

Structures d'accès aux tables :

NœudTableStructure d'accèsCommentaire
4 ARTISTE TABLE ACCESS FULL Parcours séquentiel complet — pas d'index sur NOM utilisé ici
6 FILM (index) INDEX RANGE SCAN IDX-ARTISTE-ID Parcours de l'index sur FILM.IDREALISATEUR pour chaque tuple ARTISTE
5 FILM (données) TABLE ACCESS BY ROWID Accès aux lignes FILM via leur adresse physique (ROWID) obtenue par l'index
8 SEANCE TABLE ACCESS FULL Parcours séquentiel complet — pas d'index sur HEURE_DEBUT

Types de jointure :

  • Nœud 3 — NESTED LOOPS (ARTISTE ⋈ FILM) : pour chaque ligne d'ARTISTE (boucle externe), Oracle utilise l'index IDX-ARTISTE-ID pour trouver les films correspondants (boucle interne). Efficace si ARTISTE retourne peu de lignes.
  • Nœuds 2 et 7 — SORT Join : les deux côtés (résultat du NESTED LOOPS et SEANCE) sont triés sur l'attribut de jointure IDFILM avant le MERGE JOIN.
  • Nœud 1 — MERGE JOIN ((ARTISTE⋈FILM) ⋈ SEANCE) : fusion de deux flots triés. Efficace sur de gros volumes si les données sont déjà ordonnées ou si le coût de tri est amorti.

Le plan Oracle n'applique pas la sélection A.NOM = 'Hitchcock' en premier (il fait un FULL SCAN d'ARTISTE) car il n'y a pas d'index sur ARTISTE.NOM.

Q3 3 pts Intermédiaire

Réécrire la requête dans les deux cas suivants :

a) Avec structures redondantes (en exploitant des index existants).
b) Sans structures redondantes (accès séquentiels uniquement).
Piège courant : Confondre "réécriture SQL" et "modification sémantique". Les deux versions doivent retourner exactement le même résultat. Seule la façon d'orienter l'optimiseur change (via des hints Oracle).
Oracle permet de forcer le comportement de l'optimiseur avec des hints : /*+ INDEX(alias nom_index) */ force l'utilisation d'un index, /*+ FULL(alias) */ force un accès séquentiel. Pour 3a, guidez Oracle vers les index ; pour 3b, forcez le FULL SCAN sur toutes les tables.

Solution — Q3

3a) Avec structures redondantes (utilisation des index) :

SELECT /*+ INDEX(A IDX_NOM_ARTISTE)
           INDEX(F IDX_ARTISTE_ID)
           INDEX(S IDX_HEURE_SEANCE) */
       F.TITRE
FROM   SEANCE S, FILM F, ARTISTE A
WHERE  A.NOM = 'Hitchcock'
  AND  F.IDREALISATEUR = A.IDREALISATEUR
  AND  S.IDFILM = F.IDFILM
  AND  S.HEURE_DEBUT > '20:00'

Les hints indiquent à Oracle d'utiliser :

  • IDX_NOM_ARTISTE sur ARTISTE.NOM → accès direct à Hitchcock
  • IDX_ARTISTE_ID sur FILM.IDREALISATEUR → trouve les films par réalisateur
  • IDX_HEURE_SEANCE sur SEANCE.HEURE_DEBUT → filtre les séances après 20h

3b) Sans structures redondantes (accès séquentiels forcés) :

SELECT /*+ FULL(A) FULL(F) FULL(S) */
       F.TITRE
FROM   SEANCE S, FILM F, ARTISTE A
WHERE  A.NOM = 'Hitchcock'
  AND  F.IDREALISATEUR = A.IDREALISATEUR
  AND  S.IDFILM = F.IDFILM
  AND  S.HEURE_DEBUT > '20:00'

Oracle lit l'intégralité des trois tables séquentiellement, sans utiliser aucun index.

Q4 1 pt Fondamental

Dire laquelle des deux écritures (3a ou 3b) est plus performante et justifier.

Solution — Q4

La version 3a (avec structures redondantes / index) est plus performante.

Raison principale

  • L'index sur ARTISTE.NOM permet d'accéder directement au réalisateur "Hitchcock" sans lire toute la table ARTISTE.
  • L'index sur FILM.IDREALISATEUR évite un parcours complet de FILM : seuls les films d'Hitchcock sont lus.
  • L'index sur SEANCE.HEURE_DEBUT filtre efficacement les séances tardives.

En termes d'E/S : la version 3b lit l'intégralité des trois tables (coût proportionnel au volume total), tandis que 3a lit uniquement les pages utiles (coût proportionnel au résultat + log(n) pour les accès index).

Exception : si les tables sont très petites ou si la sélectivité est faible (beaucoup de films d'Hitchcock, beaucoup de séances tardives), le scan séquentiel peut être compétitif en évitant les accès aléatoires induits par les index.
