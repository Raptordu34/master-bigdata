Exercice 3 — Optimisation de la jointure

Avancé ◎ 12 pts ⏱ 25–35 min
Objectifs
  • Recenser les stratégies classiques d'implémentation d'une jointure
  • Proposer une optimisation par restructuration de la requête (semi-jointure)
  • Calculer le coût d'une semi-jointure et le généraliser à n relations

Prérequis : Cours — Algorithmes de jointure · Cours — Optimisation de requêtes distribuées

Contexte

On considère l'opération de jointure R join S sur un attribut commun. L'objectif est d'abord d'identifier les stratégies standards, puis d'explorer une approche différente pour réduire le coût de la jointure.

Notation : on note |R| et |S| le nombre de pages (blocs) de R et S, et M le nombre de pages mémoire disponibles. Le coût est exprimé en nombre d'entrées/sorties disque.

Les trois familles d'algorithmes de jointure :

AlgorithmePrincipeCoût (pages)Condition favorable
Nested Loop Pour chaque page de R, lire toutes les pages de S |R| + |R|/M × |S| Une relation tient en mémoire
Sort-Merge Trier R et S sur l'attribut de jointure, puis fusionner 3(|R| + |S|) Données déjà triées ou tri rentable
Hash Join Hacher R et S en partitions, joindre partition par partition 3(|R| + |S|) Jointure d'équi-jointure sur gros volumes
Q1 3 pts Fondamental

Quelles sont les stratégies pour implémenter la jointure R join S ? Pour chaque stratégie, indiquer brièvement le principe et les conditions d'utilisation optimale.

Piège courant : Citer seulement le Nested Loop. Il existe au moins trois familles d'algorithmes de jointure — chacune adaptée à un contexte différent (taille des relations, disponibilité d'index, mémoire disponible).
Pensez aux trois familles : (1) boucles imbriquées, (2) tri puis fusion, (3) hachage. Pour chacune : quel est le coût en E/S ? Quand est-elle préférable aux deux autres ?

Solution — Q1

1. Nested Loop Join (boucles imbriquées) :

  • Pour chaque bloc de R, on charge les blocs de S et on cherche les correspondances.
  • Coût optimal (Block Nested Loop) : |R| + ⌈|R|/(M−2)⌉ × |S|
  • Si R tient en mémoire : coût = |R| + |S|
  • Favorable : l'une des deux relations est petite (tient en mémoire).

2. Sort-Merge Join (tri-fusion) :

  • Tri de R et S sur l'attribut de jointure, puis parcours simultané pour fusionner.
  • Coût (2 passes) : 3(|R| + |S|)
  • Favorable : données déjà partiellement triées, ou jointure suivie d'un ORDER BY.

3. Hash Join :

  • Phase 1 : partitionner R et S avec la même fonction de hachage sur l'attribut de jointure.
  • Phase 2 : joindre chaque paire de partitions (Rᵢ, Sᵢ) en mémoire.
  • Coût (2 passes) : 3(|R| + |S|)
  • Favorable : équi-jointure sur de gros volumes sans ordre requis en sortie.

4. Index Join :

  • Si un index existe sur l'attribut de jointure d'une des tables, l'utiliser pour accéder directement aux tuples correspondants.
  • Coût : |R| + |R_tuples| × coût_accès_index(S)
  • Favorable : index disponible ET forte sélectivité de la jointure.
Q2 3 pts Avancé

Que ferez-vous pour optimiser cette jointure sans utiliser les stratégies citées dans Q1 ? Décrivez l'approche proposée.

Piège courant : Chercher une quatrième variante de Nested Loop ou Hash Join. La question invite à sortir du cadre des algorithmes classiques de jointure : il s'agit d'une restructuration de la requête elle-même, pas de l'algorithme.
Le coût d'une jointure R ⋈ S est souvent dominé par le transfert de données inutiles. Existe-t-il un moyen de ne transférer que les attributs de jointure dans un premier temps, puis de compléter avec les données utiles seulement ?
La semi-jointure (⋉) : on commence par envoyer à S uniquement les valeurs de l'attribut de jointure de R. S filtre ses tuples pour ne garder que ceux qui participent à la jointure, puis on effectue la jointure finale avec ce sous-ensemble réduit.

Solution — Q2

Approche : la semi-jointure (⋉)

L'idée est de réduire S avant la jointure en ne conservant que les tuples de S qui participent effectivement à la jointure avec R. On procède en 3 étapes :

  • Étape 1 : projeter R sur l'attribut de jointure → πA(R) (petit volume : uniquement les valeurs de l'attribut)
  • Étape 2 : semi-joindre S avec cette projection → S' = S ⋉ πA(R) — ne garder que les tuples de S qui ont une correspondance
  • Étape 3 : effectuer la jointure finale complète → R ⋈ S' — S' est plus petit que S

Analogie

Intuition : au lieu de comparer tous les employés de deux entreprises, on envoie d'abord une liste de noms à la deuxième entreprise. Elle répond uniquement avec les employés concernés. On évite de transférer les dossiers complets inutilement.

Condition d'efficacité : la semi-jointure est rentable si |πA(R)| + |S'| < |S|, c'est-à-dire si beaucoup de tuples de S sont éliminés à l'étape 2 (taux de réduction élevé).

La semi-jointure est particulièrement efficace dans les systèmes distribués, où le coût de transfert réseau est dominant. Oracle et d'autres SGBD utilisent ce mécanisme sous le nom de Bloom filter join ou Adaptive Serial Filtering.

Q3 3 pts Avancé

Quel est le coût de la solution proposée en Q2 ? Exprimer ce coût en nombre d'entrées/sorties et comparer à une jointure directe.

Piège courant : Oublier le coût de l'étape 1 (projection de R et écriture de πA(R)). La semi-jointure a un coût en 3 étapes ; chacune doit être comptée.
Étape 1 : lire R, écrire πA(R) (petit). Étape 2 : lire S et πA(R), écrire S'. Étape 3 : jointure R ⋈ S' avec coût classique mais sur S' réduit. Posez f = taux de réduction de S (fraction des tuples conservés).

Solution — Q3

Posons f le facteur de réduction de S (0 < f ≤ 1). Après semi-jointure : |S'| = f × |S|.

ÉtapeOpérationCoût (E/S)
1 Lire R, projeter sur A, écrire πA(R) |R| + |πA(R)| ≈ |R| (projection petite)
2 Lire S + πA(R), écrire S' |S| + |πA(R)| + f·|S| ≈ |S|(1+f)
3 Jointure R ⋈ S' (ex : hash join) 3(|R| + f·|S|)
Total semi-jointure |R| + |S|(1+f) + 3(|R| + f·|S|)
Jointure directe (hash join) 3(|R| + |S|)

La semi-jointure est avantageuse si :

|R| + |S|(1+f) + 3(|R| + f·|S|) < 3(|R| + |S|)
→ 4|R| + |S|(1+4f) < 3|R| + 3|S|
→ |R| < |S|(2 − 4f)
→ f < (2|S| − |R|) / (4|S|)

Si f est petit (beaucoup de tuples éliminés de S) et |S| >> |R|, la semi-jointure est très intéressante. Si f ≈ 1 (peu d'éliminations), la jointure directe reste préférable.

Dans un contexte distribué, le coût de transfert réseau de πA(R) (très petit) remplace souvent un transfert complet de S, ce qui rend la semi-jointure particulièrement attractive.

Q4 3 pts Avancé

Généraliser la solution proposée en Q2 au cas d'une jointure entre n relations R₁ ⋈ R₂ ⋈ … ⋈ Rₙ.

Piège courant : Appliquer les semi-jointures dans un ordre quelconque. L'ordre dans lequel on réduit les relations impacte fortement l'efficacité. Il faut choisir un ordre qui minimise les tailles intermédiaires.
Pensez à un programme de semi-jointures : une séquence d'opérations R₁ ⋉ R₂, R₂ ⋉ R₁, R₁ ⋉ R₃… L'algorithme de semi-join reduction (ou full reducer) garantit que chaque relation est réduite à ses seuls tuples participant à la jointure finale.

Solution — Q4

Programme de semi-jointures (Full Reducer)

Pour n relations R₁ ⋈ R₂ ⋈ … ⋈ Rₙ, on construit un programme de semi-jointures qui réduit chaque relation avant la jointure finale. L'algorithme procède en deux passes :

  • Passe aller (i = 1 → n-1) : semi-joindre chaque relation avec la suivante. R₂ ← R₂ ⋉ R₁, puis R₃ ← R₃ ⋉ R₂, …, Rₙ ← Rₙ ⋉ Rₙ₋₁
  • Passe retour (i = n → 2) : propager les réductions dans l'autre sens. Rₙ₋₁ ← Rₙ₋₁ ⋉ Rₙ, …, R₁ ← R₁ ⋉ R₂
  • Jointure finale : joindre les relations réduites R₁' ⋈ R₂' ⋈ … ⋈ Rₙ' (toutes petites si le facteur de réduction est important)

Propriété clé

Après les deux passes, chaque Rᵢ' ne contient que les tuples qui participent effectivement à la jointure totale. On appelle ce résultat le full reducer ou réducteur complet.

Coût total :

  • Passe aller : (n−1) semi-jointures
  • Passe retour : (n−1) semi-jointures
  • Jointure finale sur les relations réduites
  • Coût total des semi-jointures : O(n × (|R₁| + … + |Rₙ|))
Optimisation de l'ordre : dans un contexte général (pas une chaîne), l'ordre optimal de jointure est un problème NP-difficile. En pratique, les optimiseurs utilisent la programmation dynamique ou des heuristiques (ex : joindre d'abord les tables les plus sélectives).

Pour n = 2 (cas de Q2), on retrouve bien la séquence : S' = S ⋉ R (passe aller), R' = R ⋉ S' (passe retour), puis R' ⋈ S'.
