Rappels critiques

BNLJ : Coût = B(R) + ⌈B(R)/M⌉ × B(S)
Hash Join : 3(|R|+|S|) blocs ; M ≥ √B(R) + 1
VM CONCURRENTLY → index unique obligatoire
VH : si colonne du prédicat H a été projetée hors de V → fragment impossible
COUNT(*) ≠ COUNT(DISTINCT x) quand JOIN peut dupliquer des tuples

Arbre : Quelle jointure choisir ?

Quelle jointure choisir ? Données triées sur attribut join → Sort-Merge Join Non triées, grandes tables → Hash Join (p.4) Index disponible sur table interne → Index NLJ M très petit / tables petites → BNLJ (p.3) Réduction transfert réseau (dist.) → Semi-jointure (p.2)

Arbre : Quel index choisir ?

Quel index ? Haute cardinalité + range/ORDER BY → B-Tree (p.5) Faible cardinalité + OLAP/AND/OR → Bitmap (p.5) Multi-colonnes fréquentes ensemble → B-Tree composite Jointure étoile, filtre multi-dim. → BJI (p.5) Agrégat coûteux répété → VM (p.7)

Arbre : Type de fragmentation ?

Fragmentation ? Lignes selon prédicat (WHERE) → FH σ_p(R) Colonnes selon usage (SELECT cols) → FV π_A(R) Lignes PUIS colonnes → Mixte HV Colonnes PUIS lignes → Mixte VH ⚠ risque Table faits suit dimension → Dérivée (p.1)

Arbre : VM ou Index direct ?

Index ou VM ? Agrégat répété > 100k lignes → VM Faible cardinalité OLAP → Bitmap / BJI Range + ORDER BY fréquent → B-Tree composite Multi-filtres schéma étoile → BJI + FHD

Fragmentation Horizontale (FH)

Opérateur : Ri = σ_pi(R) — sélection par prédicat
Reconstruction : R = R1 ∪ R2 ∪ … ∪ Rn

Propriétés obligatoires :

  • Complétude : ∀t ∈ R, ∃i : t ∈ Ri
  • Disjonction : Ri ∩ Rj = ∅ (prédicats mutuellement exclusifs)
  • Reconstruction : ∪ Ri = R
R1 = σ_{Pays='FR'}(R), R2 = σ_{Pays='DE'}(R)

Fragmentation Verticale (FV)

Opérateur : Ri = π_Ai(R) — projection sur colonnes
Reconstruction : F1 ⋈_PK F2 ⋈_PK … (PK dans chaque fragment)
La clé primaire DOIT être dupliquée dans chaque fragment vertical
F1 = π_{id,nom,adresse}(R), F2 = π_{id,salaire}(R) → jointure sur id

Principes de réécriture distribuée

Principe : remplacer SELECT … FROM R par une UNION de requêtes sur les fragments Ri

Algorithme :

  1. Décomposer le WHERE en prédicats
  2. Pour chaque fragment, tester si son prédicat contredit la condition
  3. Éliminer les fragments contradictoires (résultat vide garanti)
  4. Combiner les fragments restants avec UNION
Push-down : appliquer σ et π avant les jointures inter-sites → réduction des transferts réseau

Coût de transfert réseau

Modèle de coût distribué :
Coût total = Coût I/O local + Coût transfert réseau
Coût_réseau = nb_tuples_transférés × taille_tuple × coût_par_octet
Minimiser les transferts :
1. Push-down σ : filtrer avant d'envoyer
2. Push-down π : ne projeter que les colonnes nécessaires
3. Semi-jointure : envoyer seulement la clé de jointure
4. Co-localisation : placer les fragments joints sur le même site
R(10 000 tuples, 100 octets/tuple) → 1 MB à transférer
Après σ (10% sélectivité) → 100 ko seulement
Après π (5 colonnes sur 20) → 25 ko

Élimination de jointures redondantes

Si ∀ tuple ∈ R, ∃ clé correspondante ∈ S (FK garantie) et S n'apporte aucune colonne → jointure superflue, supprimable
Ne supprimer que si la contrainte FK est vérifiée en base — sinon résultat faux

Semi-jointure distribuée

Définition : SJ(R, S, A) = R ⋈ π_A(S)
Envoyer seulement les valeurs de l'attribut de jointure, pas la table entière

Protocole 3 étapes :

  1. Site S envoie π_A(S) → site R (petite projection, ~1 colonne)
  2. Site R calcule R' = R ⋈ π_A(S) → filtre local R
  3. Site R envoie R' → site S pour la jointure finale complète
Facteur de réduction : fr = |R'| / |R|
SJ bénéfique si fr << 1 (beaucoup de tuples éliminés)
SJ inutile si fr ≈ 1 (peu de tuples éliminés, 2 transferts pour rien)
Coût SJ : |π_A(S)| + |R'| transferts
vs jointure directe : |R| + résultat
Bénéfice = |R| − (|π_A(S)| + |R'|) = |R| × (1−fr) − |π_A(S)|
R=1000, S=200, fr=0.08 → R'=80
SJ = 200 ids + 80 tuples = 280 transferts
Direct = 1000 tuples → gain ×3.6
SJ coûteux si fr > 0.5 — vérifier le facteur de réduction avant d'appliquer

Vérification reconstructibilité

Complétude : ∪ prédicats_fragments = domaine total de R
Disjonction : prédicats mutuellement exclusifs
Reconstruction : ∪ Ri = R (FH) ou ⋈_PK Ri = R (FV)
R1 (Dur≥12), R2 (Dur<12)
Complétude : Dur≥12 ∪ Dur<12 = ℝ ✓
Disjonction : impossible d'avoir Dur≥12 ET Dur<12 ✓

Deux approches d'optimisation

Rule-Based Optimization (RBO) : l'optimiseur applique un ensemble de règles de réécriture algébrique sans tenir compte des statistiques sur les données. L'ordre des règles est fixe et déterministe.

Exemples de règles RBO (RBA) :

  • R1 — Push-down σ : descendre les sélections au plus près des feuilles de l'arbre algébrique
  • R2 — Push-down π : projeter le plus tôt possible pour réduire la largeur des tuples
  • R3 — Éliminer σ impossible : si le prédicat contredit le fragment → retirer
  • R4 — Simplifier jointures : éliminer jointures redondantes (FK garantie)
  • R5 — Réordonner jointures : mettre les jointures les plus sélectives en premier
Cost-Based Optimization (CBO) : l'optimiseur génère plusieurs plans candidats, estime le coût de chacun (I/O, CPU, réseau) via des statistiques (cardinalités, histogrammes, sélectivités) et retient le plan de moindre coût.
RBOCBO
BaseRègles fixesStatistiques réelles
QualitéBonne par défautMeilleure si stats à jour
Coût calculO(n règles)Exponentiel (plans)
RisqueSous-optimal si données atypiquesMauvais si stats obsolètes
En pratique : les SGBD modernes (Oracle, PostgreSQL) utilisent CBO + RBO ensemble — les règles algébriques réduisent l'espace de plans, puis CBO choisit parmi eux.

Heuristiques — Définition & exemples

Heuristique : algorithme qui trouve une solution bonne mais pas nécessairement optimale, en temps polynomial, en évitant l'exploration exhaustive de l'espace de recherche.
Pourquoi ? Le problème exact est souvent NP-difficile (ex : ordre optimal des jointures sur n tables = (2n−2)!/(n−1)! plans). L'heuristique sacrifie l'optimalité pour la praticabilité.

Exemples d'heuristiques en BD :

  • Greedy (glouton) : à chaque étape, choisir localement la meilleure action. Ex : sélectionner la coupure de fragmentation qui réduit le plus le coût immédiat.
  • Nearest neighbor : ordonner les jointures en commençant par la plus sélective, puis ajouter la table la plus proche (en coût) à chaque pas.
  • Dynamic programming (DP) : mémoriser les sous-plans optimaux déjà calculés → évite la recalcul. Optimal mais coût O(3ⁿ) → limité à ~10-15 tables.
  • Simulated annealing / Genetic : pour >15 tables, explorer l'espace des plans de façon stochastique.
Heuristique glouton (ex1) : initialiser F={R}, puis répéter : évaluer toutes les coupures (σ ou π) sur chaque fragment, appliquer celle qui réduit le plus le coût total, jusqu'à |F|=K.
→ Sous-optimal mais correct et O(K² × m × (n_pred + n_attr))
Une heuristique ne garantit pas l'optimum global — elle peut rater la vraie solution si le gain local mène à un mauvais optimum local.

Problème d'optimisation à contrainte

Forme générale :
Minimiser (ou maximiser) une fonction objectif f(x)
sous des contraintes : g(x) ≤ K, h(x) = 0, …
→ trouver x* tel que f(x*) est minimal parmi les x valides.

Exemple : fragmentation mixte (ex1, Q2)

Variables : choix des prédicats horizontaux P_H et projections verticales P_V définissant la fragmentation F

Fonction objectif :
Min Σ freq(q_i) × coût(q_i, F)
(coût total d'exécution de la charge de requêtes)

Contrainte principale :
|F| ≤ K_max
(nombre maximal de fragments admissible)

Contraintes de correction (obligatoires) :
• Complétude : tout tuple ∈ R est dans ≥ 1 fragment
• Reconstruction : ∪⋈ F_i = R
• Disjonction (FH) : chaque tuple dans 1 seul fragment H
Omettre les contraintes de correction produit une solution qui minimise le coût mais ne permet plus de reconstituer R — la fragmentation est inutilisable

Complexité du problème exact :

L'espace de recherche est de taille O(K² × (n_pred + 2^n_attr))
→ exponentiel en n_attr à cause des 2^n_attr projections verticales possibles
→ Variante polynomiale : restreindre aux groupements d'attributs prédéfinis → O(K² × m × (n_pred + n_groupes))
Lien RBO/CBO/Heuristique :
RBO = contraintes de correction appliquées comme règles fixes (pas de coût)
CBO = minimisation explicite de f(x) = coût estimé
Heuristique = résoudre le problème CBO de façon approchée en temps polynomial

BNLJ — Mécanisme & formule

Principe : charger un lot de M blocs de R en mémoire, parcourir S entièrement pour trouver les correspondances, puis passer au lot suivant de R.
Pour chaque lot de M blocs de R :
  Charger le lot en mémoire
  Pour chaque bloc de S :
    Pour chaque tuple r du lot :
      Pour chaque tuple s du bloc :
        Si r.A = s.B → émettre (r, s)
Coût = B(R) + ⌈B(R) / M⌉ × B(S)
VariableSignification
B(R)Blocs table externe — lue 1 fois en tout
B(S)Blocs table interne — relue intégralement à chaque lot
MBuffer en blocs (1 slot réservé pour lire S bloc à bloc)
⌈B(R)/M⌉Nombre de lots = nombre de fois où S est relue
Mnémo : S paie le prix de chaque lot de R — plus M est grand, moins il y a de lots, moins S est relue.
Toujours mettre la plus petite table en interne (S) pour minimiser ⌈B(R)/M⌉ × B(S)

Cas limites importants

M = 1 : 1 tuple de R à la fois → ⌈B(R)/1⌉ = B(R) lots → Coût = B(R) + B(R)×B(S) ≈ B(R)×B(S) — catastrophique
S tient en mémoire (B(S) ≤ M−1) : charger S une seule fois → 1 seul lot sur R → Coût = B(R) + B(S) — optimal
Mémoire optimale : M = √B(R) + 1 → ⌈B(R)/M⌉ ≈ √B(R) → Coût ≈ 3·B(R) (si B(R) ≤ B(S))
B(R)=900 → M_opt = √900+1 = 31 → ⌈900/31⌉=29 lots → Coût = 900 + 29×B(S)

Scénarios numériques comparés

Leçon : le push-down σ peut diviser le coût par ×100 — toujours filtrer avant de joindre

Dériver M minimal & M optimal

M minimal pour coût cible C :
B(R) + ⌈B(R)/M⌉ × B(S) ≤ C
⌈B(R)/M⌉ ≤ (C − B(R)) / B(S)
M ≥ B(R) / floor((C − B(R)) / B(S))
B(R)=1000, B(S)=500, C=6000
⌈1000/M⌉ ≤ (6000−1000)/500 = 10
M ≥ 1000/10 = 100 blocs
M optimal (coût minimum) :
M_opt = ⌊√B(R)⌋ + 1
Coût obtenu ≈ B(R) + √B(R) × B(S)
Si B(R) = B(S) → Coût ≈ 3·B(R)
B(R)=400, B(S)=800
M_opt = √400+1 = 21
Coût = 400 + ⌈400/21⌉×800 = 400 + 20×800 = 16 400 blocs
M_opt minimise les I/O mais pas nécessairement le temps si B(R) ≠ B(S) — vérifier aussi l'inversion des rôles

Hash Join 2 phases — Mécanisme

Phase 1 — Partition : appliquer h(attribut_join) sur R et S → k partitions. Tuples avec même valeur → même partition. Écrire toutes les partitions sur disque.
Phase 2 — Probe : pour chaque paire (Ri, Si), charger Ri en RAM (table de hachage), parcourir Si et sonder.
Phase 1 : lire R (1×) + écrire R (1×) = 2·B(R)
          lire S (1×) + écrire S (1×) = 2·B(S)
Phase 2 : lire R (1×) + lire S (1×)  = B(R)+B(S)
──────────────────────────────────────────────────
Total   = 3·(B(R) + B(S))
Contrainte M : chaque partition Ri doit tenir en mémoire → M ≥ √B(R) + 1
Si M < √B(R) → partitions trop grandes → spill → Hash Join récursif → coût > 3·(B(R)+B(S))
B(R)=400, B(S)=200, M=25
M ≥ √400+1 = 21 ✓ → Coût = 3×(400+200) = 1 800 blocs

Sort-Merge Join — Mécanisme

Phase 1 : tri externe de R et S sur l'attribut de jointure (2 passes si B(R) > M²)
Phase 2 : fusion des deux listes triées en 1 passage linéaire — 2 pointeurs avancent en parallèle
Coût : tri = 2·B(R) + 2·B(S) (écrire+relire) + merge = B(R)+B(S) → 3·(B(R)+B(S)) au total
Si données déjà triées (index ou ORDER BY préalable) : seulement la phase de merge → Coût = B(R)+B(S)
Sort-Merge supporte les jointures non-équi (<, >) — Hash Join ne le peut pas

Comparaison des 4 algorithmes

AlgoCoûtFavorable si…Interdit si…
BNLJB(R)+⌈B(R)/M⌉×B(S)S petite ou tient en RAMM=1, grandes tables

Quand Index NLJ bat tout : S très sélective (peu de tuples qualifiants) + index clustered → accès quasi-directs, moins de blocs que 3(B(R)+B(S))

Lecture d'un plan Oracle (EXPLAIN)

Se lit de bas en haut : les feuilles s'exécutent en premier, chaque nœud parent reçoit le résultat de ses enfants.
HASH JOIN                            ← jointure finale
  HASH GROUP BY                      ← agrégation
    TABLE ACCESS FULL : Commandes    ← scan (pas d'index)
  INDEX RANGE SCAN : idx_clients_id  ← B-Tree sur id

Structure interne d'un B-Tree

Arbre équilibré : toutes les feuilles sont à la même profondeur. Hauteur typique = 3-4 niveaux pour des millions de tuples.
         [50 | 80]           ← racine (nœud interne)
        /    |    \
  [20|35] [60|70] [85|95]  ← nœuds internes
     ↓↓↓    ↓↓↓     ↓↓↓
  (val→RID) …              ← feuilles (entrées triées)

Nœuds internes : contiennent des clés de routage. Servent uniquement à guider la descente.

Feuilles : contiennent les couples (valeur_clé, RID) où RID = pointeur vers le tuple physique. Les feuilles sont chaînées (scan en plage).

Recherche : descendre depuis la racine en comparant les clés → O(log n). Pour 1M tuples ≈ 20 entrées/nœud → hauteur ≈ 4 niveaux.
Scan en plage : atteindre la première feuille, puis parcourir la chaîne de feuilles vers la droite jusqu'à la limite haute.

Clustered vs Non-Clustered

ClusteredNon-Clustered
Tuples physiquesTriés selon l'indexNon triés (ordre quelconque)
Nombre par table1 seul possiblePlusieurs possibles
Accès plageTrès efficace (séquentiel)Aléatoire (1 I/O par tuple)
Coût INSERT/UPDATEPlus élevé (réorganisation)Plus faible
Clustered sur date → WHERE date BETWEEN '2024-01' AND '2024-06' lit des blocs contigus. Non-clustered : chaque tuple qualifiant peut être dans un bloc différent.
Avec un index non-clustered qui ramène beaucoup de tuples, chaque tuple = 1 accès disque aléatoire → peut être plus lent qu'un scan complet

Règle du préfixe (index composite)

Un index composite (A, B, C) impose que A doit être présent dans le prédicat pour que l'index soit utilisable. Sans A, l'arbre ne peut pas être traversé dans le bon ordre.
Prédicat WHEREidx(A,B,C) ?Pourquoi
A = 1✓ OuiPréfixe A présent
A = 1 AND B > 5✓ OuiPréfixe A + filtre B
A = 1 AND B = 3 AND C = 2✓ OuiToutes les colonnes
B = 3✗ NonA absent → ordre cassé
B = 3 AND C = 2✗ NonA absent
A = 1 AND C = 2~ PartielA utilisé, C filtré à la volée

✓ WHERE dept='RH'

Covering Index (Index de couverture)

Un index est dit couvrant si toutes les colonnes du SELECT et du WHERE sont dans l'index → aucun accès à la table physique (index-only scan)
CREATE INDEX cov ON Emp(dept, annee, salaire);
SELECT salaire FROM Emp WHERE dept='RH' AND annee=2022
→ Toutes les colonnes (dept, annee, salaire) dans l'index → 0 accès table
Sélectivité : si la requête ramène > 5-10% des tuples, l'optimiseur préfère un scan complet (moins d'I/O aléatoires qu'un index non-clustered)
Trop d'index = maintien coûteux sur écritures. Chaque INSERT/UPDATE doit mettre à jour tous les index de la table.

Index Bitmap — Fonctionnement

Pour chaque valeur distincte d'un attribut, on stocke un vecteur de N bits (1 bit par tuple de la table) : bit=1 si le tuple a cette valeur.
Table (6 tuples) :
id  couleur  taille
1   ROUGE    L
2   BLEU     M
3   VERT     S
4   ROUGE    L
5   BLEU     L
6   ROUGE    M

Bitmap couleur :        Bitmap taille :
ROUGE : 1 0 0 1 0 1    L : 1 0 0 1 1 0
BLEU  : 0 1 0 0 1 0    M : 0 1 0 0 0 1
VERT  : 0 0 1 0 0 0    S : 0 0 1 0 0 0
Requête multi-critères : WHERE couleur='ROUGE' AND taille='L'
→ AND bit-à-bit : 101001 AND 100110 = 100000
→ Résultat : tuple 1 seulement (bit 1 à position 1)
OR : WHERE couleur='ROUGE' OR couleur='BLEU'
→ OR bit-à-bit : 100101 OR 010010 = 110111 → tuples 1,2,4,5,6

Bitmap — Quand et pourquoi

BitmapB-Tree
CardinalitéFaible (≤100 val.)Haute (milliers+)
RequêtesMulti-AND/OR analytiquesÉgalité, plage, tri
ContexteOLAP (lecture)OLTP (lecture+écriture)
EspaceN bits / valeur distincteProportionnel à N tuples
Bitmap très coûteux en écriture concurrente : verrouillage au niveau du vecteur entier → à éviter en OLTP (INSERT/UPDATE fréquents)
Haute cardinalité → vecteur sparse (99% de zéros) → peu d'avantage, gaspillage mémoire
region (8 valeurs) → Bitmap idéal. email (unique) → Bitmap catastrophique.

BJI — Pourquoi le B-Tree ne suffit pas

Dans un schéma étoile, la table Faits est énorme. Filtrer sur plusieurs dimensions simultanément avec des B-Trees ordinaires oblige à :
1. Lire tous les tuples Faits liés à chaque dimension filtrée
2. Faire l'intersection en mémoire → toujours beaucoup d'I/O sur Faits
BJI solution : pour chaque paire (Dimension D, Table Faits F), stocker directement D_RID → liste de F_RIDs. La jointure est pré-calculée.
-- Exemple BJI(Client, Ventes) :
Client RID 42 (pays='France')  → Ventes RIDs {1,5,8,12,99}
Client RID 43 (pays='Germany') → Ventes RIDs {3,7,11}
Client RID 44 (pays='France')  → Ventes RIDs {2,6,17,88}

BJI — Pipeline d'exécution détaillé

Requête : SUM(montant) WHERE pays='France' AND cat='Electronique'
  1. Filtre dim. Client : σ_{pays='France'}(Client) → D_RIDs = {42, 44}
  2. Lookup BJI_Client : RIDs 42 et 44 → F_RIDs_A = {1,2,5,6,8,12,17,88,99}
  3. Filtre dim. Produit : σ_{cat='Electro'}(Produit) → D_RIDs = {10, 11}
  4. Lookup BJI_Produit : → F_RIDs_B = {1,4,6,12,30,88}
  5. Intersection : F_RIDs_A ∩ F_RIDs_B = {1, 6, 12, 88}
  6. Accès physique Faits : lire seulement les 4 tuples {1,6,12,88}
Clé : étapes 1-5 s'exécutent sur des entiers en mémoire — 0 accès physique à Faits avant l'étape 6
BJIs activésRéduction accès Faits
1 BJI (1 filtre dim.)~−41%
2 BJIs (2 filtres dim.)~−70%
3 BJIs (3 filtres dim.)~−93%
BJI coûteux à maintenir (INSERT dans Faits = mise à jour de tous les BJIs concernés) → pertinent uniquement en lecture intensive (OLAP)

Index de Hachage (Hash Index)

Principe : h(valeur) transforme la clé en numéro de bucket. Chaque bucket contient les RIDs des tuples ayant cette valeur → accès direct O(1).
h("Paris") = 3 → bucket 3 → {RID 12, RID 47}
h("Lyon")  = 7 → bucket 7 → {RID 5, RID 91}
Recherche "Paris" : 1 calcul → bucket 3 → résultat
OpérationHashB-Tree
Égalité =✓ O(1)✓ O(log n)
Plage BETWEEN✗✓
ORDER BY✗✓
LIKE 'a%'✗✓
Pourquoi plages impossibles : h("Paris")=3 et h("Lyon")=7 n'ont aucun ordre — valeurs proches dispersées dans des buckets arbitraires. Impossible de "parcourir" l'index de façon ordonnée.
B-Tree presque toujours préféré car il couvre égalité et plages. PostgreSQL propose Hash Index mais l'optimiseur CBO choisit souvent B-Tree.
Hash Join ≠ Hash Index : Hash Join (p.4) hache les tuples en mémoire temporairement pendant la jointure — pas un index persistant sur disque.

Opérations OLAP — avec SQL

  • Roll-up : agréger vers le haut (jour → mois → année)
    GROUP BY annee au lieu de GROUP BY jour
  • Drill-down : décomposer vers le bas (pays → région → ville)
    Ajouter region au GROUP BY
  • Slice : fixer une valeur d'une dimension
    WHERE annee = 2024
  • Dice : sous-cube multi-dimensions
    WHERE annee IN (2023,2024) AND region='Nord'
  • Pivot : croiser deux dimensions en lignes/colonnes
    CASE WHEN region='Nord' THEN montant END
ROLLUP SQL : génère automatiquement les sous-totaux
GROUP BY ROLLUP(pays, region, ville)
→ produit : (pays,region,ville) + (pays,region) + (pays) + ()
CUBE SQL : toutes les combinaisons possibles
GROUP BY CUBE(pays, annee)
→ produit : (pays,annee) + (pays) + (annee) + ()

Vues Matérialisées — Syntaxe

Analyse coût : direct vs VM

Coût direct : freq × coût_requête
Coût VM : freq × coût_lecture_VM + nb_refresh/jour × coût_refresh
Requête : JOIN 3 tables, 10M lignes → 50ms. Exécutée 1000×/jour.

Réécriture transparente

L'optimiseur détecte qu'une requête est couverte par une VM et la substitue silencieusement, sans modifier le SQL de l'utilisateur.
Conditions pour la réécriture :
1. La VM contient toutes les colonnes du SELECT
2. Le WHERE de la requête est un sous-ensemble de celui de la VM
3. Le GROUP BY est compatible
4. La VM est fraîche (pas expirée)
Si la VM est périmée (refresh non effectué), l'optimiseur peut ignorer la réécriture et exécuter la requête directement sur les tables de base

Énoncé

On dispose d'une base distribuée avec les tables suivantes :

Employes(id, nom, dept, salaire, site)
Projets(id_proj, titre, budget, site_proj)

La table Employes est fragmentée ainsi :

  • E1 = σ_{site='Paris'}(Employes) stocké sur Site A
  • E2 = σ_{site='Lyon'}(Employes) stocké sur Site B
  • Puis pour chaque fragment, projection : E1v = π_{id,nom,dept}(E1), E1s = π_{id,salaire}(E1)

La table Projets est entière sur Site C.

Questions :

  1. Quel type de fragmentation pour Employes ?
  2. Vérifier la reconstructibilité et la disjonction
  3. Réécrire la requête suivante sur les fragments :
    SELECT nom, salaire FROM Employes WHERE site='Paris' AND dept='IT'
  4. Optimiser avec push-down et éliminer fragments inutiles
  5. Écrire la requête distribuée finale

Résolution pas-à-pas

  1. Type de fragmentation :
    E1, E2 = sélection sur site → FH (horizontale)
    E1v, E1s = projection sur colonnes → FV (verticale)
    → Fragmentation globale : Mixte HV (H d'abord, puis V sur chaque fragment H)
  2. Vérifier reconstructibilité et disjonction :
    Complétude : site='Paris' ∪ site='Lyon' — couvre-t-il tous les tuples ? Non si ∃ site='Bordeaux'. ⚠ À vérifier selon le domaine.
    Disjonction : un employé ne peut avoir site='Paris' ET site='Lyon' simultanément ✓
    Reconstruction : Employes = (E1v ⋈_{id} E1s) ∪ (E2v ⋈_{id} E2s)
  3. Réécriture initiale : 
    SELECT nom, salaire FROM E1v, E1s
WHERE E1v.id=E1s.id AND site='Paris' AND dept='IT'
UNION
SELECT nom, salaire FROM E2v, E2s
WHERE E2v.id=E2s.id AND site='Lyon' AND dept='IT'
  4. Optimiser — push-down et élimination :
    Fragment E2 a site='Lyon' → prédicat site='Paris' est une contradiction → éliminer E2 complètement
    Push-down dept='IT' sur E1v : applicable car E1v contient dept ✓
    Colonnes nom ∈ E1v, salaire ∈ E1s → besoin des deux fragments E1v et E1s
  5. Requête distribuée finale : 
    -- Exécuté sur Site A uniquement :
SELECT E1v.nom, E1s.salaire
FROM E1v JOIN E1s ON E1v.id = E1s.id
WHERE E1v.dept = 'IT';
    Aucun transfert réseau nécessaire — tout sur Site A ✓

    σ_{categorie='VIP' AND region='Nord'} → 10% × 5% = 0.5% de Clients
    B(Clients_filtrés) = 100 × 0.005 = ~1 bloc
    Clients filtrés tient en M (1 << 40) → charger une seule fois
    Coût = B(Commandes) + B(Clients_filtrés) = 1000 + 1 = 1 001 blocs
  3. Hash Join :
    M ≥ √B(Clients)+1 = √100+1 = 11 → M=40 suffisant ✓
    Coût = 3×(1000 + 100) = 3 300 blocs

    Push-down BNLJ = 1 001 blocs < Hash Join = 3 300 blocs < BNLJ brut = 3 500 blocs
    → Plan Oracle optimal : 
    HASH GROUP BY
  NESTED LOOPS
    TABLE ACCESS FULL : Commandes       ← externe
    TABLE ACCESS (by index) : Clients   ← interne
      INDEX RANGE SCAN : idx_clients_id
    FILTER: categorie='VIP' AND region='Nord'
    En pratique : appliquer filtre Clients AVANT la jointure

    1000 + ⌈1000/M⌉ × 1 ≤ 1500

Ex. 2 Exercice type — Requêtes réparties & Fragmentation mixte dérivée
Énoncé

On dispose de deux relations :

  • R(Enum, Name, Age, Rank) — Employés
  • S(Enum, Jnum, Resp, Dur) — Projets (Dur en mois)

Les fragments définis sont :

Fragment Définition Type Site
S1 σDur<12(S) FH Site A
S2 σDur≥12(S) FH Site B
R1 πEnum,Name,Age(R ⋈ S1) Dérivée Site A
R2 πEnum,Rank(R ⋈ S1) Dérivée Site A
R3 πEnum,Name,Age(R ⋈ S2) Dérivée Site B
R4 πEnum,Rank(R ⋈ S2) Dérivée Site B

Hypothèse : Tout employé de R participe à au moins un projet de S (∀ Enum ∈ R, ∃ Enum ∈ S).

Requête Q :

SELECT R.Name
FROM R, S
WHERE R.Enum = S.Enum
  AND (S.Dur = 30 OR R.Age = 55)

Questions :

  1. Reconstruire R et S depuis les fragments
  2. S est-il disjoint ? R est-il disjoint ?
  3. Réécrire Q sur les fragments (avant optimisation)
  4. Optimiser en utilisant l'hypothèse
⚠ Fragmentation dérivée de R
R1, R2, R3, R4 sont des fragments dérivés : leur partitionnement dépend de S (via la jointure R⋈S1/S2), pas d'un prédicat direct sur R. Résultat : R peut être non-disjoint si un employé participe à des projets dans les deux fragments de S.

Résolution pas-à-pas

  1. Q1 — Reconstruction de S :
    S1 et S2 forment une partition de S par Dur :
    S = S1 ∪ S2 (union simple, pas de jointure nécessaire)
    Vérification : S1 ∩ S2 = ∅ car Dur<12 XOR Dur≥12 → reconstructibilité OK
  2. Q1 — Reconstruction de R :
    R est fragmenté verticalement après jointure avec S1 et S2 :
    R = (R1 ∪ R3) ⋈Enum (R2 ∪ R4)
    Pourquoi la jointure ? R1/R3 contiennent {Enum,Name,Age}, R2/R4 contiennent {Enum,Rank} → il faut les réunir sur Enum pour reconstituer tous les attributs.
    Pourquoi sans perte ? Enum est clé primaire de R → la jointure sur Enum est injective.
  3. Q2 — Disjonction :
    S est disjoint ✓ : Dur<12 et Dur≥12 sont mutuellement exclusifs → aucun tuple dans S1∩S2.
    R est non-disjoint ✗ : un employé (ex: Enum=E3) peut avoir un projet avec Dur=6 (dans S1) ET un projet avec Dur=24 (dans S2). Il apparaît donc dans R1 ET dans R3 simultanément.
    Conséquence : une requête sur R sans précaution peut compter E3 deux fois → il faut traiter les doublons (UNION sans ALL, ou DISTINCT)
  4. Q3 — Réécriture sur fragments (avant optimisation) :
    On substitue R et S par leurs fragments. La condition Dur=30 OR Age=55 génère deux branches :
    Branche Dur=30 → Dur=30 ≥ 12 → uniquement dans S2 → fragment R3 (qui porte Age,Name)
    Branche Age=55 → Age est dans R1 et R3 (fragments de R côté {Enum,Name,Age}) 
    -- Branche 1 : Dur=30 (dans S2 uniquement)
SELECT R3.Name FROM R3, S2
WHERE R3.Enum = S2.Enum AND S2.Dur = 30
UNION
-- Branche 2 : Age=55 côté S1 (Dur<12)
SELECT R1.Name FROM R1
WHERE R1.Age = 55
UNION
-- Branche 3 : Age=55 côté S2 (Dur≥12)
SELECT R3.Name FROM R3
WHERE R3.Age = 55
    Note : les branches 2 et 3 n'ont pas besoin de joindre S car on cherche juste le nom de l'employé (pas d'attribut de S dans SELECT). La jointure est implicitement satisfaite via la construction de R1/R3.
  5. Q4 — Optimisation avec l'hypothèse :
    Hypothèse : ∀ Enum ∈ R, ∃ Enum ∈ S → tout employé de R a au moins un projet → la jointure R⋈S n'élimine aucun tuple de R.
    Dans les branches Age=55, la jointure avec S était implicite mais inutile (on ne sélectionne rien de S) :
    Avant : "Employés de R avec Age=55 qui ont au moins un projet dans S" = jointure obligatoire
    Après : "Employés de R avec Age=55" → tous ont un projet par hypothèse → supprimer la jointure 
    -- Plan optimisé
SELECT R3.Name FROM R3, S2
WHERE R3.Enum = S2.Enum AND S2.Dur = 30
UNION
SELECT R1.Name FROM R1 WHERE R1.Age = 55  -- sans jointure S
UNION
SELECT R3.Name FROM R3 WHERE R3.Age = 55  -- sans jointure S
    Gain : branches 2 et 3 deviennent de simples scans de R1/R3 filtrés — élimination d'une semi-jointure coûteuse sur chaque site.
