1. Fragmentation Horizontale (FH)

Opérateur : Ri = σ_pi(R) (sélection par prédicat)

Reconstruction : R = R1 ∪ R2 ∪ ... ∪ Rn

PropriétéDéfinition
Complétude∀t ∈ R, ∃i : t ∈ Ri (aucune donnée perdue)
DisjonctionRi ∩ Rj = ∅ (pas de duplication)
ReconstructionR = ∪ Ri (opération inverse possible)

Exemple R1 = σ_{Pays='FR'}(R), R2 = σ_{Pays='DE'}(R)

2. Fragmentation Verticale (FV)

Opérateur : Ri = π_Ai(R) (projection sur colonnes)

⚠️ Clé primaire dupliquée dans chaque fragment

Reconstruction : R = F1 ⋈ F2 ⋈ ... (jointure sur PK)

Exemple F1 = π_{id,nom,adresse}(R), F2 = π_{id,salaire}(R)

3. Reconstruction — Récapitulatif

TypeOpérateurCondition
Horizontale∪Prédicats couvrent R
Verticale⋈ (PK)PK dans chaque fragment
Mixte HV∪ puis ⋈Reconstituer H, puis joindre V
Mixte VH⋈ puis ∪Joindre V, puis réunir H

4. HV vs VH — Différences clés

HV (H puis V)VH (V puis H)
OrdreFiltrer lignes → projeter colonnesProjeter colonnes → filtrer lignes
Nb fragmentsn_H × n_V≤ n_V × n_H (souvent moins)
RisqueAucun⚠️ Colonne de filtrage peut avoir été supprimée par V
VH : si la colonne du prédicat H a été projetée hors de V, le fragment H devient vide ou impossible.

5. Fragmentation Dérivée

Fragments obtenus par jointure avec d'autres fragments, puis projection :

R1 = π_cols(R ⋈ S1)
R2 = π_cols(R ⋈ S2)

Usage Distribuer une table en suivant la localisation d'une autre (semi-jointure implicite).

6. Fragmentation Non-Disjointe

Un même tuple peut apparaître dans plusieurs fragments.

Conséquences :

  • Lecture : OK, données redondantes mais complètes.
  • MAJ (INSERT/UPDATE/DELETE) : propagation requise sur tous les fragments contenant le tuple → coût élevé, risque d'incohérence.

Exemple Fragment par région ET par responsable → un vendeur couvrant 2 régions apparaît 2 fois.

7. Réécriture pour Exécution Distribuée

Algorithme de réécriture :

  1. Décomposer le WHERE en branches (conditions OR)
  2. Pour chaque prédicat, identifier quels fragments contiennent l'attribut filtré
  3. Éliminer les fragments impossibles par contradiction de prédicat
  4. Combiner les résultats des fragments restants avec UNION
-- Requête originale
SELECT * FROM R WHERE Dur=30 OR Dur=6

-- Après réécriture (fragments : R1:Dur<12, R2:Dur≥12)
SELECT * FROM R1 WHERE Dur=6   -- Dur=30 éliminé (30<12 faux)
UNION
SELECT * FROM R2 WHERE Dur=30  -- Dur=6 éliminé (6<12 vrai → dans R1)

8. Élimination Jointures Redondantes

Si ∀ tuple ∈ R, ∃ clé correspondante ∈ S (contrainte d'intégrité référentielle garantie) → la jointure avec S est superflue.

Elle peut être supprimée si S n'apporte aucune colonne au SELECT.

Push-down en distribué : appliquer σ et π avant les jointures inter-sites → réduction des transferts réseau.

9. BNLJ — Formule & Composants

Coût = B(R) + ⌈B(R) / M⌉ × B(S)
VariableSignification
B(R)Blocs table externe (lue 1 fois)
B(S)Blocs table interne (relue entièrement à chaque lot)
MTaille buffer en blocs (mémoire vive)
⌈B(R)/M⌉Nombre de lots/itérations sur R

Mnémo "R est chargé, S est relu" — S paie le prix de chaque lot.

10. BNLJ — Scénarios Numériques

B(Livraison)=2000, B(Fournisseur)=2000, débit=10 000 blocs/s

ScénarioCoût (blocs)Durée
M=1 (externe=Liv.)2000 + 2000×2000 = 4 002 000~6m40s
M=20 (externe=Liv.)2000 + 100×2000 = 202 000~20s
M=20 + push-down σ2000 + 2×2000 = 6 000~0.6s
Inversion rôles (S filtrée tient en mémoire)4 000~0.4s
Règle : choisir la plus petite table en interne ; appliquer σ avant jointure.

Inversion Si S filtrée tient dans le buffer → chargée 1 seule fois → coût = B(R) + B(S_filtré).

11. Index B-Tree

Efficace pour : égalité (=), plage (BETWEEN), tri (ORDER BY).

Règle préfixe : index (A, B, C) utilisable si prédicat porte sur :

  • A seul ✓ — (A, B) ✓ — (A, B, C) ✓
  • B seul ✗ — C seul ✗ — (B, C) sans A ✗

Sélectivité : utile si résultat < ~5-10% de la table. Au-delà, scan complet peut être préférable.

CREATE INDEX idx_nom ON R (A, B, C);
-- Utilisable : WHERE A=1 AND B>5
-- Inutilisable : WHERE B=3 (A absent)

12. Binary Join Index (BJI) — Pipeline

Structure : D_RID → {F_RIDs} (pré-matérialise la jointure D⋈F)

Pipeline 4 étapes :

  1. Filtre D : appliquer σ sur dimension → D_RIDs
  2. Lookup BJI : pour chaque D_RID → récupérer F_RIDs
  3. Intersection : croiser les sets F_RIDs de tous les BJIs actifs (opération sur identifiants, pas données physiques)
  4. Accès physique : lire uniquement les tuples de F désignés
Nb BJIs activésRéduction tuples F
1 BJI~-41%
2 BJIs~-70%
3 BJIs~-93%
Contexte : schéma en étoile OLAP. Coût : maintenance sur INSERT/UPDATE — pertinent pour lecture intensive.

15. OLAP — Pourquoi ça existe ?

Les bases OLTP (transactionnelles) sont normalisées pour éviter la redondance. Résultat : une requête analytique comme "total des ventes par région, par mois, par catégorie de produit" nécessite 5-6 jointures sur des millions de lignes → très lent.

OLAP = stocker les données dans un format dénormalisé, optimisé pour la lecture analytique en acceptant la redondance.

Idée clé : on tolère la redondance pour avoir des lectures rapides. Les données OLAP sont souvent en lecture seule (chargées en batch, pas mises à jour tuple par tuple).

Opérations OLAP typiques :

  • Roll-up : agréger vers le haut (jour → mois → année)
  • Drill-down : décomposer vers le bas (pays → région → ville)
  • Slice : fixer une dimension (ex : seulement 2025)
  • Dice : sous-cube sur plusieurs dimensions
  • Pivot : pivoter les axes d'analyse

16. Schéma en Étoile — Structure & Concept

Table de Faits = ce qu'on mesure. Très volumineuse (millions à milliards de lignes). Contient : des FK vers chaque dimension + les mesures numériques.

Tables de Dimensions = le contexte descriptif. Petites (milliers à millions de lignes). Contiennent les attributs qui servent de filtres et de regroupements.

         Client(id, nom, ville, pays)
              ↑ FK_Client
Ventes(FK_Client, FK_Produit, FK_Date, FK_Magasin,
       montant, quantite, remise)
              ↓ FK_Produit    ↓ FK_Date
         Produit(id, nom,  Date(id, jour,
          categorie, prix)  mois, annee,
                            trimestre)

Règle La table de faits ne contient que des FK et des mesures — jamais de libellés textuels (ceux-ci sont dans les dimensions).

Schéma Flocon (Snowflake) : variante où les dimensions sont elles-mêmes normalisées (ex : Produit → Catégorie → Rayon). Plus économe en espace, mais plus de jointures.

Étoile = 1 jointure par dimension. Flocon = plusieurs jointures par dimension. Étoile préféré en OLAP pour la performance.

17. Index B-Tree — Fonctionnement Interne

Un B-Tree est un arbre équilibré. Chaque nœud interne contient des clés de routage. Les feuilles contiennent les couples (valeur, pointeur vers le tuple physique).

          [50 | 80]           ← nœud racine
         /    |    \
   [20|35] [60|70] [85|95]  ← nœuds internes
      ↓↓↓    ↓↓↓     ↓↓↓
   (valeur → RID)           ← feuilles

Recherche : descendre depuis la racine en comparant les clés → O(log n), typiquement 3-4 niveaux pour des millions de tuples.

Index Clustered : les tuples sont physiquement triés dans l'ordre de l'index. 1 seul par table. Idéal pour les plages.

Index Non-Clustered : les feuilles pointent vers les tuples (qui peuvent être n'importe où). Plusieurs possibles. Chaque accès = 1 I/O aléatoire.

Si la requête ramène >5-10% des tuples, le SGBD ignore l'index et fait un scan complet (moins d'I/O aléatoires). L'optimiseur décide.

Règle préfixe (composite) : index (A,B,C) → utilisable uniquement si A est dans le prédicat. Sans A, l'arbre ne peut pas être traversé dans le bon ordre.

18. Index Bitmap — Pour OLAP

Pour chaque valeur possible d'un attribut, on stocke un vecteur de bits (1 bit par tuple) :

Couleur   tuple: 1 2 3 4 5 6
ROUGE          : 1 0 0 1 0 1
BLEU           : 0 1 0 0 1 0
VERT           : 0 0 1 0 0 0

Avantage : opérations AND/OR/NOT sur les bitmaps = très rapide, traite des millions de tuples en quelques cycles CPU.

Exemple WHERE couleur='ROUGE' AND taille='L' → AND bit-à-bit des deux bitmaps → positions des tuples qualifiants.

Idéal quand : faible cardinalité (peu de valeurs distinctes : couleur, genre, région). Mauvais sur les attributs à haute cardinalité (id, email).

B-Tree : haute cardinalité, requêtes ponctuelles, OLTP. Bitmap : faible cardinalité, requêtes analytiques multi-critères, OLAP.

19. BJI — Pourquoi le B-Tree Ne Suffit Pas

Problème du schéma étoile : la table de faits est énorme. Une requête analytique filtre sur plusieurs dimensions à la fois :

SELECT SUM(montant) FROM Ventes V
JOIN Client C ON V.FK_Client = C.id
JOIN Produit P ON V.FK_Produit = P.id
WHERE C.pays = 'France'
  AND P.categorie = 'Electronique'

Avec un B-Tree sur FK_Client : on récupère les tuples France, puis on joint Produit → on lit quand même beaucoup de tuples de Faits inutiles.

Solution BJI : pré-matérialiser la jointure sous forme de bitmap.

Le BJI entre Client et Ventes = pour chaque tuple Client, la liste des RIDs dans Ventes qui lui correspondent.

Client RID 42 (France) → Ventes RIDs {1,5,8,12,99,...}
Client RID 43 (Germany) → Ventes RIDs {3,7,11,...}

Avec 2 BJIs (Client + Produit) :

  1. BJI_Client(pays='France') → set F_RIDs A
  2. BJI_Produit(cat='Electro') → set F_RIDs B
  3. Intersection A ∩ B = F_RIDs qualifiants
  4. Lire seulement ces tuples dans Ventes
On n'a jamais touché physiquement à la table Faits jusqu'à l'étape 4. L'intersection est 100% sur des identifiants en mémoire.

20. Vues Matérialisées — Cas d'Usage Détaillés

Quand utiliser une vue mat. :

  • Agrégat coûteux recalculé souvent : SUM, COUNT, AVG sur des millions de lignes
  • Jointure répétée entre tables stables (dimensions OLAP)
  • Requête complexe exposée à des utilisateurs BI (Tableau, Power BI)

Quand ne pas utiliser :

  • Données très fréquemment mises à jour (fraîcheur compromise)
  • Requête exécutée une seule fois

Réécriture transparente : l'optimiseur détecte automatiquement que la requête est couverte par une vue mat. → la substitue sans modifier le SQL.

-- Requête utilisateur
SELECT region, SUM(montant) FROM Ventes
GROUP BY region;
-- → optimiseur utilise silencieusement
--   la vue mat. pre_ventes_region si elle existe
Vue simple (VIEW) = juste un alias de requête, recalculée à chaque appel. Vue matérialisée = résultat stocké physiquement sur disque.

21. Comparaison Algorithmes de Jointure

AlgoFormule coûtCondition
BNLJB(R) + ⌈B(R)/M⌉ × B(S)Toujours applicable
Sort-Merge3(B(R)+B(S))M suffisant pour trier
Hash Join3(B(R)+B(S))M ≥ √B(R)
Index NLJB(R) + |R| × coût_idxIndex sur attr. jointure de S
Sort-Merge et Hash Join = même coût O(B(R)+B(S)) mais Hash n'exige pas de données triées. BNLJ reste utile quand M est très petit.

BNLJ — Pseudo-code :

Pour chaque lot de M blocs de R :
  charger M blocs de R en RAM
  Pour chaque bloc b de S :
    Pour chaque r dans le lot :
      Pour chaque s dans b :
        si r.A = s.B → émettre (r,s)

Cas limite M ≥ B(R)+2 → R tient entièrement en RAM → coût = B(R)+B(S).

22. Taille en Blocs — Calcul

B(R) = ⌈ NbTuples / FacteurBlocage ⌉

Ex : 20 000 tuples, tuple=200o, bloc=4096o
FB = ⌊4096/200⌋ = 20
B(R) = ⌈20000/20⌉ = 1 000 blocs

Cardinalité d'une jointure :

|R⋈S| ≈ |R|×|S| / max(V(A,R), V(A,S))

où V(A,T) = nb valeurs distinctes de A dans T.

23. Semi-Jointure Distribuée

Problème : R sur site 1, S sur site 2 → transférer une table entière = coûteux.

  1. Projeter S sur l'attribut de jointure : S' = π_A(S) → envoyer S' (petit) au site 1
  2. Filtrer R : R' = R ⋉ S' (seules les lignes matchant)
  3. Envoyer R' au site 2 → jointure finale R' ⋈ S
Gain si |S'| + |R'| ≪ |R| + |S|. Inutile si R matche entièrement avec S.
R ⋉ S = π_R(R ⋈ S)
(tuples de R ayant un partenaire dans S)

26. Transactions Distribuées — 2PC

2PC (Two-Phase Commit) : garantir l'atomicité d'un commit sur plusieurs sites.

Phase 1 — Vote : coordinateur envoie PREPARE à chaque participant. Chacun répond VOTE_COMMIT (données sauvegardées en log) ou VOTE_ABORT.

Phase 2 — Décision :

  • Tous COMMIT → GLOBAL_COMMIT → chaque site commit
  • ≥1 ABORT → GLOBAL_ABORT → chaque site rollback
Point de blocage : si le coordinateur tombe après avoir envoyé PREPARE mais avant la décision → participants bloqués indéfiniment (verrous maintenus). 3PC ajoute une phase "pre-commit" pour contourner ce problème.

Règle d'ordre des colonnes dans l'index

1. Colonnes d'égalité (=) — 2. Colonnes de range (<, >, BETWEEN) — 3. Colonnes de tri (ORDER BY)

Requête cible

WHERE semestre='S1'    -- égalité
  AND note > 12        -- range

Index optimal

CREATE INDEX idx ON Inscriptions (semestre, note, nom);

L'index (semestre, note) seul permet déjà un Index Only Scan si nom est inclus.

Règle préfixe & sélectivité

Un index (A, B) est utilisable si A est dans le prédicat :

  • WHERE A=1 AND B>5 ✓ — préfixe A présent
  • WHERE B>5 ✗ — A absent, index inutilisable

Sélectivité : indexer si le résultat représente < ~5–10% des lignes. Au-delà, le Seq Scan est parfois préféré par l'optimiseur.

Gain mesuré exemple : 2 400 ms → ~40 ms (×60) avec index composite adapté sur (semestre, note)

Index Covering (Index Only Scan)

Inclure dans l'index toutes les colonnes du SELECT → l'accès au tas (heap) est évité :

29. Hash Join — 2 Phases Détaillées

Phase 1 — Partitionnement : lire R et S, créer k partitions sur disque

Coût phase 1 = 2(|R| + |S|) — 1 lecture + 1 écriture des partitions

Phase 2 — Probing : pour chaque paire (Ri, Si) :

  1. Charger Ri en RAM → construire table de hachage
  2. Scanner Si → sonder la table de hachage
Coût phase 2 = |R| + |S| — relecture des partitions
Total = 3(|R| + |S|) = phase 1 + phase 2

Condition mémoire : M ≥ √|R| + 1

(les partitions de R doivent tenir en mémoire)

Cas optimal (one-pass) : si |R| ≤ M → phase 1 inutile

Coût one-pass = |R| + |S| (borne inférieure atteinte !)

Exemple R=100 pages, S=1000, M=12

M ≥ √100+1 = 11 ✓ → coût = 3×1100 = 3300 pages

Sort-Merge3(B(R)+B(S))M ≥ √B(R)Données partiellement triées, ORDER BY final

Choix pratique Données triées → Sort-Merge. Non triées, grandes → Hash Join. Index dispo sur S → Index NLJ.

32. Semi-Jointure — Coût Détaillé & Condition d'Avantage

f = facteur de réduction = fraction de S matchant avec R (0 < f ≤ 1)

3 étapes et coûts :

ÉtapeOpérationCoût
1Projeter R sur attribut jointure A → envoyer π_A(R)|R|
2Semi-joindre S avec π_A(R) → écrire S' = f·|S||S| + |π_A(R)|
3Hash Join R ⋈ S'3(|R| + f·|S|)
Total semi-jointure ≈ 4|R| + |S|(1 + 4f)

Coût jointure directe (Hash Join) = 3(|R| + |S|)

Condition d'avantage : semi-jointure < Hash Join direct

4|R| + |S|(1+4f) < 3(|R|+|S|)
→ f < (2|S| − |R|) / (4|S|)

Attention Inutile si f ≈ 1 (R et S matchent presque entièrement). Efficace seulement si f est petit (transfert réseau réduit).

33. Full Reducer — Jointures n-Way

Problème : ordonner n jointures est NP-difficile → approche par réduction successive via semi-jointures.

Algorithme Full Reducer (2 passes) :

Passe avant (gauche → droite) :

R₂ ← R₂ ⋉ R₁
R₃ ← R₃ ⋉ R₂
...
Rₙ ← Rₙ ⋉ Rₙ₋₁

Passe arrière (droite → gauche) :

Rₙ₋₁ ← Rₙ₋₁ ⋉ Rₙ
...
R₁ ← R₁ ⋉ R₂

Jointure finale : R₁ ⋈ R₂ ⋈ ... ⋈ Rₙ sur les relations réduites

Propriété : après les 2 passes, chaque tuple restant contribue au résultat final — aucun tuple "dangling".

Contexte Optimisation distribuée où le coût réseau domine — on minimise les tuples transférés avant la jointure finale.

34. Borne Inférieure de Coût & Mémoire Optimale BNLJ

Borne inférieure absolue : toute jointure R1 ⋈ R2 nécessite de lire chaque tuple au moins une fois :

Coût minimum = |R1| + |R2| (impossible de faire moins)

Exemple R1=100 pages, R2=1000 → borne inf = 1100 pages

Atteindre la borne avec BNLJ :

Condition : R1 (la plus petite) tient dans M−2 pages de buffer

M ≥ |R_petite| + 2 → coût = |R1| + |R2| (optimal)

Exemple M ≥ 100+2 = 102 → coût = 1100 pages = borne inférieure ✓

Impact de M sur le coût BNLJ (R1=100, R2=1000) :

M (pages)Coût BNLJRatio / borne
3100 + 100×1000 = 100 100×91
12100 + 10×1000 = 10 100×9
52100 + 2×1000 = 2 100×1.9
102100 + 1×1000 = 1 100×1.0 ✓
Chaque doublement de M réduit drastiquement le coût. Investir en mémoire = levier majeur d'optimisation.

41. Méthodologie de Sélection d'Index

3 étapes :

  1. Identifier les candidats : parcourir chaque requête, relever chaque attribut dans prédicat WHERE/JOIN/ORDER BY → liste des candidats avec leur fréquence
  2. Classifier le type selon le prédicat dominant et la cardinalité :
    • Égalité + haute cardinalité → B-tree
    • Égalité + faible cardinalité (< 100 valeurs distinctes) → Bitmap
    • Égalité seule, pas de range ni sort → Hash
    • Range ou ORDER BY → B-tree obligatoire
  3. Sélectionner le top-k : priorité aux attributs couvrant le plus de requêtes, puis à la sélectivité la plus forte

Exemple (base HR, 7 requêtes R1–R7)

CandidatTypeRequêtes

Travaille(NSS)B-treeR5, R6, R7
Travaille(N°Pro, Heures)B-tree compositeR4, R5, R6
Règle composite : colonne la plus sélective en 1ère position ; ajouter les colonnes additionnelles si elles éliminent un accès table (index covering).
