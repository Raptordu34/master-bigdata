Un index de jointure binaire (BJI, Binary Join Index) entre une table de dimensions D et la table de faits F mémorise, pour chaque tuple di ∈ D, l'ensemble des RIDs des tuples de F qui se joignent avec di sur la clé étrangère.

Principe d'utilisation en 4 étapes :

  • Appliquer le prédicat de sélection sur D → obtenir les D_RIDs qualifiants.
  • Lire le BJI pour chaque D_RID → obtenir les F_RIDs correspondants dans la table de faits.
  • Si plusieurs BJIs sont utilisés (plusieurs dimensions filtrées), intersecter les ensembles de F_RIDs.
  • Accéder physiquement aux seuls tuples de F désignés par l'ensemble résultant.

Avantage clé : on réduit considérablement le nombre de tuples de la table de faits lus, sans jamais parcourir F en entier. La phase d'intersection opère sur des identifiants (RIDs), pas sur les données.

Structure : un BJI peut être vu comme une liste de couples (D_RID, F_RID) pour tous les tuples qui se joignent. En pratique, on le groupe par D_RID : pour chaque tuple de la dimension, on stocke la liste des F_RIDs correspondants dans la table de faits.

Exemple — BJI(Client, Ventes) sur C.CID = V.CID

  • C_RID=1 (Pascal, CID=111) → V_RIDs : {7, 8, 16}
  • C_RID=2 (Eric, CID=212) → V_RIDs : {6, 9, 15, 17, 21, 22, 23, 24}
  • C_RID=3 (Ladjel, CID=313) → V_RIDs : {10, 25, 26, 27}
  • C_RID=4 (Yamine, CID=414) → V_RIDs : {5, 11, 12}
  • C_RID=5 (Yves, CID=515) → V_RIDs : {4, 13, 14, 18}
  • C_RID=6 (Gilles, CID=616) → V_RIDs : {1, 2, 3, 19, 20}

Pour la condition C.City = 'Poitiers', les C_RIDs qualifiants sont {1, 2, 6}. L'union de leurs V_RIDs donne les candidats dans Ventes : {1, 2, 3, 6, 7, 8, 9, 15, 16, 17, 19, 20, 21, 22, 23, 24} — soit 16 tuples sur 27.

Analogie

Image mentale : le BJI est comme un carnet de correspondances pré-établi. Chaque client (dimension) a une liste pré-calculée de toutes ses ventes (faits). Pour répondre à une question sur les clients de Poitiers, il suffit de consulter les carnets des 3 clients concernés et d'agréger leurs listes — sans parcourir les 27 lignes de Ventes une par une.

Avec plusieurs BJIs, on fait l'intersection des listes : seules les ventes présentes dans toutes les listes (client de Poitiers ET produit Beauté ET mois de Juin) sont finalement lues.

Méthode générale

  • Identifier les prédicats de sélection sur les tables de dimensions.
  • Pour chaque dimension concernée, retrouver les D_RIDs qualifiants.
  • Utiliser le BJI pour projeter chaque ensemble de D_RIDs en V_RIDs.
  • Intersecter les V_RID sets (opération sur identifiants, sans lire Ventes).
  • Accéder physiquement aux seuls tuples de Ventes identifiés.
Q13 Intermédiaire

Proposer trois scénarii d'optimisation de la requête en utilisant :

  • Scénario 1 : un seul index de jointure binaire
  • Scénario 2 : deux index de jointure binaire
  • Scénario 3 : trois index de jointure binaire
Piège courant : Créer un BJI entre deux tables de dimensions (ex : Client–Produit). Un BJI est toujours défini entre une table de dimensions D et la table de faits F — jamais entre deux dimensions. L'objectif est de pré-localiser les tuples de Ventes, pas de relier des dimensions entre elles.
La requête Q comporte trois prédicats sur des dimensions : C.City = 'Poitiers', P.Type = 'Beauté', T.Mois = 'Juin'. Il existe donc naturellement un BJI possible pour chaque dimension (Client→Ventes, Produit→Ventes, Temps→Ventes). Les scénarii correspondent à en utiliser 1, 2 ou 3.
Scénario 1 : choisir l'une des trois dimensions, par exemple Client, et utiliser BJI(Client, Ventes). Scénario 2 : ajouter BJI(Produit, Ventes) et intersecter les deux V_RID sets. Scénario 3 : utiliser les trois BJIs et intersecter les trois ensembles pour obtenir directement les tuples de Ventes satisfaisant toutes les conditions de jointure.

Solution — Q13

Chaque scénario utilise un ou plusieurs BJI(Dimension, Ventes). Le principe est identique : filtrer la dimension, obtenir les V_RIDs par l'index, puis intersecter si plusieurs index sont actifs.

Scénario 1 — Un seul BJI

BJI utilisé : BJI(Client, Ventes) sur C.CID = V.CID

  • Sélectionner dans Client les tuples avec City = 'Poitiers' → C_RIDs = {1, 2, 6}
  • Via BJI : V_RIDs candidats = {1, 2, 3, 6, 7, 8, 9, 15, 16, 17, 19, 20, 21, 22, 23, 24} (16 tuples)
  • Lire ces 16 tuples de Ventes, puis joindre avec Produit et Temps pour appliquer les prédicats restants.

Scénario 2 — Deux BJIs

BJIs utilisés : BJI(Client, Ventes) + BJI(Produit, Ventes)

  • Client, City='Poitiers' → C_RIDs={1,2,6} → V_RIDs_C = {1,2,3,6,7,8,9,15,16,17,19,20,21,22,23,24}
  • Produit, Type='Beauté' → P_RIDs={5,6} → V_RIDs_P = {1,2,5,6,16,21,22,23,24,25,26,27}
  • Intersection V_RIDs_C ∩ V_RIDs_P = {1, 2, 6, 16, 21, 22, 23, 24} (8 tuples)
  • Lire ces 8 tuples de Ventes, puis vérifier T.Mois='Juin' par jointure avec Temps.

Scénario 3 — Trois BJIs

BJIs utilisés : BJI(Client, Ventes) + BJI(Produit, Ventes) + BJI(Temps, Ventes)

  • V_RIDs_C = {1,2,3,6,7,8,9,15,16,17,19,20,21,22,23,24} (via Client, Poitiers)
  • V_RIDs_P = {1,2,5,6,16,21,22,23,24,25,26,27} (via Produit, Beauté)
  • Temps, Mois='Juin' → T_RID={6} → V_RIDs_T = {2, 5, 13, 16, 17, 25} (via BJI(Temps, Ventes))
  • Intersection V_RIDs_C ∩ V_RIDs_P ∩ V_RIDs_T = {2, 16} (2 tuples)
  • Accéder uniquement à ces 2 tuples de Ventes — toutes les conditions sont satisfaites sans jointure supplémentaire.

Q14 Avancé

Montrer comment l'optimiseur de requêtes utilise ces index dans les trois scénarii pour traiter la requête Q.

Piège courant : Penser que l'optimiseur lit d'abord toute la table Ventes, puis applique les filtres. Avec un BJI, le traitement se décompose en deux phases bien distinctes : (1) résolution de l'index au niveau RID — sans accéder aux données de Ventes — puis (2) accès physique ciblé aux seuls tuples identifiés.
Pour chaque scénario, l'optimiseur suit les mêmes étapes générales : (1) sélection sur la dimension concernée, (2) lecture du BJI pour obtenir les V_RIDs, (3) union des V_RIDs pour cette dimension, (4) intersection avec les autres dimensions si plusieurs BJIs, (5) accès direct aux tuples de Ventes résultants.
Dans le scénario 3, après intersection des trois ensembles de V_RIDs, l'optimiseur obtient directement {2, 16}. Il n'accède qu'à ces 2 tuples dans Ventes. Toutes les conditions dimensionnelles ayant été résolues au niveau des index, aucune jointure supplémentaire n'est nécessaire pour calculer le Count(*).

Solution — Q14

Scénario 1 — BJI(Client, Ventes)

  • Parcourir Client, sélectionner City = 'Poitiers' → C_RIDs = {1, 2, 6}.
  • Lire BJI(Client, Ventes) pour C_RIDs = {1, 2, 6} → V_RIDs = {1,2,3,6,7,8,9,15,16,17,19,20,21,22,23,24}.
  • Accéder aux 16 tuples de Ventes désignés.
  • Pour chaque tuple de Ventes, joindre avec Produit (vérifier P.Type='Beauté') et Temps (vérifier T.Mois='Juin').

Scénario 2 — BJI(Client) + BJI(Produit)

  • Client, City='Poitiers' → C_RIDs = {1, 2, 6} → V_RIDs_C = {1,2,3,6,7,8,9,15,16,17,19,20,21,22,23,24}.
  • Produit, Type='Beauté' → P_RIDs = {5, 6} → V_RIDs_P = {1,2,5,6,16,21,22,23,24,25,26,27}.
  • Intersection V_RIDs_C ∩ V_RIDs_P = {1,2,6,16,21,22,23,24} — opération sur RIDs, sans lire Ventes.
  • Accéder aux 8 tuples de Ventes, puis vérifier T.Mois='Juin' par jointure avec Temps.

Scénario 3 — BJI(Client) + BJI(Produit) + BJI(Temps)

  • V_RIDs_C = {1,2,3,6,7,8,9,15,16,17,19,20,21,22,23,24} (Client, Poitiers).
  • V_RIDs_P = {1,2,5,6,16,21,22,23,24,25,26,27} (Produit, Beauté).
  • Temps, Mois='Juin' → T_RID = {6} → V_RIDs_T = {2,5,13,16,17,25} (BJI(Temps, Ventes)).
  • Intersection finale : {1,2,3,6,7,8,9,15,16,17,19,20,21,22,23,24} ∩ {1,2,5,6,16,21,22,23,24,25,26,27} ∩ {2,5,13,16,17,25} = {2, 16}.
  • Accéder aux 2 tuples de Ventes V_RID ∈ {2, 16} — toutes les conditions sont satisfaites, Count(*) = 2.
Point clé : dans tous les scénarii, les intersections d'ensembles de V_RIDs s'effectuent au niveau de l'index, sans déclencher de lecture physique dans la table de faits. Seule la dernière étape (accès aux tuples désignés) touche effectivement les données de Ventes.
Q15 Fondamental

Quel est le résultat retourné par cette requête ?

Piège courant : Compter les tuples de Ventes qui satisfont seulement l'une des conditions (ex : uniquement CID de Poitiers = 16 lignes, ou uniquement PID de type Beauté = 12 lignes). Il faut que les trois prédicats soient simultanément vérifiés sur un même tuple de Ventes : CID d'un client de Poitiers, ET PID d'un produit Beauté, ET TID de Juin.
Utilisez le résultat du Scénario 3 (Q13) : l'intersection des trois ensembles de V_RIDs donne directement les tuples de Ventes qualifiant toutes les conditions. Comptez ces V_RIDs.

Solution — Q15

Les seuls tuples de Ventes satisfaisant simultanément les trois conditions sont V_RID = 2 et V_RID = 16 :

Q16 Fondamental

Est-ce que le résultat dépend des scénarii ?

Piège courant : Confondre résultat (valeur retournée par la requête SQL) et performance (coût d'exécution, nombre de tuples lus). Les scénarii influencent uniquement l'efficacité du plan d'accès — pas la valeur retournée par Count(*).
Un index est une structure auxiliaire qui accélère l'accès aux données, mais ne modifie jamais les données elles-mêmes. La requête SQL est sémantiquement identique dans les trois scénarii — seul le chemin d'accès physique change.

Solution — Q16

Non, le résultat ne dépend pas des scénarii.

Dans les trois scénarii, la requête SQL est strictement la même. Les index de jointure binaire sont des structures d'optimisation physique : ils accélèrent l'accès sans modifier le contenu ni la sémantique des données.

ScénarioTuples Ventes accédésCount(*)
1 — BJI(Client)16 (puis filtrage sur Produit et Temps)2
2 — BJI(Client) + BJI(Produit)8 (puis filtrage sur Temps)2
3 — 3 BJIs2 (accès direct, résultat immédiat)2
Principe général : un plan d'exécution différent peut produire le même résultat avec moins de ressources. C'est précisément le rôle de l'optimiseur : trouver le chemin d'accès le plus efficace pour une même requête sémantique, sans jamais en altérer le résultat.
Q17 Avancé

Quel est le nombre de tuples de la table des faits que l'optimiseur a passé en revue (scanné) lors de l'exécution de la requête ?

Piège courant : Répondre 27 (taille totale de Ventes) en supposant un parcours séquentiel complet. Avec des BJIs, l'optimiseur ne parcourt pas Ventes pendant la phase d'index : il opère sur des RIDs. Seuls les tuples désignés par l'intersection finale sont physiquement lus.
La réponse dépend du scénario utilisé. Pour chaque scénario, comptez le nombre de V_RIDs dans l'ensemble obtenu après intersections — c'est le nombre de tuples de Ventes que l'optimiseur doit lire physiquement pour calculer le résultat.
Scénario 1 : l'ensemble V_RIDs après 1 BJI contient 16 éléments → 16 tuples lus. Scénario 2 : intersection de 2 ensembles → 8 tuples. Scénario 3 : intersection des 3 ensembles → 2 tuples. Plus on ajoute de BJIs, moins on lit de tuples dans la table de faits.

Solution — Q17

Le nombre de tuples de Ventes physiquement scannés dépend directement du scénario :

ScénarioV_RIDs après intersectionsTuples Ventes scannésRéduction vs scan complet (27)
Sans BJI (scan séquentiel)tous les 2727—
1 — BJI(Client){1,2,3,6,7,8,9,15,16,17,19,20,21,22,23,24}16−41 %
2 — BJI(Client) + BJI(Produit){1,2,6,16,21,22,23,24}8−70 %

Avec 3 BJIs (scénario 3), l'optimiseur ne scanne que 2 tuples sur 27, soit une réduction de 93 % du volume d'accès physique à la table des faits.

En production, une table de faits peut contenir des milliards de tuples. Le gain proportionnel des BJIs est alors déterminant pour rendre les requêtes OLAP exécutables en temps interactif.

Pour aller plus loin
  • Si vous n'utilisez qu'un seul BJI, lequel choisir ? Comparez les sélectivités : Client (3 qualifiants sur 6 → 16 V_RIDs), Produit (2/6 → 12 V_RIDs), Temps (1/6 → 6 V_RIDs). Plus la dimension est sélective, plus le BJI seul réduit les tuples de Ventes à lire.

  • Un BJI occupe de l'espace disque supplémentaire et doit être maintenu lors des INSERT/UPDATE dans Ventes ou dans les dimensions. Comment arbitrer entre le coût de maintenance et le gain en lecture dans un contexte décisionnel ?
