Q1.1 – Q1.3 : VM Performances Étudiants

Fondamental ◎ 3 questions ⏱ 20–30 min
Objectifs

  • Comparer l'exécution directe et via vue matérialisée
  • Analyser les avantages des VMs sur de grands volumes de données

Prérequis : Schéma de la BD — section précédente

Q1.1 Vue des performances étudiantes Fondamental

Créer une vue matérialisée VM_Performance_Etudiants qui affiche, pour chaque étudiant : son ID, son nom complet (Nom + Prénom), son niveau d'études, le nombre total de cours suivis et sa moyenne générale.

Piège courant : Oublier les colonnes non agrégées dans le GROUP BY. Toute colonne du SELECT qui n'est pas dans une fonction d'agrégation (COUNT, AVG…) doit apparaître dans le GROUP BY.
Utilisez un LEFT JOIN (pas INNER JOIN) pour conserver les étudiants sans aucune inscription (ils apparaîtront avec Nb_Cours_Suivis = 0 et Moyenne_Generale = NULL).
La concaténation en SQL standard : e.Nom || ' ' || e.Prenom AS Nom_Complet. Groupez sur e.ID_etudiant, e.Nom, e.Prenom, e.Niveau pour rester portable sur tous les SGBD.

Solution — Q1.1

CREATE MATERIALIZED VIEW VM_Performance_Etudiants AS
SELECT
    e.ID_etudiant,
    e.Nom || ' ' || e.Prenom          AS Nom_Complet,
    e.Niveau,
    COUNT(i.ID_cours)                 AS Nb_Cours_Suivis,
    ROUND(AVG(i.Note), 2)             AS Moyenne_Generale
FROM   Etudiants e
LEFT JOIN Inscriptions i ON e.ID_etudiant = i.ID_etudiant
GROUP BY
    e.ID_etudiant, e.Nom, e.Prenom, e.Niveau;
Pourquoi LEFT JOIN ? Un étudiant inscrit à zéro cours doit quand même apparaître dans la vue (avec Nb_Cours_Suivis = 0 et Moyenne_Generale = NULL). Un INNER JOIN exclurait ces étudiants.

Sans GROUP BY : SELECT AVG(Note) FROM Inscriptions retourne une seule ligne — la moyenne de toutes les notes confondues.

Avec GROUP BY e.ID_etudiant : retourne une ligne par étudiant, chacune avec sa propre moyenne.

Règle fondamentale : toute colonne dans le SELECT qui n'est pas une agrégation doit apparaître dans le GROUP BY. C'est pourquoi on écrit GROUP BY e.ID_etudiant, e.Nom, e.Prenom, e.Niveau. PostgreSQL l'accepte avec seulement ID_etudiant (car c'est une PK), mais écrire toutes les colonnes reste la pratique portable et explicite.
Q1.2 Requête directe vs vue matérialisée Fondamental

Un responsable pédagogique veut les performances de tous les étudiants de niveau Master. Écrire les deux versions de la requête : directe (sans VM) et via la vue matérialisée.

Piège courant : Penser que les deux requêtes ont le même coût d'exécution. La requête directe refait à chaque fois le JOIN et le GROUP BY sur toutes les tables sources ; la requête via VM lit directement des données pré-calculées.
La requête via VM est un simple SELECT * FROM VM_Performance_Etudiants WHERE Niveau = 'Master'. Le SGBD lit directement les lignes stockées — aucun JOIN ni GROUP BY à l'exécution.

Solution — Q1.2

1. Requête directe (sans VM)

-- Accès direct aux tables de base
SELECT
    e.ID_etudiant,
    e.Nom || ' ' || e.Prenom  AS Nom_Complet,
    e.Niveau,
    COUNT(i.ID_cours)         AS Nb_Cours_Suivis,
    ROUND(AVG(i.Note), 2)     AS Moyenne_Generale
FROM   Etudiants e
LEFT JOIN Inscriptions i
    ON e.ID_etudiant = i.ID_etudiant
WHERE  e.Niveau = 'Master'
GROUP BY
    e.ID_etudiant,
    e.Nom, e.Prenom,
    e.Niveau;

2. Requête réécrite (avec VM)

-- Lecture directe de la vue pré-calculée
SELECT *
FROM   VM_Performance_Etudiants
WHERE  Niveau = 'Master';
Le SGBD lit directement les lignes stockées dans la VM — aucun JOIN ni GROUP BY à l'exécution.
Q1.3 Analyse des avantages Intermédiaire

Analyser les avantages de la vue matérialisée dans le contexte suivant : 50 000 étudiants · 500 000 inscriptions · requête exécutée 100 fois par jour.

Piège courant : Ne pas mentionner la fraîcheur des données comme inconvénient. La VM stocke un instantané — les données peuvent être décalées d'un cycle de rafraîchissement.
Comparez le coût pour 100 exécutions : mode direct = 100 × (JOIN 550 000 lignes + GROUP BY) ; mode VM = 1 rafraîchissement nocturne + 100 lectures légères.
Pensez aux 5 critères : opérations par exécution, coût CPU/I/O, coût total journée, temps de réponse utilisateur, fraîcheur des données, et coût de stockage supplémentaire.

Solution — Q1.3

Critère Requête directe Via VM_Performance_Etudiants
Opérations à chaque exécution Scan Etudiants (50 000 lignes) + Scan Inscriptions (500 000) + JOIN + GROUP BY + AVG Scan de la VM (~50 000 lignes précalculées) + filtre WHERE simple
Coût CPU / I/O par exécution Élevé — plusieurs centaines de milliers de lectures disque Très faible — lecture séquentielle d'une seule table
Coût total sur la journée (×100) 100 × (coût JOIN + agrégation) → charge CPU constante 1 rafraîchissement nocturne + 100 lectures légères
Temps de réponse utilisateur Plusieurs secondes (à l'échelle millions de lignes) Millisecondes
Fraîcheur des données Toujours à jour Données du dernier rafraîchissement
Coût de stockage Aucun surcoût ~50 000 lignes supplémentaires stockées

Conclusion : Avec 100 exécutions quotidiennes, le coût d'un JOIN de 550 000 lignes est payé 100 fois par jour en mode direct, contre une seule fois (lors du rafraîchissement) en mode VM. Pour un usage en lecture intensive (tableau de bord, reporting), la vue matérialisée est le choix optimal.

La légère perte de fraîcheur (données décalées d'un cycle de rafraîchissement) est généralement acceptable dans un contexte académique où les notes et inscriptions ne changent pas à la minute.

Réécriture automatique (Query Rewriting) : des SGBD comme Oracle ou PostgreSQL peuvent réécrire automatiquement la requête directe pour utiliser la VM lorsque le planificateur détecte qu'elle est équivalente et plus efficace — sans modifier le code applicatif.
