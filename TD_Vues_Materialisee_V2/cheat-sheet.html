MÉMENTO SQL : FONDAMENTAUX & VUES MATÉRIALISÉES (1/2)

Structure de base & Syntaxe
Ordre d'exécution logique : FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY 
SELECT col1, SUM(col2) AS total
FROM Table1 t1
LEFT JOIN Table2 t2 ON t1.id = t2.id
WHERE t1.cond = 'val'
GROUP BY col1
HAVING SUM(col2) > 100
ORDER BY total DESC;
Mots-clés clés :
  • DISTINCT : Supprime les doublons.
  • AS : Alias de colonne ou table (optionnel).
  • IN / BETWEEN / LIKE : Opérateurs de filtrage.
  • IS NULL / IS NOT NULL : Test de vacuité.
Jointures : Règles d'Or
  • INNER JOIN : Retourne uniquement les correspondances exactes.
  • LEFT JOIN : Garde toutes les lignes de la table de gauche, complète avec NULL à droite si pas de match.
Utiliser LEFT JOIN pour ne pas perdre d'entités "vides" (ex: étudiants sans aucune note).
Agrégations & Group By
Fonctions : COUNT(), SUM(), AVG(), MIN(), MAX().
La Règle d'Or : Toute colonne présente dans le SELECT qui n'est pas enveloppée dans une fonction d'agrégation DOIT figurer dans le GROUP BY. 
-- CORRECT (Portable)
SELECT Niveau, COUNT(*) 
FROM Etudiants 
GROUP BY Niveau;
Vues Matérialisées (VM) : Concept
Définition : Une VM est une vue dont le résultat est stocké physiquement sur disque (contrairement à une vue simple qui réexécute la requête à chaque appel).
Avantages :
  • Performances extrêmes en lecture (pré-calcul).
  • Économie CPU/IO (on ne refait pas les JOIN/GROUP BY).
Inconvénients :
  • Données potentiellement périmées (staleness).
  • Consommation d'espace disque.
  • Coût de maintenance (rafraîchissement).
Syntaxe VM (PostgreSQL/Standard)
-- Création
CREATE MATERIALIZED VIEW VM_Vents AS
SELECT mois, SUM(montant) FROM Ventes
GROUP BY mois;

-- Rafraîchissement (Manuel ou Script)
REFRESH MATERIALIZED VIEW VM_Vents;

-- Suppression
DROP MATERIALIZED VIEW VM_Vents;
Modélisation : Schéma en Étoile
  • Table de Faits : Au centre, contient les mesures (ex: Ventes) et les FK.
  • Tables de Dimensions : Autour, contiennent les attributs descriptifs (ex: Temps, Client, Produit).
Les VM sont idéales pour agréger les faits selon les dimensions les plus consultées.
Analyse Comparative : Direct vs VM
CritèreRequête DirecteVia VM
CalculÀ chaque exécutionUne seule fois (MAJ)

La VM est rentable si :
Coût(Refresh) + Σ Coût(Lecture VM) < Σ Coût(Lecture Directe)
Fonctions Utilitaires
Texte :
  • || : Concaténation (ex: nom || ' ' || prenom).
  • UPPER() / LOWER() : Casse.
  • SUBSTR(str, start, len) : Extraction.
Maths :
  • ROUND(val, nb_decimales).
  • COALESCE(val, 0) : Remplace NULL par 0.
Dates :
  • EXTRACT(YEAR FROM date).
  • CURRENT_DATE.

MÉMENTO SQL : CONCEPTS AVANCÉS & OPTIMISATION (2/2)

Sélection de VM : Le Problème
Objectif : Choisir le meilleur sous-ensemble de vues S pour minimiser le coût global du workload W.
Complexité : NP-Difficile (Sac-à-dos). Pour n requêtes, 2ⁿ combinaisons possibles.
Fonction Objectif : 
Minimiser Σ (freq_qᵢ * Coût(qᵢ, S))
Contraintes :
  • Espace B : Σ Taille(VMᵢ) ≤ B
  • Temps T : Σ Coût_MAJ(VMᵢ) ≤ T_max
Logique Conditionnelle
SELECT nom,
  CASE 
    WHEN note >= 10 THEN 'Admis'
    WHEN note >= 8 THEN 'Rattrapage'
    ELSE 'Ajourné'
  END AS Statut
FROM Resultats;
Heuristique Gloutonne (Greedy)
Algorithme utilisé en pratique pour sélectionner les VM :
  1. Initialiser S = ∅.
  2. Calculer pour chaque vue candidate vᵢ son bénéfice marginal : Bénéfice = Σ gain_sur_requêtes_W.
  3. Calculer le ratio : Ratio = Bénéfice / Taille(vᵢ).
  4. Sélectionner la vue avec le meilleur ratio tenant dans le budget restant.
  5. Mettre à jour les coûts et recommencer jusqu'à saturation.
Favoriser les VM réutilisables (celles qui couvrent plusieurs requêtes différentes).
Sous-requêtes & CTE
CTE (Common Table Expression) : Plus lisible que les sous-requêtes imbriquées. 
WITH Moyennes AS (
  SELECT id_etudiant, AVG(note) as moy
  FROM Inscriptions GROUP BY id_etudiant
)
SELECT * FROM Moyennes WHERE moy > 15;
Fonctions de Fenêtrage (Window)
Permettent des calculs sur un ensemble de lignes sans les grouper. 
SELECT nom, note,
  RANK() OVER(ORDER BY note DESC) as rang,
  AVG(note) OVER(PARTITION BY classe) as moy_classe
FROM Etudiants;
  • PARTITION BY : Le "Group By" de la fenêtre.
  • ORDER BY : Définit l'ordre pour le calcul (ex: rang, cumul).
Pièges & Conseils d'Examen
NULL dans les agrégats : COUNT(col) ignore les NULL, mais COUNT(*) compte tout.
Filtrage : WHERE filtre avant agrégation, HAVING filtre après.
Query Rewriting : Certains SGBD utilisent une VM automatiquement même si vous interrogez la table de base, si la VM est équivalente.
CONSEIL : En examen, vérifiez toujours si une VM ne peut pas simplifier une requête complexe en utilisant des données pré-agrégées.
