Q1.6 Charge d'enseignement Intermédiaire

Créer une vue matérialisée VM_Charge_Enseignants qui calcule, pour chaque enseignant : son nom complet et son département, le nombre de cours enseignés, le total d'heures d'enseignement et la moyenne du nombre d'étudiants par cours.

Piège courant : Division par zéro pour les enseignants sans affectation. Toujours protéger la division avec NULLIF(..., 0) pour éviter une erreur d'exécution.
Jointure : Enseignants en LEFT JOIN Affectations a LEFT JOIN Inscriptions i ON a.ID_cours = i.ID_cours. Utilisez COALESCE(SUM(a.Nb_heures), 0) pour obtenir 0 au lieu de NULL quand un enseignant n'a pas d'affectation.
Pour la moyenne d'étudiants par cours : ROUND(COUNT(i.ID_inscription) * 1.0 / NULLIF(COUNT(DISTINCT a.ID_cours), 0), 1). Le * 1.0 force la division en nombre décimal (évite la division entière).

Solution — Q1.6

CREATE MATERIALIZED VIEW VM_Charge_Enseignants AS
SELECT
    en.ID_enseignant,
    en.Nom || ' ' || en.Prenom              AS Nom_Complet,
    en.Departement,
    COUNT(DISTINCT a.ID_cours)              AS Nb_Cours_Enseignes,
    COALESCE(SUM(a.Nb_heures), 0)           AS Total_Heures,
    ROUND(
        COUNT(i.ID_inscription) * 1.0
        / NULLIF(COUNT(DISTINCT a.ID_cours), 0),
    1)                                       AS Moy_Etudiants_Par_Cours
FROM   Enseignants en
LEFT JOIN Affectations a  ON en.ID_enseignant = a.ID_enseignant
LEFT JOIN Inscriptions  i  ON a.ID_cours       = i.ID_cours
GROUP BY
    en.ID_enseignant, en.Nom, en.Prenom, en.Departement;
Formule de Moy_Etudiants_Par_Cours : On divise le nombre total d'inscriptions (toutes affectations confondues) par le nombre de cours distincts enseignés. NULLIF(..., 0) évite une division par zéro pour les enseignants sans affectation. COALESCE(SUM(...), 0) retourne 0 au lieu de NULL pour un enseignant sans affectation.

Pourquoi NULLIF et COALESCE ? — Explication pas à pas

Imaginons un enseignant sans aucune affectation (par exemple, un nouveau recruté qui n'a pas encore de cours assigné) :

  • COUNT(DISTINCT a.ID_cours) → 0 (pas de cours)
  • SUM(a.Nb_heures) → NULL (SUM sur zéro ligne = NULL en SQL)
  • Sans COALESCE : Total_Heures = NULL au lieu de 0 — trompeur dans un rapport.
  • Sans NULLIF : nb_inscriptions / 0 → erreur d'exécution (division par zéro).

Solution combinée :

  • COALESCE(SUM(a.Nb_heures), 0) → si SUM retourne NULL, on affiche 0.
  • NULLIF(COUNT(DISTINCT a.ID_cours), 0) → si le diviseur est 0, on le remplace par NULL, ce qui rend la division nulle (et non une erreur).

Le comportement de x / NULL en SQL est NULL (pas une erreur), ce qui est le résultat souhaité pour un enseignant sans cours.

Exemple de résultat attendu :

Nom_Complet Departement Nb_Cours Total_Heures Moy_Etudiants
Martin SophieInformatique312028.3
Dubois PierreMathématiques28022.5
Leroy AnneInformatique416031.0
Cas d'usage — suivi RH et planification : Cette vue permet à l'administration d'identifier les enseignants surchargés (Total_Heures élevé) ou ceux avec un grand nombre d'étudiants par cours, afin d'adapter les ressources pédagogiques et les tailles de groupes.
Q1.7 Rafraîchissement de VM_Performance_Etudiants Intermédiaire

Écrire les commandes SQL de rafraîchissement de VM_Performance_Etudiants. Proposer et justifier une fréquence de rafraîchissement adaptée au contexte (50 000 étudiants, 500 000 inscriptions, 100 consultations/jour).

Piège courant : Lancer REFRESH MATERIALIZED VIEW VM_Performance_Etudiants sans CONCURRENTLY en production. Sans ce mot-clé, toutes les lectures sont bloquées pendant le rafraîchissement — inacceptable en production avec 100 consultations/jour.
Deux variantes : REFRESH MATERIALIZED VIEW vm; (bloquant) et REFRESH MATERIALIZED VIEW CONCURRENTLY vm; (non bloquant, nécessite un index UNIQUE).
Pour justifier la fréquence, raisonnez sur le rythme de mise à jour des données sources : si les notes sont saisies quotidiennement, un rafraîchissement hebdomadaire laisse les données périmées trop longtemps. Un rafraîchissement à chaque INSERT serait trop coûteux (recalcul sur 500 000 lignes à chaque saisie).

Solution — Q1.7

1. Commandes SQL de rafraîchissement

Rafraîchissement complet 
-- Supprime et recrée toutes les lignes
REFRESH MATERIALIZED VIEW
    VM_Performance_Etudiants;

Simple, toujours disponible. Verrouille la vue pendant le recalcul — la vue est inaccessible pendant quelques secondes.

Rafraîchissement concurrent (PostgreSQL) 
-- Ne met à jour que les lignes modifiées
-- Nécessite un index UNIQUE sur la VM
REFRESH MATERIALIZED VIEW CONCURRENTLY
    VM_Performance_Etudiants;

La vue reste lisible pendant le rafraîchissement. Requiert CREATE UNIQUE INDEX ON VM_Performance_Etudiants (ID_etudiant);

2. Fréquence de rafraîchissement recommandée : quotidienne

-- Exemple avec pg_cron (PostgreSQL)
SELECT cron.schedule(
    'refresh-vm-perf-etudiants',
    '0 2 * * *',                          -- chaque jour à 02h00
    $$REFRESH MATERIALIZED VIEW CONCURRENTLY VM_Performance_Etudiants;$$
);

3. Justification du choix

Fréquence Avantages Inconvénients Adapté ici ?
Temps réel (trigger) Données toujours fraîches Coût élevé à chaque INSERT/UPDATE sur Inscriptions — 500 000 lignes recalculées à la moindre modification Non — disproportionné
Horaire Données fraîches à l'heure près Recalcul 24×/jour inutile si les notes ne changent pas souvent Possible mais excessif
Quotidienne (recommandée) Données fraîches chaque matin — cohérent avec le rythme académique. Coût minimal (1 recalcul/jour vs 100 requêtes) Données potentiellement décalées de 24h max Oui — optimal
Hebdomadaire Très faible charge Données périmées en milieu de semaine si des inscriptions ou notes ont été saisies Non — trop long
Raisonnement final : Les inscriptions et les notes évoluent quotidiennement (saisies par les enseignants en fin de journée). Un rafraîchissement nocturne quotidien garantit que les 100 consultations de la journée s'appuient sur des données à jour du matin, tout en limitant le coût de recalcul à une seule agrégation par 24h sur les 500 000 lignes de la table Inscriptions.

Analogie — le journal quotidien

Choisir la fréquence de rafraîchissement, c'est comme choisir la périodicité d'un journal :

  • Temps réel → réimprimer le journal à chaque nouvelle dépêche : prohibitif, pas de valeur ajoutée.
  • Quotidien → une édition le matin, lue toute la journée : équilibre optimal entre fraîcheur et coût.
  • Hebdomadaire → les nouvelles du lundi sont périmées le vendredi : inacceptable si les données changent chaque jour.

Dans un contexte académique, les notes et inscriptions changent en journée mais pas à la seconde — le rafraîchissement quotidien (nocturne) est le bon équilibre.
