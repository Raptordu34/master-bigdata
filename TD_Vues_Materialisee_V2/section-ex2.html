Q2.1 Formalisation du problème Avancé

Formaliser le problème de sélection de vues matérialisées : définir la fonction objectif, les contraintes, et préciser la complexité du problème.

Piège courant : Croire que l'on peut résoudre ce problème de façon exacte et efficace. C'est un problème NP-difficile (réductible au problème du sac-à-dos) — on doit recourir à des heuristiques.
La fonction objectif minimise le coût total d'exécution du workload : Cost(W, S) = Σ freq(qᵢ) × coût(qᵢ, S) où S est l'ensemble des VMs sélectionnées.
Deux contraintes : espace disque (Σ taille(vᵢ) ≤ B) et coût de maintenance (Σ coût_MAJ(vᵢ) ≤ T_max). Pour n requêtes, il y a jusqu'à 2ⁿ sous-ensembles à évaluer.

Solution — Q2.1

Fonction objectif

On cherche à sélectionner un ensemble S de vues matérialisées qui minimise le coût total d'exécution du workload W :

Minimiser : Cost(W, S) = Σqᵢ ∈ W freq(qᵢ) × coût(qᵢ, S)

Où coût(qᵢ, S) est le coût d'exécution de qᵢ en utilisant les meilleures vues disponibles dans S (ou les tables de base si aucune VM n'est utile).

Contraintes

Contrainte d'espace

La somme des tailles des VMs sélectionnées doit rester sous un budget :

Σvᵢ ∈ S taille(vᵢ) ≤ B

B = espace disque disponible pour les VMs

Contrainte de maintenance

Le coût total de mise à jour des VMs lors d'une modification de la base doit rester acceptable :

Σvᵢ ∈ S coût_MAJ(vᵢ) ≤ T_max

T_max = temps maximal de rafraîchissement toléré

Complexité : ce problème est NP-difficile (réductible au problème du sac-à-dos). Pour n requêtes, il y a jusqu'à 2ⁿ sous-ensembles de VMs candidates à évaluer. On doit donc recourir à des heuristiques ou algorithmes approchés.

Explosion exponentielle — chiffres concrets

Un workload réel contient typiquement des centaines de requêtes candidates. La solution exacte est structurellement impossible : c'est pour cela que l'heuristique gloutonne (Q2.3) est utilisée en pratique.

Q2.2 Solution naïve Intermédiaire

Décrire la solution naïve pour ce problème : comment fonctionne-t-elle, quels sont ses avantages et pourquoi est-elle impraticable en pratique ?

Piège courant : Penser qu'avec 5 requêtes la solution naïve est "raisonnable". 2⁵ = 32 sous-ensembles seulement ici, mais un workload réel compte des centaines de requêtes : 2¹⁰⁰ est astronomique.
La solution naïve énumère toutes les sous-expressions de toutes les requêtes, évalue chaque sous-ensemble possible, et retient celui qui minimise Cost(W, S) sous les contraintes. Son problème est exponentiel en n (le nombre de requêtes).

Solution — Q2.2

Algorithme naïf
  1. Générer toutes les sous-expressions de toutes les requêtes du workload.
  2. Pour chaque sous-ensemble possible de ces expressions → calculer Cost(W, S).
  3. Retenir le sous-ensemble S* qui minimise Cost(W, S) sous les contraintes.
Avantage
  • Garantit la solution optimale
  • Simple à comprendre conceptuellement
Inconvénient
  • Complexité exponentielle : 2ⁿ sous-ensembles à évaluer
  • Totalement impraticable dès que n > 20–30 requêtes
  • L'espace de stockage des VMs serait prohibitif si toutes sont créées
Exemple sur notre workload (5 requêtes) : On pourrait générer jusqu'à 2⁵ = 32 sous-ensembles de vues. Chacune devrait être évaluée en termes de bénéfice sur les 5 requêtes. C'est faisable ici, mais explose pour un workload de 100+ requêtes réelles.
Q2.3 Heuristique de sélection Avancé

Décrire une heuristique gloutonne pour sélectionner les vues matérialisées. L'appliquer aux 5 requêtes du workload pour identifier les vues candidates les plus pertinentes.

Piège courant : Sélectionner des VMs spécialisées pour une seule requête chacune. L'heuristique doit favoriser la réutilisation : une VM couvrant 3 requêtes vaut mieux que trois VMs spécialisées si elle tient dans le budget.
L'algorithme glouton calcule à chaque étape le bénéfice marginal de chaque vue candidate : bénéfice(vᵢ) = Σ freq(qⱼ) × [coût(qⱼ, S) − coût(qⱼ, S ∪ {vᵢ})], puis sélectionne celle avec le meilleur ratio bénéfice / taille.
Pour les 5 requêtes données, identifiez les recouvrements : Q1 et Q4 partagent le JOIN Client ⋈ Ventes ; Q2 et Q5 partagent le JOIN Produit ⋈ Ventes. Une VM couvrant ces sous-expressions bénéficiera à plusieurs requêtes.

Solution — Q2.3

Algorithme glouton

Entrées : workload W = {q₁…qₙ} avec fréquences, budget B.
Initialisation : S = ∅, budget_restant = B.

Répéter jusqu'à saturation du budget :

  • Pour chaque vue candidate vᵢ non encore sélectionnée, calculer son bénéfice marginal :
    bénéfice(vᵢ) = Σ freq(qⱼ) × [coût(qⱼ, S) − coût(qⱼ, S ∪ {vᵢ})]
  • Calculer le ratio : ratio(vᵢ) = bénéfice(vᵢ) / taille(vᵢ)
  • Sélectionner la vue v* avec le plus grand ratio qui tient dans le budget restant.
  • Ajouter v* à S, déduire taille(v*) du budget.
  • Mettre à jour les coûts des requêtes (qui peuvent maintenant utiliser v*).

Critères de l'heuristique :
  • Fréquence d'exécution : une requête exécutée 1 000 fois/jour pèse 10× plus qu'une exécutée 100 fois.
  • Coût de calcul : une requête avec 4 JOINs bénéficie davantage d'une VM qu'une requête simple.
  • Coût de stockage : une VM volumineuse doit apporter un bénéfice proportionnellement plus grand.
  • Réutilisation : une VM couvrant 3 requêtes est préférable à trois VMs spécialisées si elle tient dans le budget.

Ces chiffres sont illustratifs. En pratique, les coûts sont estimés via EXPLAIN ou des statistiques d'utilisation réelles.
