Q1.8 Requêtes sur les vues Fondamental

En utilisant les vues matérialisées créées, écrire les requêtes SQL pour :

  • a) Les 10 étudiants avec la meilleure moyenne générale
  • b) Le département avec la meilleure moyenne étudiante
  • c) Les enseignants ayant plus de 100 heures de cours
Piège courant : Utiliser ORDER BY … DESC sans NULLS LAST. Sous PostgreSQL, DESC place les valeurs NULL en tête par défaut — les étudiants sans note apparaîtraient avant ceux avec une bonne moyenne.
Ces requêtes sont très simples : un SELECT … ORDER BY … LIMIT ou un WHERE direct sur les VMs, sans aucune jointure ni GROUP BY à écrire. C'est l'avantage des données pré-calculées.

Solution — Q1.8

a) Les 10 étudiants avec la meilleure moyenne générale

SELECT
    ID_etudiant,
    Nom_Complet,
    Niveau,
    Nb_Cours_Suivis,
    Moyenne_Generale
FROM   VM_Performance_Etudiants
ORDER BY Moyenne_Generale DESC NULLS LAST
LIMIT  10;

b) Le département avec la meilleure moyenne étudiante

SELECT
    Departement,
    Nb_Cours,
    Nb_Enseignants,
    Moyenne_Etudiants
FROM   VM_Stats_Departement
ORDER BY Moyenne_Etudiants DESC NULLS LAST
LIMIT  1;

c) Les enseignants ayant plus de 100 heures de cours

SELECT
    Nom_Complet,
    Departement,
    Nb_Cours_Enseignes,
    Total_Heures,
    Moy_Etudiants_Par_Cours
FROM   VM_Charge_Enseignants
WHERE  Total_Heures > 100
ORDER BY Total_Heures DESC;
Remarque sur la simplicité des requêtes : Ces trois requêtes sont triviales grâce aux vues matérialisées : un simple SELECT … ORDER BY … LIMIT ou WHERE sur une table pré-calculée, sans aucune jointure ni agrégation à l'exécution. Sans les VMs, chaque requête nécessiterait un JOIN multi-tables + GROUP BY sur les 500 000 lignes d'Inscriptions.
Q1.9 Avantages des vues matérialisées Fondamental

Pourquoi les VMs sont-elles avantageuses pour ces requêtes comparé à l'exécution directe sur les tables de base ?

Piège courant : Répondre uniquement "c'est plus rapide" sans quantifier ni expliquer le mécanisme. Une bonne réponse compare le nombre d'opérations, la lecture disque et l'indexabilité.
Pensez aux 5 aspects : opérations à l'exécution, lecture disque, indexabilité, temps de réponse, et charge serveur cumulée sur 100 exécutions par jour.

Solution — Q1.9

AspectSans VM (tables de base)Avec VM
Opérations à l'exécution JOIN Etudiants ⋈ Inscriptions (550 000 lignes) + GROUP BY + ORDER BY Scan VM (~50 000 lignes) + ORDER BY simple
Lecture disque Toutes les pages des deux tables + index Pages de la VM uniquement
Indexabilité Index sur tables de base, peu utiles après JOIN Index direct sur Niveau, Moyenne_Generale, Total_Heures
Temps de réponse Secondes (voire dizaines de secondes) Millisecondes
Charge serveur (×100/jour) 100 × recalcul complet 100 × lecture légère + 1 rafraîchissement nocturne

Analogie — le caissier et le tableau de bord

Imaginez un caissier interrogé 100 fois par jour : "quel est le total des ventes de chaque étudiant depuis la rentrée ?"

  • Sans VM : à chaque question, il retourne aux archives brutes — il relit et recalcule toutes les inscriptions depuis le début. Chaque réponse prend 30 secondes.
  • Avec VM : il regarde le rapport imprimé ce matin. La réponse est instantanée. Si un étudiant s'est inscrit ce matin même, il ne figure pas encore dans le rapport — mais ce léger décalage est acceptable dans un contexte de reporting pédagogique.
Q1.10 Index sur vue matérialisée Intermédiaire

Proposer un index approprié pour optimiser les recherches d'étudiants par niveau dans la vue VM_Performance_Etudiants.

Piège courant : Créer un index sur chaque colonne séparément alors qu'un index composite (Niveau, Moyenne_Generale DESC) couvre à la fois le filtre WHERE Niveau = 'Master' et le tri ORDER BY Moyenne_Generale DESC en un seul parcours.
La requête la plus fréquente est WHERE Niveau = '…' ORDER BY Moyenne_Generale DESC. Un index sur (Niveau, Moyenne_Generale DESC NULLS LAST) couvre les deux conditions avec un seul index-only scan.
Attention : avec REFRESH MATERIALIZED VIEW (sans CONCURRENTLY), les index doivent être recréés automatiquement. Avec REFRESH CONCURRENTLY, ils sont maintenus en place.

Solution — Q1.10

-- Index pour les filtres WHERE Niveau = 'Master'
CREATE INDEX idx_vm_perf_niveau
    ON VM_Performance_Etudiants (Niveau);

-- Index pour les tris ORDER BY Moyenne_Generale DESC
CREATE INDEX idx_vm_perf_moyenne
    ON VM_Performance_Etudiants (Moyenne_Generale DESC NULLS LAST);

-- Index combiné pour les deux en une seule passe
CREATE INDEX idx_vm_perf_niveau_moy
    ON VM_Performance_Etudiants (Niveau, Moyenne_Generale DESC NULLS LAST);
Impact : avec idx_vm_perf_niveau_moy, la requête WHERE Niveau = 'Master' ORDER BY Moyenne_Generale DESC est satisfaite entièrement par l'index (index-only scan) — le moteur ne touche même pas les pages de données de la VM.

Attention : les index sur une vue matérialisée doivent être recréés (ou mis à jour automatiquement selon le SGBD) après un REFRESH MATERIALIZED VIEW complet. Avec CONCURRENTLY, ils sont maintenus automatiquement.

Qu'est-ce qu'un index-only scan ?

Normalement, pour répondre à une requête, le moteur :

  1. Consulte l'index pour trouver les pointeurs vers les lignes correspondantes.
  2. Lit les pages de données de la table pour récupérer les colonnes demandées.

Avec un index-only scan, l'index contient toutes les colonnes nécessaires à la requête (ici Niveau et Moyenne_Generale). L'étape 2 est entièrement évitée — le moteur ne lit jamais les pages de données.

Analogie : c'est comme trouver la réponse directement dans le sommaire d'un livre, sans même avoir à ouvrir le chapitre correspondant.

Condition : toutes les colonnes du SELECT et du WHERE doivent être incluses dans l'index. C'est pourquoi un index composite bien conçu est souvent plus efficace que deux index simples séparés.

Q1.11 Compromis des vues matérialisées Intermédiaire

Analyser les avantages et inconvénients des vues matérialisées dans le contexte universitaire.

Piège courant : Présenter les vues matérialisées comme une solution universelle. Elles sont inadaptées quand les données changent très fréquemment (temps réel) ou quand l'espace disque est très contraint.
Structurez votre réponse en deux colonnes : avantages (performance, indexabilité, réécriture auto) et inconvénients (stockage, fraîcheur, complexité de maintenance). Concluez avec le bilan pour le contexte académique.

Solution — Q1.11

Avantages — Performance

  • Temps de réponse drastiquement réduit pour les requêtes analytiques (ms vs s)
  • Allègement de la charge CPU : les agrégats coûteux sont calculés une seule fois
  • Indexation possible directement sur les résultats agrégés
  • Réécriture automatique : le SGBD peut utiliser la VM sans modifier les requêtes applicatives
  • Idéal pour les dashboards consultés fréquemment (100×/jour)

Inconvénients — Stockage & Fraîcheur

  • Espace disque supplémentaire (~50 000 lignes × nb de VMs)
  • Données potentiellement obsolètes entre deux rafraîchissements
  • Coût de maintenance : le recalcul complet bloque la VM pendant quelques secondes
  • Complexité de gestion : décider quelles vues créer, quand les rafraîchir
  • Risque de désynchronisation si les politiques de rafraîchissement sont mal définies
Bilan pour le contexte universitaire : Le contexte académique est très favorable aux vues matérialisées : les données évoluent lentement (inscriptions et notes saisies par lots), les requêtes analytiques sont fréquentes (tableaux de bord, suivi pédagogique) et une légère obsolescence des données (24h) est parfaitement acceptable. Le gain de performance justifie largement le surcoût de stockage.
Q1.12 Mise à jour du schéma Avancé

Si on ajoute une nouvelle colonne Mention VARCHAR(20) dans la table Inscriptions, faut-il recréer la vue matérialisée VM_Performance_Etudiants ? Distinguez les deux cas.

Piège courant : Penser qu'une modification de la table source invalide toujours la VM. Ce n'est pas le cas si la nouvelle colonne n'est pas référencée dans la définition de la VM.
Cas 1 : la VM ne référence pas Mention dans son SELECT → elle continue de fonctionner sans modification. Cas 2 : on veut inclure Mention dans la VM → il faut la recréer avec CREATE OR REPLACE MATERIALIZED VIEW.
La règle générale : une VM est indépendante des modifications de schéma des tables sources tant que les colonnes qu'elle référence existent toujours. Ajouter une colonne est non-destructif. Renommer ou supprimer une colonne référencée est destructif.

Solution — Q1.12

Cas 1 — Mention non utilisée dans la VM

Non, la VM n'a pas besoin d'être recréée. Elle continue de fonctionner normalement car sa définition ne référence pas la colonne Mention. Un simple REFRESH suffit si des notes ont été modifiées en parallèle. 

-- Aucune modification de la VM nécessaire.
-- Optionnel : rafraîchir si des données ont changé
REFRESH MATERIALIZED VIEW
    VM_Performance_Etudiants;

Cas 2 — On veut inclure Mention dans la VM

Oui, il faut recréer la VM avec CREATE OR REPLACE MATERIALIZED VIEW pour ajouter Mention à la définition. 

CREATE OR REPLACE MATERIALIZED VIEW
    VM_Performance_Etudiants AS
SELECT
    e.ID_etudiant,
    e.Nom || ' ' || e.Prenom AS Nom_Complet,
    e.Niveau,
    COUNT(i.ID_cours)    AS Nb_Cours_Suivis,
    ROUND(AVG(i.Note),2) AS Moyenne_Generale,
    MODE() WITHIN GROUP
        (ORDER BY i.Mention) AS Mention_Freq
FROM   Etudiants e
LEFT JOIN Inscriptions i
    ON e.ID_etudiant = i.ID_etudiant
GROUP BY
    e.ID_etudiant, e.Nom,
    e.Prenom, e.Niveau;
Règle générale : une vue matérialisée est indépendante des modifications de schéma des tables sources tant que les colonnes qu'elle référence dans son SELECT et ses JOIN existent toujours. Ajouter une colonne à une table source est une opération non destructive pour les VMs existantes.
