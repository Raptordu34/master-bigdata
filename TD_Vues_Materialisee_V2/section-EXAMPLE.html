Exercice 3 — Optimisation et Indexation SQL

Intermédiaire ◎ 14 pts ⏱ 30–45 min
Objectifs
  • Identifier les goulots d'étranglement d'une requête SQL
  • Choisir le type d'index adapté à un cas d'usage donné
  • Écrire et mesurer l'effet d'une vue matérialisée

Prérequis : Cours 4 — Modèle relationnel · Cours 6 — Optimisation de requêtes

Contexte — Système de gestion pédagogique

Vous travaillez sur la base de données d'un établissement d'enseignement supérieur. Le DBA constate des requêtes lentes sur les relevés de notes et vous demande de diagnostiquer et corriger les performances.

TableColonnesLignes
EtudiantsID_etudiant, nom, prenom, date_naissance12 000
CoursID_cours, intitule, credits, ID_dept340
InscriptionsID_etudiant, ID_cours, semestre, note180 000
Note : Aucun index secondaire n'existe pour l'instant — seules les clés primaires sont indexées. La requête de référence pour les questions 1 et 2 est donnée ci-dessous. 
-- Requête de référence (actuellement : ~2 400 ms)
SELECT e.nom, e.prenom, c.intitule, i.note
FROM   Etudiants e
JOIN   Inscriptions i ON i.ID_etudiant = e.ID_etudiant
JOIN   Cours c        ON c.ID_cours    = i.ID_cours
WHERE  i.semestre = '2024-A'
  AND  i.note     >= 10
ORDER BY e.nom;
Un index B-tree est efficace pour les filtres d'égalité, les intervalles et les tris compatibles avec son ordre. Si la requête filtre sur semestre puis sur note, l'ordre des colonnes dans l'index influe directement sur son utilité.

Lecture d'un plan : un plan d'exécution indique dans quel ordre PostgreSQL lit, filtre, joint et trie les données.

Réflexe utile : avant de créer un index, vérifier d'abord où le moteur lit trop de lignes ou trie trop tard dans le pipeline.

  • Seq Scan : parcours complet de table, parfois normal sur de petits volumes, coûteux sur de gros ensembles.
  • Index Scan : accès ciblé si la condition de filtre correspond à un index exploitable.
  • Sort : étape potentiellement chère si le tri porte sur beaucoup de lignes et déborde en disque.

Repère

Si une étape lit des dizaines de milliers de lignes pour n'en conserver que quelques milliers, le problème est souvent un filtrage trop tardif.

NœudQuestion à se poserSignal d'alerte
Seq ScanLa table est-elle trop grosse pour une lecture complète ?Beaucoup de lignes supprimées par le filtre
SortLe tri pourrait-il être évité par un index ?external merge ou gros volume trié
Hash JoinLa jointure est-elle correcte mais alimentée par trop de lignes ?Temps élevé malgré une jointure raisonnable

Exemple

Si Inscriptions contient seulement 200 lignes, un Seq Scan reste souvent acceptable. Avec 180 000 lignes, le même parcours devient coûteux : le besoin d'indexation change d'échelle.

Analogie

Intuition : un index joue le rôle d'un sommaire dans un manuel. Sans lui, le moteur relit page après page ; avec lui, il saute directement au bon chapitre puis affine la recherche.

Exemple développé

Supposons qu'un relevé de notes soit demandé pour 3 200 inscriptions pertinentes parmi 180 000 lignes stockées. Sans index composite, PostgreSQL parcourt toute la table, filtre tardivement, puis trie le résultat.

  • Lecture initiale : 180 000 lignes consultées.
  • Lignes utiles : environ 3 200.
  • Effet attendu après indexation : accès direct au sous-ensemble filtré.
CREATE INDEX idx_inscriptions_sem_note
    ON Inscriptions (semestre, note);

L'exemple développé montre le contexte, l'action, puis le gain attendu. Il ne se contente pas d'une commande isolée.

Analogie développée

Image mentale : imaginez une bibliothèque universitaire sans catalogue. Pour trouver un ouvrage, il faut inspecter chaque rayon l'un après l'autre.

L'index est ce catalogue. Il n'écrit pas le contenu à votre place, mais il vous dit immédiatement où aller. Un index composite revient à classer d'abord par discipline, puis par auteur : on réduit progressivement l'espace de recherche.

  • Sans index : exploration exhaustive.
  • Avec index simple : accès rapide à une première catégorie.
  • Avec index composite : accès rapide à la bonne catégorie puis au bon sous-ensemble.
Q1 4 pts Fondamental

En utilisant EXPLAIN ANALYZE, identifiez les deux opérations les plus coûteuses dans le plan d'exécution de la requête de référence. Justifiez pourquoi elles sont coûteuses.

Piège courant : Confondre le coût estimé (cost=…) avec le temps mesuré (actual time=…). EXPLAIN seul n'exécute pas la requête — utiliser obligatoirement EXPLAIN ANALYZE.
Cherchez les nœuds avec Seq Scan (parcours séquentiel) sur la table la plus volumineuse. Le coût explose quand le moteur parcourt 180 000 lignes sans index.
Deuxième opération coûteuse : le tri (Sort) sur e.nom porte sur l'ensemble des lignes filtrées. Sans index sur nom, il ne peut pas être évité.

Solution — Q1

  • Lancer EXPLAIN ANALYZE SELECT … pour obtenir le vrai plan d'exécution.
  • Repérer Seq Scan on Inscriptions — parcours de 180 000 lignes sans filtrage préalable par index sur semestre ou note.
  • Repérer Sort on e.nom — tri en mémoire ou sur disque de plusieurs milliers de lignes.
mydb=# EXPLAIN ANALYZE SELECT e.nom, …; QUERY PLAN --------------------------------------------------------------------------- Sort (cost=4820..4830 rows=3200 width=72) (actual time=2100..2150 rows=3200) Sort Key: e.nom Sort Method: external merge Disk: 420kB -> Hash Join (cost=210..580 rows=3200 width=72) (actual time=18..1900) -> Seq Scan on Inscriptions (cost=0..4800 rows=3200 width=20) Filter: ((semestre = '2024-A') AND (note >= 10)) Rows Removed by Filter: 176800 # → Seq Scan = coûteux (176 800 lignes ignorées) # → Sort Method: external merge = débordement sur disque

Le Seq Scan élimine 176 800 lignes sur 180 000 : un index sur (semestre, note) réduirait ce coût à O(log n).

Q2 5 pts Intermédiaire

Proposez et créez les index qui permettraient d'optimiser la requête de référence. Justifiez chaque choix (type d'index, colonnes, ordre).

Piège courant : Créer un index sur chaque colonne de la table. Chaque index ralentit les INSERT/UPDATE. Il faut cibler uniquement les colonnes filtrées (WHERE) et triées (ORDER BY).
Un index composite sur (semestre, note) couvre le filtre WHERE. Pour le tri ORDER BY e.nom, un index sur nom dans Etudiants permet au planificateur d'éviter le sort explicite.

Méthode

  • Repérer d'abord les colonnes de filtre dans WHERE.
  • Vérifier ensuite si un tri ou une jointure fréquente justifie un second index.
  • Éviter les index isolés inutiles sur des colonnes peu sélectives.
  • Valider la proposition avec un nouveau EXPLAIN ANALYZE.

Solution — Q2

-- Index 1 : filtre composite sur Inscriptions
-- semestre en premier (égalité exacte) → note ensuite (range ≥ 10)
CREATE INDEX idx_inscriptions_sem_note
    ON Inscriptions (semestre, note);

-- Index 2 : tri ORDER BY sur Etudiants.nom
CREATE INDEX idx_etudiants_nom
    ON Etudiants (nom);
Pourquoi composite ?
Un index (semestre, note) couvre les deux prédicats en un seul parcours B-tree. L'ordre des colonnes suit la règle du préfixe le plus sélectif.
Résultat attendu
Avec ces deux index : Seq Scan → Index Scan, temps de 2 400 ms → ~40 ms. Gain ×60.
Opération avantOpération aprèsGain
Seq Scan (180 000 lignes)Index Scan (~3 200 lignes)×56
Sort (external merge, disque)Index Scan (trié natif)Sans tri

Contre-exemple

Un index sur note seul semble séduisant, mais il n'aide pas à filtrer d'abord sur semestre = '2024-A'. Le moteur risque alors de parcourir trop d'entrées pour rester performant.

Checklist avant validation :
  • L'index proposé correspond-il à un vrai filtre ou tri fréquent ?
  • L'ordre des colonnes de l'index est-il justifié ?
  • Le gain est-il vérifié par un plan d'exécution et pas seulement supposé ?
Q3 5 pts Avancé

Le rapport de notes par cours est interrogé 500 fois par jour, toujours avec les mêmes agrégats. Implémentez une vue matérialisée vm_stats_cours contenant : la moyenne, la note maximale, la note minimale et le nombre d'inscrits par cours et par semestre.

a) Écrivez le DDL de la vue matérialisée.
b) Ajoutez un index sur la vue pour accélérer les recherches par semestre.
c) Quelle commande permet de rafraîchir la vue sans bloquer les lectures en production ?
Piège courant : Lancer REFRESH MATERIALIZED VIEW vm_stats_cours sans CONCURRENTLY. Sans ce mot-clé, toutes les lectures sont bloquées pendant le rafraîchissement — inacceptable en production.
Syntaxe de base : CREATE MATERIALIZED VIEW nom AS SELECT … WITH DATA;. Utilisez GROUP BY ID_cours, semestre pour les agrégats.
Pour REFRESH CONCURRENTLY, la vue doit posséder un index unique. Créez-le sur (ID_cours, semestre) avant d'utiliser le refresh concurrent.

Solution — Q3

a) DDL de la vue matérialisée

CREATE MATERIALIZED VIEW vm_stats_cours AS
    SELECT
        i.ID_cours,
        c.intitule,
        i.semestre,
        ROUND(AVG(i.note), 2)  AS moyenne,
        MAX(i.note)            AS note_max,
        MIN(i.note)            AS note_min,
        COUNT(*)               AS nb_inscrits
    FROM Inscriptions i
    JOIN Cours c ON c.ID_cours = i.ID_cours
    GROUP BY i.ID_cours, c.intitule, i.semestre
WITH DATA;

b) Index pour les recherches par semestre

-- Index unique requis pour REFRESH CONCURRENTLY
CREATE UNIQUE INDEX idx_vm_cours_pk
    ON vm_stats_cours (ID_cours, semestre);

-- Index secondaire pour les filtres sur semestre seul
CREATE INDEX idx_vm_cours_semestre
    ON vm_stats_cours (semestre);

c) Rafraîchissement sans blocage

-- CONCURRENTLY : les lectures restent possibles pendant le refresh
REFRESH MATERIALIZED VIEW CONCURRENTLY vm_stats_cours;

Planifier ce REFRESH via pg_cron ou un job externe (toutes les heures, la nuit, ou après chaque vague d'inscriptions).
